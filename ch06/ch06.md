# Chapter 6: Persistent Memory — Giving Claude a Notebook

> *"The difference between a contractor you hire once and a trusted colleague is that the colleague remembers what you told them last week."*

## What This Chapter Is About

Every time you restart Nanocode, Claude forgets everything. Your preferred coding style, the name of the project, the bug you were chasing — gone. Claude starts each session as a stranger.

In this chapter we fix that with a `Memory` class that writes to `.nanocode/memory.md` and a new tool — `SaveMemory` — that lets Claude decide what is worth remembering. When Nanocode starts up, it reads the memory file and injects its contents as a `system` prompt, so Claude walks into every session already knowing the important things.

By the end of this chapter you'll understand:
- What the `system` prompt is and how it differs from ordinary messages
- How `Memory` bridges disk and Claude's context window
- Why `ToolContext` exists and what circular-dependency problem it avoids
- How cross-session memory works end-to-end
- What a saved memory file looks like in practice

The new file is `ch06/nanocode.py`. Run it with `python ch06/nanocode.py` and try telling Claude something important, then asking it to remember it with `save_memory`.

---

## The Problem: Claude Has No Long-Term Memory

A language model is stateless. Each API call is isolated — Claude does not remember the previous call. The conversation history we built in Chapter 3 gives Claude *within-session* memory: it can refer back to what you said ten messages ago. But that list lives in Python's memory, not on disk. When the process exits, it's gone.

The gap is between sessions:

```
Session 1:
  You: "Always use type hints in Python."
  Claude: "Noted, I'll use type hints."
  [you quit]

Session 2:
  You: "Fix the bug in utils.py"
  Claude: [writes code without type hints — forgot everything]
```

We need to persist something to disk so that Session 2 starts informed.

## The System Prompt: Claude's Standing Orders

The Anthropic API has a field called `system`. Unlike messages in the `messages` list, the `system` field is not part of the conversation thread — it is a persistent instruction that precedes every reply Claude ever makes in that session.

Think of it as a briefing document. Before a detective begins a case, they read the file: who the client is, what the background is, what they already know. The `system` prompt is that file — it shapes how Claude interprets everything that follows, without appearing as a visible message.

```
┌──────────────────────────────────────────────────────────────────┐
│                     API Payload Structure                        │
│                                                                  │
│  {                                                               │
│    "model": "claude-sonnet-4-6",                                 │
│    "max_tokens": 4096,                                           │
│                                                                  │
│    "system": "I am Nanocode, a helpful coding assistant.         │
│               The user prefers type hints in all Python.         │
│               Current project: blog-engine."                     │
│         │                                                        │
│         └─── NOT in messages[] ── sits above the conversation    │
│                                                                  │
│    "messages": [                                                  │
│      {"role": "user",      "content": "Fix the bug in utils.py"},│
│      {"role": "assistant", "content": [...]}                     │
│    ]                                                             │
│  }                                                               │
└──────────────────────────────────────────────────────────────────┘
```

Whatever is in `system`, Claude treats it as the permanent context for the session. If the memory file says "the user always wants docstrings," Claude will write docstrings without being asked.

## The `Memory` Class

`Memory` is a thin wrapper around a file. It handles creation, loading, and saving:

```python
class Memory:
    """Persistent scratchpad for the agent."""

    def __init__(self, path=".nanocode/memory.md"):
        self.path = path
        self._ensure_exists()
        self.content = self._load()

    def _ensure_exists(self):
        """Create memory file with default content if needed."""
        os.makedirs(os.path.dirname(self.path), exist_ok=True)
        if not os.path.exists(self.path):
            default = "I am Nanocode, a helpful coding assistant.\n"
            with open(self.path, "w") as f:
                f.write(default)

    def _load(self):
        """Load content from disk."""
        with open(self.path, 'r') as f:
            return f.read()

    def save(self, content):
        """Update memory content and persist to disk."""
        self.content = content
        with open(self.path, 'w') as f:
            f.write(content)
```

Walk through what happens at startup:

```
Memory(".nanocode/memory.md")
         │
         ├─ _ensure_exists()
         │       │
         │       ├─ os.makedirs(".nanocode/", exist_ok=True)
         │       │       creates the folder if it doesn't exist
         │       │
         │       └─ if file missing → write default content
         │               "I am Nanocode, a helpful coding assistant."
         │
         └─ self.content = _load()
                 reads the file into memory
                 (the object now mirrors the disk)
```

`_ensure_exists()` uses `exist_ok=True` on `makedirs` — if `.nanocode/` already exists, it does nothing. If the memory file already exists, we skip writing the default. This means the first run creates the file with sensible defaults, and every subsequent run loads whatever Claude has accumulated.

`save()` updates both `self.content` and the file at the same time. They are always in sync — there's no "dirty" state to track.

## The `.nanocode/` Folder Convention

Why put memory in `.nanocode/` rather than a top-level file?

The agent runs inside your project directory. If we wrote `memory.md` at the root, it would clutter your project alongside your source files. By convention, tools that manage their own state use a dotfile directory — `.git/`, `.venv/`, `.cache/`. The leading dot hides it from casual `ls` output and signals "this is managed by a tool, not a project artifact."

Future chapters will put additional agent artifacts here: conversation logs, task queues, and configuration. Keeping them together makes it easy to add `.nanocode/` to `.gitignore` if you don't want to commit the agent's private state.

## The `ToolContext` Problem

Here is a question: if `SaveMemory.execute()` needs to call `memory.save()`, how does it get access to the `Memory` object?

The naive answer is: give every tool a reference to the `Agent`. Then tools can do `agent.memory.save(...)`.

This is a bad idea. An `Agent` owns tools. If tools also own the `Agent`, you have a circular reference:

```
Agent ──owns──▶ [ReadFile, WriteFile, SaveMemory]
  ▲                              │
  └──────── owns back ───────────┘
```

Circular dependencies make code hard to test, hard to reason about, and prone to memory leaks. The tools are supposed to be independent, testable units — they shouldn't need to know about the whole agent.

The solution is `ToolContext`: a small plain object that carries just what tools need, created fresh for each tool call:

```python
class ToolContext:
    """What tools need to know about the agent's state."""

    def __init__(self, memory=None):
        self.memory = memory  # Memory object or None
```

Think of it as an envelope. Each time `_execute_tool` is called, it stuffs the current state into an envelope and passes it to the tool:

```
┌─────────────────────────────────────────────────────────┐
│                     _execute_tool()                     │
│                                                         │
│  context = ToolContext(memory=self.memory)               │
│         │                                               │
│         │   ┌─────────────────────────────┐             │
│         └──▶│      ToolContext             │             │
│             │  .memory = Memory object     │             │
│             └─────────────────────────────┘             │
│                         │                               │
│                         ▼                               │
│           tool.execute(context, **args)                  │
│                                                         │
│  Tool receives a self-contained package of              │
│  state — no reference back to Agent needed.             │
└─────────────────────────────────────────────────────────┘
```

The tool uses what it needs (`context.memory`) and ignores the rest. The `Agent` never becomes a dependency.

## Updated Tool Signatures

Because tools now receive `context` as a first argument, all `execute()` methods changed their signature:

```python
# ch05: no context
def execute(self, path):
    ...

# ch06: context is first
def execute(self, context, path):
    ...
```

`ReadFile` and `WriteFile` ignore `context` — they don't need memory. `SaveMemory` uses it:

```python
class SaveMemory:
    """Updates the agent's internal memory/scratchpad."""
    name = "save_memory"
    description = "Updates your internal memory/scratchpad. Use this to remember user preferences."
    input_schema = {
        "type": "object",
        "properties": {
            "content": {"type": "string", "description": "The full text to save."}
        },
        "required": ["content"]
    }

    def execute(self, context, content):
        print(f"  → Saving memory")
        if context.memory is None:
            return "Error: Memory not available"
        context.memory.save(content)
        return "Memory updated successfully."
```

`SaveMemory` does one thing: call `context.memory.save(content)`. The `if context.memory is None` guard exists because `ToolContext` is designed to be generic — in a future chapter, some tools might run in a context where memory is intentionally absent.

Notice that `content` in the schema is the full new memory text, not a delta. Claude rewrites the entire memory file each time it saves. This is the same pattern as `WriteFile` — simpler to reason about than incremental patching.

## Wiring Memory into the Brain

The `Claude` constructor now accepts `memory`:

```python
class Claude(Brain):
    def __init__(self, memory=None, tools=None):
        self.memory = memory
        ...

    def think(self, conversation):
        payload = {
            "model": self.model,
            "max_tokens": 4096,
            "messages": conversation
        }
        if self.memory:
            payload["system"] = self.memory.content
        if self.tools:
            payload["tools"] = self.tools
        ...
```

Every call to `think()` injects the current memory content as `payload["system"]`. If Claude updated the memory during the session (via `save_memory`), `memory.content` is already updated — so the next API call will reflect the new content automatically. No restart needed.

## Wiring Memory into `_execute_tool`

One line changed in `_execute_tool`:

```python
def _execute_tool(self, name, args):
    """Execute a tool by name with given arguments."""
    tool = get_tool(self.tools, name)
    if tool is None:
        return f"Error: Tool '{name}' not found"
    try:
        context = ToolContext(memory=self.memory)   # ← new
        return tool.execute(context, **args)         # ← context is first arg
    except TypeError as e:
        return f"Error: Invalid arguments - {e}"
```

`ToolContext` is instantiated here, not stored on `Agent`. A new one is created for every tool call. Since `ToolContext` is a thin value object (it holds a reference to the existing `Memory`, it doesn't copy it), this is inexpensive.

## Startup: The Complete Picture

```python
def main():
    brain_name = os.getenv("NANOCODE_BRAIN", "claude")
    memory = Memory()                                          # 1
    brain = BRAINS[brain_name](memory=memory, tools=tool_definitions(tools))  # 2
    agent = Agent(brain=brain, tools=tools, memory=memory, brain_name=brain_name)  # 3

    print(f"⚡ Nanocode v0.5 (Memory Enabled)")
    ...
```

1. `Memory()` reads (or creates) `.nanocode/memory.md` — the agent's notebook is open
2. The brain receives `memory=memory` — every API call will include `memory.content` as `system`
3. The agent receives `memory=memory` — every tool call will receive it via `ToolContext`

Both `brain` and `agent` hold a reference to the *same* `Memory` object. When `SaveMemory` calls `context.memory.save(content)`, it updates the `Memory` object's `self.content`. On the very next call to `brain.think()`, the updated `self.content` goes into `payload["system"]`. There is no synchronization problem because everything shares one object.

## What a Memory File Looks Like

After a few sessions, `.nanocode/memory.md` might contain:

```markdown
I am Nanocode, a helpful coding assistant.

## User Preferences
- Always use type hints in Python functions
- Write docstrings in Google style
- Prefer f-strings over .format()

## Current Project
Project: blog-engine
Stack: Python + FastAPI + SQLite
Main entry point: src/main.py

## Known Issues
- The /posts endpoint has an off-by-one error in pagination (line 87 of routes.py)
- Unit tests for the auth module are missing

## Notes
- User is learning Python; explain decisions briefly when making non-obvious choices
```

Claude wrote this. You didn't. Over time, as you point things out and Claude learns your preferences, the memory accumulates a richer and richer picture of your project and working style.

The format is plain Markdown. There's no schema to enforce — Claude decides how to organize it. This is intentional: rigid schemas break when the information doesn't fit; free-form text is always writable, even if noisier.

## Cross-Session Memory: The Full Loop

Here is the complete timeline of how a preference survives from one session to the next:

```
SESSION 1
─────────────────────────────────────────────────────────────────

  You:    "Please always use type hints."
  Claude: "Got it. Would you like me to save this to memory?"
  You:    "Yes."
  Claude: [calls save_memory with updated content]
           ↓
  Memory.save() writes to .nanocode/memory.md
           ↓
  File now contains: "...User prefers type hints in all Python."

  [You quit — process exits — conversation list is gone]


SESSION 2
─────────────────────────────────────────────────────────────────

  Memory() reads .nanocode/memory.md
           ↓
  "...User prefers type hints in all Python." → self.content
           ↓
  brain = Claude(memory=memory, ...)
           ↓
  Every API call: payload["system"] = "...User prefers type hints..."
           ↓
  You:    "Add a helper function to utils.py."
  Claude: [writes function with type hints, without being asked]
```

The preference crossed the session boundary through the file system.

## Summary

| Concept | What It Does |
|---------|-------------|
| `Memory` class | Reads, creates, and saves `.nanocode/memory.md` — the agent's notebook |
| `_ensure_exists()` | Creates `.nanocode/` and the default memory file on first run |
| `system` prompt | API field that sits above `messages[]` — shapes all of Claude's replies |
| `ToolContext` | Lightweight envelope carrying agent state (memory) into tool `execute()` calls |
| `SaveMemory` tool | Lets Claude persist new information by rewriting the memory file |
| Cross-session memory | `Memory` reads disk at startup; `SaveMemory` writes disk at runtime — file bridges sessions |
| `.nanocode/` folder | Convention for agent-managed artifacts, separate from your project files |

## What We Didn't Add (And Why)

Memory is append-only in practice — Claude rewrites the whole file, so old facts disappear only if Claude decides to drop them. There's no automatic pruning. Chapter 10 adds context compaction, which addresses a related problem: the conversation history growing too large. A similar compaction strategy could apply to memory, but for now the file is small enough that it doesn't matter.

There's no encryption or access control on the memory file. It's a plain Markdown file any process can read. This is a reasonable default for a local coding assistant — if you're running Nanocode on a shared machine, you'd want to restrict permissions on `.nanocode/` with `chmod 700 .nanocode`.

`ToolContext` currently only carries `memory`. In Chapter 8 it will also carry `mode` (read-only or read-write), so that tools can respect the agent's operating mode without any changes to their signatures.
