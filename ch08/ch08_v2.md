# Chapter 8: Codebase Awareness

So far, the agent can only work with files you explicitly name. "Read `main.py`." "Write `utils.py`." You, the user, must know where everything is and tell the agent where to look. The agent has no independent sense of the project it's working in.

That's backward. A capable coding assistant should be able to explore a codebase autonomously — map its structure, find where a function is defined, search for every usage of a class. You shouldn't need to narrate the directory layout.

This chapter adds two tools that give the agent that independence: `ListFiles` for mapping structure and `SearchCodebase` for finding content.

---

## ListFiles: Mapping the Territory

```python
class ListFiles:
    name = "list_files"
    description = "Lists all files in the project structure. Useful to understand the project layout."
    input_schema = {
        "type": "object",
        "properties": {
            "path": {"type": "string", "description": "The root path (default '.')"}
        }
    }

    def execute(self, context, path="."):
        file_list = []
        for root, dirs, files in os.walk(path):
            if any(p in root for p in [".git", "__pycache__", "venv", ".nanocode"]):
                continue

            level = root.replace(path, '').count(os.sep)
            indent = ' ' * 4 * level
            file_list.append(f"{indent}{os.path.basename(root)}/")
            subindent = ' ' * 4 * (level + 1)
            for f in files:
                file_list.append(f"{subindent}{f}")

        return "\n".join(file_list)
```

The heart of `ListFiles` is `os.walk()`. Walk descends a directory tree recursively, yielding a tuple of `(root, dirs, files)` at each level. `root` is the current directory path. `dirs` is a list of subdirectory names. `files` is a list of filenames in that directory.

We skip certain directories unconditionally. `.git` contains internal version control data — Claude has no use for it and it would flood the output. `__pycache__` holds compiled bytecode. `venv` is your virtual environment, potentially tens of thousands of files from installed packages. `.nanocode` is the agent's own state directory. Showing Claude any of these would waste tokens and produce noise.

The indentation calculation turns the raw `os.walk` output into a visual tree. `level` is the nesting depth of the current directory, computed by counting path separators. Each level gets four spaces of indentation. The result is a tree that Claude can read at a glance:

```
./
    src/
        main.py
        utils.py
    tests/
        test_main.py
    requirements.txt
```

## SearchCodebase: Finding Content

```python
class SearchCodebase:
    name = "search_codebase"
    description = "Searches the entire codebase for a text string. Useful to find where functions or variables are defined."
    input_schema = {
        "type": "object",
        "properties": {
            "query": {"type": "string", "description": "The string to search for"},
            "path":  {"type": "string", "description": "The root path (default '.')"}
        },
        "required": ["query"]
    }

    def execute(self, context, query, path="."):
        results = []
        for root, dirs, files in os.walk(path):
            if any(p in root for p in [".git", "__pycache__", "venv", ".nanocode"]):
                continue

            for file in files:
                file_path = os.path.join(root, file)
                try:
                    with open(file_path, 'r', encoding='utf-8', errors='ignore') as f:
                        for i, line in enumerate(f):
                            if query.lower() in line.lower():
                                results.append(f"{file_path}:{i+1}: {line.strip()}")
                except Exception:
                    continue

        return "\n".join(results) if results else "No matches found."
```

`SearchCodebase` walks the same directory tree as `ListFiles`, but opens every file and scans it line by line. The search is case-insensitive — `query.lower() in line.lower()` — because code identifiers often appear in different cases across different contexts, and a search that misses `MyClass` when you search for `myclass` isn't useful.

Each match is formatted as `filepath:lineno: matched_line`. This is the standard grep output format — familiar to developers, and parseable by Claude. When Claude sees output in this format, it knows the file path, the exact line number, and the content that matched.

Two error-handling choices here are worth noting. `errors='ignore'` in the file open call means binary files — compiled executables, images, other non-text content — are skipped without crashing. Without it, opening a `.pyc` file or a `.png` would raise a `UnicodeDecodeError`. The bare `except Exception: continue` inside the loop swallows any remaining errors on a per-file basis: a locked file, a permission error, a corrupted file — none of these abort the search. You get partial results instead of a crash.

## Why These Tools Transform the Agent

Before these tools, the agent's knowledge of a codebase came entirely from what you told it. You were the navigator — describing the terrain, pointing to relevant files, telling the agent where things were.

After these tools, the agent navigates itself.

Ask it to refactor a module, and it will list the files first to understand the project layout. Ask it to find all callers of a function, and it will search the codebase rather than waiting for you to enumerate them. Ask it to add a new feature, and it will explore the existing code to understand the conventions and patterns already in use.

This is the difference between a tool you operate and a collaborator you brief. You describe the goal. The agent figures out what it needs to know to pursue it.
