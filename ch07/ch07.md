# Chapter 7: Plan Mode — The Safety Harness

> *"A surgeon who cuts before measuring is a danger. A coding agent that writes before planning is the same."*

## What This Chapter Is About

By Chapter 6, Nanocode could read files, write files, and remember things across sessions. It was genuinely useful. It was also genuinely dangerous.

Give it a prompt like "clean up my project" and it would start overwriting files immediately — no preview, no confirmation, no plan. One misunderstood instruction and hours of work could vanish.

This chapter adds the safety harness: **Plan Mode**. The agent defaults to read-only. It can explore your codebase, think through a solution, and write its intentions to `PLAN.md` — but it cannot touch any other file until you explicitly say so.

By the end of this chapter you'll understand:
- Why plan mode is the default, not an option
- How `ToolContext` carries safety state from the agent down to every tool
- Why enforcement lives in the tool, not the agent loop
- How `PLAN.md` gives the agent a voice even when it cannot act
- The `--act` flag pattern for explicit opt-in to dangerous operations

---

## The Problem: Agents Can Be Destructive

Imagine you have a directory with 40 Python files. You ask Nanocode: "Refactor all uses of `old_function` to `new_function`."

Without plan mode, the agent:
1. Reads a file — fine
2. Writes the modified version — fine, probably
3. Does it 39 more times — also fine... unless it made the same mistake 40 times

You have no checkpoint. No way to review what it planned before it executed. By the time you see the output, your filesystem has already changed.

The carpenter's rule applies: *measure twice, cut once.* Plan mode enforces that the agent measures — reads, thinks, writes a plan to `PLAN.md` — before it is allowed to cut.

```
                    PLAN MODE (default)
  ┌─────────────────────────────────────────────────┐
  │                                                 │
  │   read_file   ──────────────────────▶  ALLOWED  │
  │   save_memory ──────────────────────▶  ALLOWED  │
  │   write_file (PLAN.md) ─────────────▶  ALLOWED  │
  │   write_file (anything else) ───────▶  BLOCKED  │
  │                                                 │
  └─────────────────────────────────────────────────┘

                     ACT MODE (opt-in)
  ┌─────────────────────────────────────────────────┐
  │                                                 │
  │   read_file   ──────────────────────▶  ALLOWED  │
  │   save_memory ──────────────────────▶  ALLOWED  │
  │   write_file (PLAN.md) ─────────────▶  ALLOWED  │
  │   write_file (anything else) ───────▶  ALLOWED  │
  │                                                 │
  └─────────────────────────────────────────────────┘
```

## The ToolContext: Passing State Down the Chain

In previous chapters, tools received only the arguments Claude passed them — `path`, `content`, and so on. But now tools need to know something the *agent* knows: what mode are we in?

`ToolContext` is the messenger. It's a plain Python object that the agent creates just before calling a tool and passes as the first argument to `execute()`.

```python
class ToolContext:
    """What tools need to know about the agent's state."""

    def __init__(self, mode=None, memory=None):
        self.mode = mode      # "plan" or "act"
        self.memory = memory  # Memory object or None
```

Two fields. Nothing else. This is deliberate minimalism — `ToolContext` is not a kitchen sink. It carries exactly the state that tools cannot derive from their own arguments.

Here is how it flows:

```
  Agent._execute_tool(name, args)
       │
       │  creates
       ▼
  ToolContext(mode=self.mode, memory=self.memory)
       │
       │  passed as first argument
       ▼
  tool.execute(context, **args)
       │
       ├── context.mode  ──▶  "plan" or "act"
       └── context.memory ──▶  Memory object
```

The agent owns the mode. The tool receives it. No global variable, no class-level state, no shared mutable object. The context is constructed fresh for every tool call.

In `_execute_tool`:

```python
def _execute_tool(self, name, args):
    """Execute a tool by name with given arguments."""
    tool = get_tool(self.tools, name)
    if tool is None:
        return f"Error: Tool '{name}' not found"
    try:
        context = ToolContext(mode=self.mode, memory=self.memory)
        return tool.execute(context, **args)
    except TypeError as e:
        return f"Error: Invalid arguments - {e}"
```

The key line is `context = ToolContext(mode=self.mode, memory=self.memory)`. This is where the agent's current mode becomes the tool's operating constraint. The tool cannot bypass this — it receives `context.mode` and must act accordingly.

## WriteFile: Mode-Aware Execution

`WriteFile` is the only tool that changes behavior based on mode. The logic has three branches, evaluated in order:

```python
def execute(self, context, path, content):
    # Always allow writing to PLAN.md
    if os.path.basename(path) == "PLAN.md":
        print(f"  → Writing {path}")
        try:
            with open(path, 'w', encoding='utf-8') as f:
                f.write(content)
            return "Plan saved successfully to PLAN.md"
        except Exception as e:
            return f"Error saving plan: {e}"

    # Block in plan mode
    if context.mode == "plan":
        return f"BLOCKED: Cannot write to '{path}' in plan mode. Write to 'PLAN.md' instead."

    # Allow in act mode
    print(f"  → Writing {path}")
    try:
        with open(path, 'w', encoding='utf-8') as f:
            f.write(content)
        return f"Successfully wrote {len(content)} characters to {path}"
    except Exception as e:
        return f"Error writing file: {e}"
```

Walk through the branches:

**Branch 1 — The PLAN.md exception.** `os.path.basename(path) == "PLAN.md"` catches any path whose filename is `PLAN.md` (e.g., `./PLAN.md`, `docs/PLAN.md`, just `PLAN.md`). This check runs before the mode check. `PLAN.md` is always writable — plan mode or act mode, the agent can always document its intentions.

**Branch 2 — Block in plan mode.** If the path is not `PLAN.md` and the mode is `"plan"`, return a `BLOCKED:` message. The message also hints at what the agent *should* do: write to `PLAN.md` instead. Claude will read this message as a tool result and adjust its approach.

**Branch 3 — Allow in act mode.** If neither of the above conditions fired, we're in act mode and the path is not `PLAN.md`. Write normally and report the character count.

Notice the order matters: `PLAN.md` is checked first, *before* the mode check. This is important. In plan mode, a write to `PLAN.md` must succeed. If we reversed the order — mode check first — then a call like `write_file(path="PLAN.md", ...)` in plan mode would be blocked before we ever looked at the filename.

```
  write_file called
       │
       ▼
  Is path PLAN.md?  ──── YES ──▶  Write it. Return "Plan saved."
       │
       NO
       │
       ▼
  Is mode "plan"?  ──── YES ──▶  Return "BLOCKED: ..."
       │
       NO
       │
       ▼
  Write the file. Return "Successfully wrote N chars."
```

## Why Enforce in the Tool, Not the Agent?

You might wonder: why not check the mode in `_execute_tool` and skip calling the tool at all?

```python
# This would work, but it's the wrong place
def _execute_tool(self, name, args):
    if self.mode == "plan" and name == "write_file":
        return "BLOCKED: plan mode"
    ...
```

This approach has a hidden fragility. The agent loop is general-purpose code that changes with every chapter. It is easy to accidentally add a code path that bypasses this check — a new branch, a refactored method, a future contributor who doesn't know the rule.

Tools, by contrast, are self-contained. Each tool owns its own behavior. The safety check lives inside `WriteFile.execute()`, right next to the write operation it protects. There is no way to call that write operation without passing through the check.

This is the principle of **defense in depth**: enforce safety at the last possible point before the dangerous action, not at some earlier point that might be bypassed.

Think of it like a bank vault. The vault door has its own combination lock. The fact that you already showed ID at the front desk does not open the vault. The vault enforces its own access rules regardless of what happened before.

## The PLAN.md Exception: Giving the Agent a Voice

Why allow `PLAN.md` at all? Why not make plan mode truly read-only?

Because the agent needs somewhere to *communicate*. If you ask "what would you do to refactor this codebase?", the agent might think for several steps — reading files, tracing dependencies, identifying patterns. That reasoning needs to go somewhere. Printing it to the terminal works for a single session, but it disappears. `PLAN.md` is the agent's scratchpad, visible to you, persistent on disk.

The workflow looks like this:

```
  You: "Refactor all uses of old_function to new_function"
       │
       ▼
  Agent [plan mode]
    ├── read_file("module_a.py")   ──▶ finds 3 uses
    ├── read_file("module_b.py")   ──▶ finds 7 uses
    ├── read_file("tests/test_a.py") ──▶ finds 2 uses
    └── write_file("PLAN.md", ...)  ──▶ documents all 12 locations
                                        and proposed changes
       │
       ▼
  You read PLAN.md, decide it looks right
       │
       ▼
  You: /mode act
       │
       ▼
  You: "Execute the plan"
       │
       ▼
  Agent [act mode]
    ├── write_file("module_a.py", ...)
    ├── write_file("module_b.py", ...)
    └── write_file("tests/test_a.py", ...)
```

`PLAN.md` is the contract between the agent and the user. The agent writes its intentions. You review and approve. Then you flip the mode.

## The Agent Class: Mode as First-Class State

`Agent` gains a `mode` field in its constructor:

```python
class Agent:
    def __init__(self, brain, tools, memory=None, mode="plan", brain_name="claude"):
        self.brain = brain
        self.tools = list(tools)
        self.memory = memory
        self.mode = mode  # "plan" or "act"
        self.brain_name = brain_name
        self.conversation = []
```

The default is `"plan"`. You have to explicitly opt into `"act"`. This is not an accident — safe by default is a design requirement, not a preference. A new user who doesn't know about modes gets plan mode automatically.

`self.mode` is mutable — the `/mode` command changes it at runtime. This is intentional. You might start a session exploring a codebase in plan mode, then switch to act mode to apply a specific fix. The mode travels with the agent instance.

## The /mode Command: _handle_mode_command

The `/mode` command is parsed inside `handle_input` before any message reaches the brain:

```python
def handle_input(self, user_input):
    if user_input.strip() == "/q":
        raise AgentStop()

    if user_input.strip() == "/switch":
        return self._switch_brain()

    if not user_input.strip():
        return ""

    # Handle mode switching
    if user_input.strip().startswith("/mode"):
        return self._handle_mode_command(user_input)

    self.conversation.append({"role": "user", "content": user_input})
    ...
```

The mode command is intercepted before appending to `self.conversation`. This matters: `/mode act` is a local command, not a message to Claude. We don't want it showing up in the conversation history that gets sent to the API.

`_handle_mode_command` is deliberately simple:

```python
def _handle_mode_command(self, user_input):
    """Handle /mode command to switch between plan and act."""
    parts = user_input.strip().split()
    if len(parts) > 1 and parts[1] == "act":
        self.mode = "act"
        return "Switched to ACT MODE (Writing Enabled)"
    else:
        self.mode = "plan"
        return "Switched to PLAN MODE (Read-Only)"
```

Split on whitespace, check the second word. If it's `"act"`, switch to act. Anything else — `/mode plan`, `/mode`, `/mode banana` — switches to plan. This is the safe default: if you're not explicit about enabling writes, you get reads.

```
  User types "/mode act"
       │
       ▼
  parts = ["/mode", "act"]
       │
  len(parts) > 1?  YES
  parts[1] == "act"?  YES
       │
       ▼
  self.mode = "act"
  return "Switched to ACT MODE"

  ---

  User types "/mode"
       │
       ▼
  parts = ["/mode"]
       │
  len(parts) > 1?  NO
       │
       ▼
  self.mode = "plan"
  return "Switched to PLAN MODE"
```

## The --act Flag: Explicit Opt-In at Launch

For users who know they want act mode from the start, there's a CLI flag:

```python
def main():
    # Parse mode from CLI
    mode = "act" if len(sys.argv) > 1 and sys.argv[1] == "--act" else "plan"
    ...
    agent = Agent(brain=brain, tools=tools, memory=memory, mode=mode, brain_name=brain_name)
```

Usage:

```bash
# Read-only mode (default):
python ch07/nanocode.py

# Writing enabled:
python ch07/nanocode.py --act
```

The flag is `--act`, not `--mode=act` or `--enable-writes`. The naming is deliberate. `--act` is short, memorable, and its meaning is hard to misread. When you type `--act`, you know you are telling the agent to act — to make changes.

Compare this to a hypothetical `--unsafe` or `--no-readonly` flag. Those names frame safety as something to be removed. `--act` frames it positively: you are enabling action, not disabling protection.

The prompt itself always shows the current mode:

```python
user_input = input(f"[{agent.brain_name}:{agent.mode}] ❯ ")
```

Sample prompts:

```
[claude:plan] ❯
[claude:act] ❯
[deepseek:plan] ❯
```

The mode is visible before every keystroke. There is no "I thought I was in plan mode" confusion because the mode is always on screen. The square brackets create a status bar — exactly the pattern used by editors like vim and tools like git's interactive rebase.

## Which Tools Are Affected?

| Tool | Plan Mode | Act Mode |
|------|-----------|----------|
| `read_file` | Allowed | Allowed |
| `write_file` → `PLAN.md` | Allowed | Allowed |
| `write_file` → any other file | BLOCKED | Allowed |
| `save_memory` | Allowed | Allowed |

Only `write_file` has mode-aware behavior. `read_file` and `save_memory` are always permitted. Reading is never dangerous (it cannot change your files). Saving memory writes to `.nanocode/memory.md` — the agent's internal scratchpad, not your project files.

The simplicity of this table is a feature. One tool has one restriction. The rule is easy to remember, easy to explain, and easy to audit.

## Summary

| Concept | Key Detail |
|---------|-----------|
| `ToolContext.mode` | Carries `"plan"` or `"act"` from agent to every tool |
| `WriteFile` branch order | `PLAN.md` check runs before mode check — always writable |
| `BLOCKED:` message | Returned as tool result — Claude reads it and adjusts |
| Defense in depth | Enforcement lives in the tool, not in the agent loop |
| `Agent.mode` default | `"plan"` — safe by default, opt-in to danger |
| `_handle_mode_command` | Any non-`"act"` argument defaults to plan mode |
| `--act` CLI flag | Explicit opt-in at launch; no buried toggles |
| `[brain:mode]` prompt | Mode always visible; eliminates user confusion |
