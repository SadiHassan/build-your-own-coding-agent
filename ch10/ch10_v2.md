# Chapter 10: Local Models with Ollama

Every API call in this book so far has gone to a server you don't control, spending money from an account you fund. That's fine for most use cases. But sometimes you want something different: a model that runs on your own hardware, processes data that never leaves your machine, costs nothing per call, and works without an internet connection.

Ollama makes this possible. It's a tool that runs open-source language models locally — the same way Docker runs containers. You install Ollama, pull a model, and it starts serving requests on `localhost`. The interface it exposes is compatible with the Anthropic API, which means it fits into our existing Brain abstraction almost perfectly.

---

## Setting Up Ollama

Before writing any code, you need Ollama running:

```bash
# macOS
brew install ollama

# Linux
curl -fsSL https://ollama.com/install.sh | sh

# Start the server
ollama serve

# Pull a capable coding model
ollama pull qwen3-coder:30b
```

With Ollama running and a model pulled, you can verify it works:

```bash
curl http://localhost:11434/api/generate \
  -d '{"model": "qwen3-coder:30b", "prompt": "Hello"}'
```

If you get a response, Ollama is ready. Now the code.

## The Ollama Brain

```python
class Ollama(Brain):
    context_limit = 32_000

    def __init__(self, memory=None, tools=None):
        self.memory = memory
        self.tools = tools or []
        self.model = os.getenv("OLLAMA_MODEL", "qwen3-coder:30b")
        self.url = "http://localhost:11434/v1/messages"
        self._detect_context_limit()

    def think(self, conversation):
        headers = {
            "x-api-key": "ollama",
            "anthropic-version": "2023-06-01",
            "content-type": "application/json"
        }
        payload = {
            "model": self.model,
            "max_tokens": 4096,
            "messages": conversation
        }
        if self.memory:
            payload["system"] = self.memory.content
        if self.tools:
            payload["tools"] = self.tools

        response = request_with_retry(self.url, headers, payload)
        data = response.json()
        self.last_input_tokens = data.get("usage", {}).get("input_tokens", 0)
        return self._parse_response(data["content"])

    def _parse_response(self, content):
        # identical to Claude and DeepSeek
```

The `think()` method is nearly identical to `Claude.think()`. The URL points to `localhost` instead of `api.anthropic.com`. The `x-api-key` header is set to the string `"ollama"` — Ollama requires the header to be present (it mimics the Anthropic wire format) but ignores the value. Everything else — the payload structure, the response parsing, the `raw_content` handling — is identical.

This is the payoff of the Brain abstraction. Adding a new provider with a compatible API format requires changing almost nothing in the existing code. The `Ollama` class is essentially a configuration object: different URL, different model name, same protocol.

## Detecting the Context Limit

Different Ollama models have different context windows. Mistral 7B might have 8,192 tokens. Llama 3 might have 131,072. Qwen3 Coder might have 32,768. The compaction logic from Chapter 9 needs the right threshold for whichever model is running.

Rather than hardcode a value, we query Ollama at startup:

```python
def _detect_context_limit(self):
    try:
        response = requests.post(
            "http://localhost:11434/api/show",
            json={"model": self.model},
            timeout=2
        )
        model_info = response.json().get("model_info", {})
        for key, value in model_info.items():
            if key.endswith(".context_length"):
                self.context_limit = value
                return
    except Exception:
        pass  # Keep the default of 32,000
```

Ollama's `/api/show` endpoint returns metadata about a model, including a field whose key ends with `.context_length`. The exact key varies by model — it might be `llama.context_length` or `qwen2.context_length` — so we search for any key with the right suffix.

The timeout is 2 seconds. If Ollama isn't running or the query fails for any reason, we catch the exception silently and keep the default of 32,000. The agent starts up without blocking or crashing. If the default happens to be wrong, compaction will kick in at the wrong threshold — a minor inefficiency, not a failure.

## Context Window Differences

The context limits reflect real hardware constraints:

```
Claude Sonnet   200,000 tokens  ── cloud, billed per token
DeepSeek Chat   128,000 tokens  ── cloud, billed per token
Ollama (local)   32,000 tokens  ── your machine, free
```

32,000 tokens sounds like a lot — it's roughly 24,000 words — but in an active agent session with file reads and command output accumulating, it fills faster than you expect. The 75% compaction threshold from Chapter 9 is especially important for Ollama: you compact at ~24,000 tokens instead of waiting until you hit the wall.

Local models also tend to be slower than cloud models for the same capability level. A 30B parameter model running on a laptop will think more slowly than Claude Sonnet on Anthropic's hardware. But it's free, private, and works offline. The trade-off is worth it for many use cases.

## Registering Ollama

One line adds Ollama to the provider registry:

```python
BRAINS = {
    "claude":   Claude,
    "deepseek": DeepSeek,
    "ollama":   Ollama,
}
```

Now you can start the agent with any of the three:

```bash
# Cloud models
python nanocode.py
NANOCODE_BRAIN=deepseek python nanocode.py

# Local model
NANOCODE_BRAIN=ollama python nanocode.py
```

And you can switch between them mid-session with `/switch`, cycling through all three. The conversation history travels intact — both cloud and local models can continue a conversation started by the other, because the history is just a list of strings.

## The Architecture at This Point

With three brains, full tool support, persistent memory, plan/act modes, context compaction, and a test suite, the agent is nearly complete. Here is what the architecture looks like:

```
main()
  │
  ▼
Agent
  ├── conversation []       (session memory)
  ├── mode: plan/act        (safety gate)
  ├── memory: Memory        (persistent memory)
  └── brain: Brain
           ├── Claude        → api.anthropic.com
           ├── DeepSeek      → api.deepseek.com
           └── Ollama        → localhost:11434

Tools (available to all brains):
  ReadFile, WriteFile, EditFile
  ListFiles, SearchCodebase
  SaveMemory, RunCommand
```

Everything coordinates through the agentic loop: think, execute tools, feed results back, repeat until done. The only thing missing is a way for the agent to look something up that happened after its training cutoff. Chapter 11 adds that.
