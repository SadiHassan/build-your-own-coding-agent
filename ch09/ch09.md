# Chapter 9: The Full Loop — Surgical Edits, Shell Commands, and Guarding Against Infinity

> *"A surgeon doesn't replace the whole patient to fix one stitch. A good agent doesn't rewrite the whole file to fix one line."*

## What This Chapter Is About

Up to now, Nanocode could read files, write files, plan, and act. But it had two gaps that real coding work quickly exposes:

1. It could only rewrite files wholesale — replacing every line to fix one bug.
2. It had no way to run code — it could write tests but never execute them.

This chapter closes both gaps. We add `EditFile` for surgical text replacement and `RunCommand` for shell access. Together they unlock the most powerful pattern in agentic coding: **write → run → see errors → fix → repeat**.

We also harden the loop itself. A naive `while True` agentic loop is a liability — if Claude gets stuck in a tool-calling cycle, your process runs forever. We cap it at 50 iterations. And we solve the context growth problem: every tool result added to the conversation means more tokens sent with every subsequent call. Eventually you hit the model's limit. Context compaction solves this by summarizing old history before it becomes a problem.

Finally, the tests get an upgrade. Until now, tests required a real API key. `FakeBrain` eliminates that dependency — tests are fast, free, and deterministic.

By the end of this chapter you'll have:
- `EditFile` — replace one chunk of text without touching the rest of the file
- `RunCommand` — execute shell commands and return stdout/stderr
- A 50-iteration safety cap on the agentic loop with a clean `for...else` signal
- Context compaction that summarizes before the context window fills
- A `FakeBrain` test double that needs no API key

## The New Tools

### `EditFile` — The Scalpel

Consider the difference between these two approaches to fixing a bug:

**WriteFile approach:** Read 500 lines → send all 500 to Claude → Claude generates all 500 lines back → write 500 lines. Risk: Claude might silently drop a function or change a comment while "copying" code it didn't need to touch.

**EditFile approach:** Tell Claude exactly which text to replace. Claude sends two strings — the old and the new. We do the replacement ourselves. Nothing else changes.

`EditFile` takes three arguments: `path`, `old_text`, and `new_text`.

```python
class EditFile:
    name = "edit_file"
    description = "Replaces specific text in a file. Use for surgical edits instead of rewriting entire files."
    input_schema = {
        "type": "object",
        "properties": {
            "path": {"type": "string", "description": "Path to the file"},
            "old_text": {"type": "string", "description": "Exact text to find and replace"},
            "new_text": {"type": "string", "description": "Text to replace it with"}
        },
        "required": ["path", "old_text", "new_text"]
    }

    def execute(self, context, path, old_text, new_text):
        print(f"  → Editing {path}")
        if context.mode == "plan":
            return "BLOCKED: Cannot edit files in plan mode."

        try:
            with open(path, 'r', encoding='utf-8') as f:
                content = f.read()
            if old_text not in content:
                return f"Error: Could not find the specified text in {path}"
            new_content = content.replace(old_text, new_text, 1)
            with open(path, 'w', encoding='utf-8') as f:
                f.write(new_content)
            return f"Successfully edited {path}"
        except Exception as e:
            return f"Error editing file: {e}"
```

The critical line is:

```python
new_content = content.replace(old_text, new_text, 1)
```

The `1` at the end means "replace only the first occurrence." This is important: if `old_text` appears twice, you don't want to silently change both. If Claude needs to change a second occurrence, it should make a second `edit_file` call — explicit and auditable.

The `if old_text not in content` check comes *before* the replace. If Claude gives us text that doesn't exist in the file — maybe it hallucinated a function signature, or the file changed since it was last read — we return an error immediately. The file is untouched, and Claude gets clear feedback to try again.

**Why exact match?** Because fuzzy matching would let small differences (an extra space, a different quote style) silently fail to match. Exact match makes the contract clear: you must read the file first, then specify *exactly* what you saw. This forces Claude to ground its edits in reality.

```
WriteFile (sledgehammer):          EditFile (scalpel):

  Before:                            Before:
  ┌──────────────┐                   ┌──────────────┐
  │ line 1       │                   │ line 1       │
  │ line 2  ◀───┼──fix this          │ line 2  ◀───┼──fix this
  │ line 3       │                   │ line 3       │
  │ ...500 lines │                   │ ...500 lines │
  └──────────────┘                   └──────────────┘

  Claude gets all 500 lines.           Claude gets:
  Claude rewrites all 500 lines.         old_text = "line 2"
  Risk: silent changes elsewhere.        new_text = "line 2 (fixed)"

                                     Only that text changes.
                                     Everything else is untouched.
```

### `RunCommand` — The Hands

Reading and writing files is passive. Running commands is active — it lets Nanocode execute what it builds.

```python
class RunCommand:
    name = "run_command"
    description = "Executes a terminal command. Use this to run scripts, tests, or install packages. Blocked in plan mode."
    input_schema = {
        "type": "object",
        "properties": {
            "command": {"type": "string", "description": "The shell command to run (e.g., 'python test.py')"}
        },
        "required": ["command"]
    }

    def execute(self, context, command):
        print(f"  → Running: {command[:50]}...")
        if context.mode == "plan":
            return "BLOCKED: Cannot run commands in plan mode. Switch to act mode first."

        try:
            result = subprocess.run(
                command,
                shell=True,
                capture_output=True,
                text=True,
                timeout=int(os.environ.get("NANOCODE_TIMEOUT", "30")),
                cwd=os.getcwd()
            )

            output = ""
            if result.stdout:
                output += f"STDOUT:\n{result.stdout}\n"
            if result.stderr:
                output += f"STDERR:\n{result.stderr}\n"
            if not output:
                output = "(No output)"

            return output.strip()

        except subprocess.TimeoutExpired:
            return "Error: Command timed out."
        except Exception as e:
            return f"Error executing command: {e}"
```

Let's unpack each `subprocess.run` argument:

**`shell=True`** — Passes the command string to the system shell (`/bin/sh` on Unix, `cmd.exe` on Windows). This enables pipelines (`pytest tests/ | head -20`), environment variable expansion (`$HOME`), shell builtins (`cd`, `export`), and glob patterns (`*.py`). Without it, you'd have to pass a list of strings and lose all of that.

Why is `shell=True` considered risky in general? Because it can enable shell injection — if untrusted user input was passed directly into the command string, an attacker could append `;rm -rf /` and it would execute. But Nanocode runs locally, for you, on your machine. The "user" constructing the command is Claude, whose output you're already trusting to write files. The risk profile is the same as any other thing Claude might write.

**`capture_output=True`** — Captures stdout and stderr separately instead of letting them print directly to the terminal. This is essential: Claude needs to *read* the output, not just see it scroll by.

**`text=True`** — Returns stdout/stderr as Python strings (decoded from bytes using the system default encoding) instead of raw `bytes` objects. Saves us a `.decode()` call.

**`timeout=int(os.environ.get("NANOCODE_TIMEOUT", "30"))`** — Commands are limited to 30 seconds by default. Configurable via the `NANOCODE_TIMEOUT` environment variable. Without a timeout, `npm install` on a slow connection, a hung test, or an accidental `sleep 9999` would block the agent indefinitely. The `TimeoutExpired` exception is caught and returns a clean error message to Claude.

**Separate STDOUT and STDERR** — Many tools write normal output to stdout and errors to stderr. A failing pytest run puts the test names to stdout and the tracebacks to stderr. Returning both separately gives Claude full information about what went wrong.

## The Coding Agent Feedback Loop

With `RunCommand` and `EditFile` together, Nanocode can now participate in a genuine development cycle:

```
  ┌─────────────────────────────────────────────────────────────┐
  │                  The Coding Feedback Loop                   │
  │                                                             │
  │   User: "Write tests for utils.py and make them pass"       │
  │                        │                                    │
  │                        ▼                                    │
  │              ┌─────────────────┐                            │
  │              │  WriteFile      │  Claude writes             │
  │              │  test_utils.py  │  test_utils.py             │
  │              └────────┬────────┘                            │
  │                       │                                     │
  │                       ▼                                     │
  │              ┌─────────────────┐                            │
  │              │  RunCommand     │  pytest test_utils.py      │
  │              │  pytest ...     │                            │
  │              └────────┬────────┘                            │
  │                       │                                     │
  │              ┌────────┴────────┐                            │
  │              │                 │                            │
  │         PASS ▼            FAIL ▼                            │
  │    ┌──────────────┐  ┌──────────────────┐                  │
  │    │  Done!       │  │  EditFile        │  fix the bug      │
  │    │  Report to   │  │  (fix the error) │  surgically       │
  │    │  user        │  └────────┬─────────┘                  │
  │    └──────────────┘           │                            │
  │                               │  loop back                 │
  │                               └──▶ RunCommand              │
  │                                    pytest again            │
  └─────────────────────────────────────────────────────────────┘
```

This is the core loop of agentic software development. It's why `RunCommand` matters — without it, the agent can write code but can never verify it works.

## Hardening the Agentic Loop

### The 50-Iteration Cap

The previous agentic loop used `while True`. That's fine until Claude gets stuck — calling a tool, getting a result, calling the same tool again, looping forever. This can happen when:

- A tool returns an error that Claude can't resolve but keeps retrying
- Claude misreads a result and calls another tool based on incorrect assumptions
- A bug in the tool logic causes deterministic failure on every call

The fix is simple: cap the loop.

```python
def _agentic_loop(self):
    output_parts = []
    max_iterations = 50

    for _iteration in range(max_iterations):
        thought = self.brain.think(self.conversation)

        # ... rest of loop body ...

        if not thought.tool_calls:
            break

        # ... execute tools ...

    else:
        output_parts.append("(Stopped: too many iterations)")

    return "\n".join(output_parts)
```

The `for _iteration in range(50)` loop runs at most 50 times. The underscore prefix on `_iteration` signals to readers (and linters) that the variable is intentionally unused — we only care about the count.

### Why `for...else`?

Python's `for...else` is one of its least-known features. The `else` clause on a `for` loop runs **only if the loop completes without hitting a `break`**. If the loop breaks early (normal completion — Claude stopped requesting tools), the `else` is skipped. If the loop runs all 50 iterations without breaking, the `else` runs and appends the warning message.

```
  Iteration 1  ─▶  think()  ─▶  tool calls  ─▶  execute tools
  Iteration 2  ─▶  think()  ─▶  tool calls  ─▶  execute tools
  ...
  Iteration N  ─▶  think()  ─▶  no tool calls  ─▶  BREAK
                                                      │
                                                      │ else clause
                                                      │ is SKIPPED
                                                      ▼
                                                 normal exit

  vs.

  Iteration 1  ─▶  think()  ─▶  tool calls  ─▶  execute tools
  Iteration 2  ─▶  think()  ─▶  tool calls  ─▶  execute tools
  ...
  Iteration 50 ─▶  think()  ─▶  tool calls  ─▶  execute tools
                                                      │
                                              range exhausted
                                              no break occurred
                                                      │
                                                      ▼
                                               else clause RUNS
                                         "(Stopped: too many iterations)"
```

This is more elegant than tracking a boolean flag:

```python
# Without for...else (more verbose):
exhausted = True
for _iteration in range(max_iterations):
    ...
    if not thought.tool_calls:
        exhausted = False
        break
if exhausted:
    output_parts.append("(Stopped: too many iterations)")
```

The `for...else` expresses the same logic with less noise.

## Context Compaction

### The Problem

Every time Claude responds, we append its response to `self.conversation`. Every time a tool runs, we append the result. This is correct — Claude needs the full history to maintain coherent reasoning. But it accumulates without bound.

Claude's context window is 200,000 tokens. A single tool result (like the output of a large file read) might be 2,000 tokens. Fifty tool calls in a session = 100,000 tokens of results, plus the original messages, plus Claude's responses. You can hit 75% of the limit faster than you'd expect.

When the limit is exceeded, the API returns an error. Your agent crashes mid-task.

### The Solution

Before the conversation grows too large, summarize it. Replace the growing history with a single summary message. The agent continues from there with a fresh context but preserved understanding of what it was doing.

```
Before compaction:
┌─────────────────────────────────────────────────────────┐
│ conversation                                            │
│ ┌──────────────────────┐                               │
│ │ user: "Add logging"  │  ← message 1                  │
│ │ assistant: [plan]    │  ← message 2                  │
│ │ user: [tool result]  │  ← message 3                  │
│ │ assistant: [edit]    │  ← message 4                  │
│ │ ...                  │                               │
│ │ 180,000 tokens used  │  ← 90% of 200k limit          │
│ └──────────────────────┘                               │
└─────────────────────────────────────────────────────────┘

After _compact_conversation():
┌─────────────────────────────────────────────────────────┐
│ conversation                                            │
│ ┌──────────────────────────────────────────────────┐   │
│ │ user: "Previous conversation summary: We added   │   │
│ │ logging to utils.py. Created log_utils.py with   │   │
│ │ a Logger class. Tests pass. Next: add to main."  │   │
│ └──────────────────────────────────────────────────┘   │
│ ~800 tokens used  ← fresh start                        │
└─────────────────────────────────────────────────────────┘
```

Here is the full implementation:

```python
def _compact_conversation(self):
    """Summarize old messages to stay within context limits."""
    print("(Compacting conversation...)")
    history = "\n".join(
        f"{m['role']}: {str(m['content'])[:500]}"
        for m in self.conversation
    )
    prompt = [{
        "role": "user",
        "content": f"Summarize this conversation for continuity. "
                   f"Focus on what was accomplished, what's in progress, "
                   f"and key decisions:\n\n{history}"
    }]
    thought = self.brain.think(prompt)
    self.conversation = [
        {"role": "user", "content": f"Previous conversation summary: {thought.text}"},
    ]
```

Step by step:

1. **Build a text snapshot of history.** Each message is rendered as `role: content`, capped at 500 characters to avoid sending an enormous prompt for the summary call itself. This is a pragmatic approximation — the goal is to give Claude enough context to write a good summary, not to replay the full history.

2. **Ask Claude to summarize.** This is a fresh API call with a new one-message conversation — it doesn't use `self.conversation`. The summary prompt explicitly asks for what was accomplished, what's in progress, and key decisions. These three categories cover what the agent needs to continue coherently.

3. **Replace history with the summary.** `self.conversation` is set to a single user message containing the summary. From Claude's perspective, the next call will look like the user is starting with a helpful status update.

### When Does Compaction Trigger?

Compaction is triggered after each `think()` call:

```python
thought = self.brain.think(self.conversation)

if self.brain.last_input_tokens > self.brain.context_limit * 0.75:
    self._compact_conversation()
```

`last_input_tokens` is updated by `think()` from the API response's `usage.input_tokens` field. The threshold is 75% of the context limit.

Why 75%? Because the compaction call itself needs token budget. If we waited until 99% and then made a summarization call, the summary call might also exceed the limit. At 75%, there's a comfortable 50,000-token buffer for the summarization call and the response.

The `Brain` base class now holds these two attributes:

```python
class Brain:
    context_limit = 200_000  # Overridden per-instance in subclasses
    last_input_tokens = 0    # Updated after each think() call
```

`Claude` keeps the 200,000 default. `DeepSeek` overrides with 128,000. When we add `Ollama` in Chapter 10, it will use 32,000 — or whatever the model actually reports.

## Testing Without an API Key: `FakeBrain`

Until now, every test that touched the agent required real API calls — slow, expensive, and non-deterministic. `FakeBrain` eliminates all three problems.

```python
class FakeBrain:
    """Fake brain for testing - returns predictable responses."""
    context_limit = 200_000
    last_input_tokens = 0

    def __init__(self, responses=None, memory=None, tools=None):
        self.memory = memory
        self.tools = tools or []
        self.responses = responses or [
            Thought(
                text="Fake response",
                raw_content=[{"type": "text", "text": "Fake response"}]
            )
        ]
        self.call_count = 0

    def think(self, conversation):
        if self.call_count < len(self.responses):
            response = self.responses[self.call_count]
            self.call_count += 1
            return response
        return Thought(
            text="No more responses",
            raw_content=[{"type": "text", "text": "No more responses"}]
        )
```

`FakeBrain` implements the same interface as `Claude` and `DeepSeek` — it has `context_limit`, `last_input_tokens`, and a `think()` method that returns a `Thought`. The agent doesn't know or care that it's fake.

You pre-load it with a list of `Thought` objects. Each call to `think()` pops the next one. When the list is exhausted, it returns a `"No more responses"` sentinel so tests fail loudly instead of silently looping.

This pattern is called a **test double** — a stand-in object that mimics the real thing's interface without its real behavior. It lets you write tests like:

```python
def test_compact_conversation_summarizes():
    summary = Thought(
        text="Summary of conversation",
        raw_content=[{"type": "text", "text": "Summary of conversation"}]
    )
    responses = [
        Thought(text="Response 1", raw_content=[...]),
        summary,
    ]
    brain = FakeBrain(responses=responses)
    brain.last_input_tokens = 200_000  # Simulate hitting 75% threshold
    agent = Agent(brain=brain, tools=tools, mode="act")

    agent.handle_input("Do something")

    assert len(agent.conversation) <= 4
    assert "Summary" in str(agent.conversation[0]["content"])
```

This test verifies context compaction — a complex, multi-step behavior — without a single API call. It runs in milliseconds. It produces the same result every time.

**Why does this matter?** Consider what running tests against real Claude would look like:
- Each test run costs money (token fees)
- Tests take 5-10 seconds each (API latency)
- Claude might respond differently each run (non-determinism)
- You can't run tests on a plane or in a region where Anthropic is blocked

`FakeBrain` is how you write a test suite you'll actually want to run.

## Summary

| Concept | Key Detail |
|---------|-----------|
| `EditFile` | Replaces first occurrence of `old_text` with `new_text`. Exact match required. Blocked in plan mode. |
| `RunCommand` | `subprocess.run(shell=True, capture_output=True, text=True, timeout=30)`. Returns STDOUT and STDERR. Blocked in plan mode. |
| `NANOCODE_TIMEOUT` | Environment variable to override the default 30-second command timeout. |
| Iteration cap | `for _iteration in range(50)` instead of `while True`. Prevents infinite loops. |
| `for...else` | The `else` clause runs only when the loop exhausts without a `break`. Elegant exhaustion detection. |
| Context compaction | Triggers at 75% of `context_limit`. Summarizes history, replaces conversation with one message. |
| `Brain.last_input_tokens` | Updated from `usage.input_tokens` after each `think()`. Used to trigger compaction. |
| `FakeBrain` | Pre-scripted `Thought` responses. Implements `Brain` interface. Enables fast, free, deterministic tests. |
