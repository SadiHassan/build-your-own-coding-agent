# Chapter 10: Local Brains — Running LLMs on Your Own Machine

> *"A library that requires an internet connection to open is not really yours. A model that runs on your machine is."*

## What This Chapter Is About

Every API call so far has left your machine. Your code, your questions, Claude's answers — all of it travels to Anthropic's servers in the cloud, gets processed, and comes back. For most work, that's fine. But sometimes it isn't:

- **Privacy.** Your codebase might contain secrets, unreleased features, or proprietary algorithms you can't send to a third party.
- **Cost.** A long agentic session with many tool calls generates thousands of tokens. That adds up.
- **Availability.** Planes, basements, conference hotel Wi-Fi — connectivity is not guaranteed.

This chapter adds a third brain: **Ollama**. Ollama is a tool that downloads and runs LLMs locally on your machine. Once a model is downloaded, it runs without internet. Your code never leaves your laptop.

The remarkable part: Ollama implements the same HTTP wire protocol as Anthropic. The `_parse_response` method, the `request_with_retry` helper, the `Thought` and `ToolCall` classes — none of that changes. We write one new class, add it to the `BRAINS` registry, and everything works.

By the end of this chapter you'll have:
- `Ollama(Brain)` — a third brain that talks to a local model
- `_detect_context_limit()` — dynamically queries the model's actual context window
- `BRAINS["ollama"]` — switchable via `/switch` or `NANOCODE_BRAIN=ollama`
- A working local agent that needs no API key

## What Is Ollama?

Ollama is a program that runs on your machine and manages local LLM models. Think of it as Docker, but for AI models instead of containers.

```
Without Ollama (cloud):             With Ollama (local):

  Your Machine                        Your Machine
  ┌──────────────────┐                ┌────────────────────────────────┐
  │                  │                │                                │
  │  nanocode.py     │                │  nanocode.py                   │
  │       │          │                │       │                        │
  │       │ HTTP     │                │       │ HTTP                   │
  │       ▼          │                │       ▼                        │
  │  ─────────────── │                │  localhost:11434               │
  │  INTERNET ───────┼──▶ Anthropic   │  ┌──────────────────────────┐  │
  │  ─────────────── │    API Server  │  │  Ollama                  │  │
  │                  │    (USA/EU)    │  │  ┌────────────────────┐  │  │
  └──────────────────┘                │  │  │  qwen3-coder:30b   │  │  │
                                      │  │  │  (local model)     │  │  │
  Code leaves your machine.           │  │  └────────────────────┘  │  │
  Anthropic sees your prompts.        │  └──────────────────────────┘  │
  Billed per token.                   │                                │
  Needs internet.                     └────────────────────────────────┘

                                      Code stays on your machine.
                                      No third party sees your prompts.
                                      No token fees.
                                      Works offline.
```

Ollama downloads models once (they're large — `qwen3-coder:30b` is about 20 GB) and then serves them locally. The API it exposes is intentionally compatible with Anthropic's API, which means code written for Anthropic can talk to Ollama with minimal changes.

## Setting Up Ollama

### Install

```bash
brew install ollama
```

On Linux, use the official install script:

```bash
curl -fsSL https://ollama.ai/install.sh | sh
```

### Start the Server

```bash
ollama serve
```

This starts the Ollama API server on `localhost:11434`. Keep this terminal open (or run it as a background service).

### Pull a Model

```bash
ollama pull qwen3-coder:30b
```

This downloads the model. It's a one-time download of roughly 20 GB. The default model in Nanocode is `qwen3-coder:30b`, which is a strong coding model. You can override it:

```bash
OLLAMA_MODEL=llama3.1:8b python nanocode.py --act
```

Smaller models (like `llama3.1:8b` at ~5 GB) are faster but less capable. Larger models are slower but produce better code. Start with whatever fits comfortably in your machine's RAM.

### Verify It Works

```bash
ollama run qwen3-coder:30b "Write a Python hello world"
```

If you see a response, you're ready to connect Nanocode.

## The `Ollama` Class

Here is the complete implementation:

```python
class Ollama(Brain):
    """Ollama local model (Anthropic-compatible API, with tool support)."""
    context_limit = 32_000

    def __init__(self, memory=None, tools=None):
        self.memory = memory
        self.tools = tools or []
        self.model = os.getenv("OLLAMA_MODEL", "qwen3-coder:30b")
        self.url = "http://localhost:11434/v1/messages"
        self._detect_context_limit()

    def _detect_context_limit(self):
        """Query Ollama for the model's context window size."""
        try:
            response = requests.post(
                "http://localhost:11434/api/show",
                json={"model": self.model},
                timeout=2
            )
            model_info = response.json().get("model_info", {})
            for key, value in model_info.items():
                if key.endswith(".context_length"):
                    self.context_limit = value
                    return
        except Exception:
            pass  # Keep default

    def think(self, conversation):
        headers = {
            "x-api-key": "ollama",
            "anthropic-version": "2023-06-01",
            "content-type": "application/json"
        }
        payload = {
            "model": self.model,
            "max_tokens": 4096,
            "messages": conversation
        }
        if self.memory:
            payload["system"] = self.memory.content
        if self.tools:
            payload["tools"] = self.tools

        print(f"({self.model} is thinking...)")
        response = request_with_retry(self.url, headers, payload)
        data = response.json()
        self.last_input_tokens = data.get("usage", {}).get("input_tokens", 0)
        return self._parse_response(data["content"])

    def _parse_response(self, content):
        """Convert Anthropic response format to Thought."""
        text_parts = []
        tool_calls = []

        for block in content:
            if block["type"] == "text":
                text_parts.append(block["text"])
            elif block["type"] == "tool_use":
                tool_calls.append(ToolCall(
                    id=block["id"],
                    name=block["name"],
                    args=block["input"]
                ))

        return Thought(
            text="\n".join(text_parts) if text_parts else None,
            tool_calls=tool_calls,
            raw_content=content
        )
```

If you look at `Claude.think()` and `Ollama.think()` side by side, the differences are:

| | `Claude` | `Ollama` |
|---|---|---|
| URL | `https://api.anthropic.com/v1/messages` | `http://localhost:11434/v1/messages` |
| `x-api-key` | Real API key from env var | Hard-coded `"ollama"` |
| `context_limit` | `200_000` (inherited from `Brain`) | `32_000` (class attribute) |
| `_detect_context_limit` | Not present | Called in `__init__` |
| `_parse_response` | Identical | Identical |

The entire API communication layer — `request_with_retry`, `_parse_response`, token tracking — is reused unchanged. This is the payoff for building against a clean interface from the start.

## Code Walkthrough: `_detect_context_limit`

Different Ollama models have different context windows. `qwen3-coder:30b` may support 32k tokens. A future model might support 128k. If we hardcode `32_000` and a model actually supports 128k, compaction will trigger far too early, producing unnecessary overhead. If the model only supports 8k and we assume 32k, the agent will feed it more than it can handle.

`_detect_context_limit` queries Ollama at startup to find the actual value:

```python
def _detect_context_limit(self):
    try:
        response = requests.post(
            "http://localhost:11434/api/show",
            json={"model": self.model},
            timeout=2
        )
        model_info = response.json().get("model_info", {})
        for key, value in model_info.items():
            if key.endswith(".context_length"):
                self.context_limit = value
                return
    except Exception:
        pass  # Keep default
```

Step by step:

1. **POST to `/api/show`.** This is Ollama's metadata endpoint. It returns detailed information about the currently loaded model, including architecture details, quantization, and parameters.

2. **Look in `model_info`.** The response contains a `model_info` dict with keys like `llama.context_length` or `qwen2.context_length` — the prefix varies by model architecture. We don't know the prefix ahead of time, so we iterate over all keys.

3. **Find the key ending in `.context_length`.** The `endswith(".context_length")` check handles any architecture prefix — `llama.context_length`, `qwen2.context_length`, `gemma.context_length`, etc.

4. **Update `self.context_limit` and return.** If found, we set the instance attribute (overriding the class default) and return immediately.

5. **Fail gracefully.** The entire body is wrapped in `try/except Exception`. If Ollama isn't running, if the model isn't loaded, if the response format changes — any of these raises an exception that we silently catch. `self.context_limit` stays at the class default of `32_000`. The agent continues to work; compaction is just slightly conservative.

The `timeout=2` is intentionally short. This is a startup check, not a long-running operation. If Ollama doesn't respond in 2 seconds, it's either not running or the model isn't loaded — either way, we want to fail fast and proceed with the default.

```
  Ollama.__init__()
        │
        ├── self.model = os.getenv("OLLAMA_MODEL", "qwen3-coder:30b")
        │
        ├── self.url = "http://localhost:11434/v1/messages"
        │
        └── _detect_context_limit()
                │
                ▼
         POST /api/show {"model": "qwen3-coder:30b"}
                │
       ┌────────┴──────────────────────┐
       │                               │
  Ollama running?                Ollama not running?
       │                               │
       ▼                               ▼
  model_info = {                  Exception caught
    "qwen2.context_length": 32768   self.context_limit = 32_000
    "qwen2.attention.head_count": 40   (default, unchanged)
    ...
  }
       │
       ▼
  Find key ending ".context_length"
       │
       ▼
  self.context_limit = 32768
```

## The Wire Protocol: Why This Works

Ollama implements the Anthropic Messages API format. The request and response JSON structures are identical to what you'd send to `api.anthropic.com`. This is a deliberate choice by Ollama's developers — it means any code written for Anthropic can be pointed at Ollama by changing the URL.

The `x-api-key: "ollama"` header deserves an explanation. The Anthropic API specification requires the `x-api-key` header to be present. Ollama mimics this requirement — it validates that the header exists, but doesn't actually validate the value. Any non-empty string works. We use `"ollama"` as a readable placeholder.

```
Anthropic API call:                    Ollama API call:

POST https://api.anthropic.com         POST http://localhost:11434
     /v1/messages                           /v1/messages
x-api-key: sk-ant-api03-...            x-api-key: ollama
anthropic-version: 2023-06-01          anthropic-version: 2023-06-01
content-type: application/json         content-type: application/json

{"model": "claude-sonnet-4-6",         {"model": "qwen3-coder:30b",
 "max_tokens": 4096,                    "max_tokens": 4096,
 "messages": [...],                     "messages": [...],
 "tools": [...]}                        "tools": [...]}

          │                                        │
          ▼                                        ▼
  Response: same JSON format             Response: same JSON format
  {"content": [...], "usage": {...}}     {"content": [...], "usage": {...}}
```

Both responses are parsed by the same `_parse_response` method. Both token counts are read from the same `usage.input_tokens` field. The compaction logic, the tool dispatch, the conversation management — nothing changes.

## The `BRAINS` Registry

The registry now holds three entries:

```python
BRAINS = {
    "claude": Claude,
    "deepseek": DeepSeek,
    "ollama": Ollama,
}
```

When the user types `/switch`, `_switch_brain()` cycles through this dict in insertion order. With three brains, the cycle is: `claude` → `deepseek` → `ollama` → `claude` → ...

```
  BRAINS registry:
  ┌──────────────────────────────────────────────────────────┐
  │                                                          │
  │   "claude"    ──▶  Claude class                          │
  │       │              URL: api.anthropic.com              │
  │       │              context_limit: 200,000              │
  │       │              auth: ANTHROPIC_API_KEY             │
  │       │                                                  │
  │   "deepseek"  ──▶  DeepSeek class                        │
  │       │              URL: api.deepseek.com               │
  │       │              context_limit: 128,000              │
  │       │              auth: DEEPSEEK_API_KEY              │
  │       │                                                  │
  │   "ollama"    ──▶  Ollama class                          │
  │                     URL: localhost:11434                 │
  │                     context_limit: 32,000 (or detected) │
  │                     auth: "ollama" (placeholder)        │
  │                                                          │
  └──────────────────────────────────────────────────────────┘

  /switch cycles:  claude  →  deepseek  →  ollama  →  claude  →  ...

  NANOCODE_BRAIN=ollama  selects directly on startup.
```

You can also select a brain at startup using the environment variable:

```bash
NANOCODE_BRAIN=ollama python nanocode.py --act
```

This sets `brain_name = os.getenv("NANOCODE_BRAIN", "claude")` in `main()`, which is then used to instantiate the right class from the registry. The rest of `main()` is unchanged.

## Context Window Trade-offs

Local models typically have smaller context windows than cloud models. This affects how aggressively compaction fires:

```
  Context windows across brains:

  Claude (200k)   ████████████████████████████████████████  200,000 tokens
                                             ▲
                                         compaction at 75%: 150,000 tokens

  DeepSeek (128k) ████████████████████████████             128,000 tokens
                                    ▲
                                compaction at 75%: 96,000 tokens

  Ollama (32k)    ████████                                   32,000 tokens
                      ▲
                  compaction at 75%: 24,000 tokens
```

With a 32k context window, compaction fires much earlier. A single large file read might consume 10,000 tokens — meaning three file reads and you're already near the threshold. This is a real constraint with local models.

In practice, this means:
- With Ollama, prefer `edit_file` over `write_file` (fewer tokens in tool results)
- Prefer `search_codebase` to locate the right section before `read_file`
- Be aware that long conversations will compact more frequently

The compaction mechanism from Chapter 9 handles this automatically. You don't need to think about it manually — but understanding the constraint helps you set expectations about session length.

```
[claude:act] ❯ /switch

...
```

The first request to a newly loaded model is slower while Ollama loads the model weights into memory. Subsequent requests are faster.

## Troubleshooting

**`Connection refused` or `Network error`**
Ollama isn't running. Start it with `ollama serve` in another terminal.

**`model not found`**
The model hasn't been pulled yet. Run `ollama pull qwen3-coder:30b` (or whatever model you've set in `OLLAMA_MODEL`).

**Slow responses**
Large models are slower on CPU. If you have a GPU, Ollama will use it automatically if the model fits in VRAM. For CPU-only machines, smaller models (`llama3.1:8b`, `qwen2.5-coder:7b`) are significantly faster.

**Model runs out of context mid-task**
The model's actual context limit is smaller than the default 32k, and `_detect_context_limit` didn't find it. Try a smaller conversation, or check `ollama show <model>` to find the real context length and set `context_limit` manually.
