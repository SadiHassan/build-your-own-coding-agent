## What the Agent Does

The agent doesn't immediately write code. First, it searches the web:

```
  → Searching web for 'pygame snake game Python tutorial 2024'
  → Searching web for 'pygame install current version'
```

It's checking current documentation. It wants to know the up-to-date pygame API, not rely on potentially outdated training data. This is `SearchWeb` working exactly as intended.

Then it lists your current directory structure to understand the context:

```
  → Listing .
```

Then it starts writing. Not the game itself — the structure first. It creates `snake_game/`, writes a `requirements.txt` with the pygame dependency, writes a `README.md` explaining setup and controls.

Then the main game file. This is where the agentic loop earns its keep. Claude writes `snake_game/snake.py` — several hundred lines covering the game loop, snake movement, collision detection, food respawning, and score rendering. Then it runs it:

```
  → Running: cd snake_game && pip install -r requirements.txt
  → Running: python -m pytest snake_game/ -v 2>&1 || python snake_game/snake.py --test
```

If an import fails because pygame isn't installed correctly, it reads the error and adjusts. If there's a runtime error, it reads the traceback, finds the file and line, and edits the specific section. It doesn't rewrite the whole file — it uses `EditFile` to make the precise change.

After one or two iterations of this loop, the game runs.

---

## Watching the Loop

What you're watching when the agent works through errors is the feedback loop from Chapter 9 in action. Each iteration is:

1. Run the code
2. Read the output
3. Identify the problem
4. Make a targeted fix
5. Run again

This cycle — which a developer does manually dozens of times per day — happens autonomously. The agent doesn't wait for you to paste in the error message. It ran the command, read the error, and fixed it before you could finish your coffee.

The difference between this and a chatbot is precisely the tools. Without `RunCommand`, the agent couldn't execute code. Without `ReadFile`, it couldn't see the codebase it's working in. Without `EditFile`, every fix would be a full file rewrite. Without `SearchWeb`, it might use a deprecated API. Every tool we added across the chapters is doing work here.

## The Finished Game

When the agent is done, `snake_game/` will contain:

```
snake_game/
    snake.py          — the game
    requirements.txt  — pygame dependency
    README.md         — setup instructions
```

The `README.md` will explain how to install dependencies and run the game. The code will be clean Python with comments — the agent writes readable code because its training has exposed it to a lot of readable code, and because you can ask it to add comments if you want.

To play it:

```bash
cd snake_game
pip install -r requirements.txt
python snake.py
```

Arrow keys to move. Score increments when the snake eats food. Game over when the snake hits a wall or itself. Everything the brief asked for.

## Where to Go Next

The agent you have is complete but not finished. Here are directions worth exploring:

**New tools.** A `FetchURL` tool that retrieves and parses a webpage. A `GitTool` that can commit changes, create branches, read diffs. A `DatabaseTool` that can run SQL queries. Every tool you add expands what the agent can do without changing the Agent or Brain code.

**New brains.** Google's Gemini and OpenAI's GPT-4 both support compatible wire protocols or have thin wrappers. Adding a new brain is a new class and one line in the `BRAINS` dictionary.

**Better memory.** The current memory system is a flat Markdown file. You could replace it with a structured format, add versioning, or build a retrieval system that pulls relevant memories based on the current task rather than sending everything every time.

**Streaming.** The current implementation waits for the complete response before displaying anything. The Anthropic API supports streaming — receiving and displaying tokens as they're generated. The agent would feel more responsive.

**A web interface.** The current UI is a terminal. The `handle_input()` interface is clean enough that wrapping it in a FastAPI endpoint and a simple frontend is not much work.

But none of these require rethinking the architecture. They're extensions of what you already have. That's the sign of a design that worked.
