# Chapter 5: Tools — Giving the Agent Hands

Until now, our agent has been a sophisticated chatbot. It remembers conversations, it retries on failure, it can switch providers. But it cannot *do* anything. It can tell you what a Python function should look like, but it cannot write the file. It can suggest a fix for a bug, but it cannot apply the edit. It can reason endlessly about your codebase, but it has never actually seen it.

This chapter changes everything. We add tools — the mechanism that lets Claude reach out of the conversation and interact with your actual filesystem. More importantly, we introduce the agentic loop: the engine that keeps running until every tool call is complete. After this chapter, the agent isn't just a chatbot. It's an autonomous worker.

---

## What a Tool Is

A tool is a Python class that represents a capability the agent can use. It has three pieces of metadata — a name, a description, and a schema — plus an `execute` method that does the actual work.

```python
class ReadFile:
    name = "read_file"
    description = "Reads a file from the filesystem. Use this to examine code."
    input_schema = {
        "type": "object",
        "properties": {
            "path": {"type": "string", "description": "The relative path to the file"}
        },
        "required": ["path"]
    }

    def execute(self, path):
        with open(path, 'r', encoding='utf-8') as f:
            lines = f.readlines()
        numbered_lines = [f"{i+1} | {line}" for i, line in enumerate(lines)]
        return "".join(numbered_lines)
```

The metadata exists for Claude's benefit. The `name` is how Claude refers to the tool. The `description` tells Claude *when* to use it — Claude reads the description to decide whether this tool is relevant to the current task. The `input_schema` is a JSON Schema definition that tells Claude *how* to call it — what arguments to provide, what types they should be, which ones are required.

These three attributes are defined at the class level, not in `__init__`. That's intentional: they belong to the class itself, not to any individual instance. Every `ReadFile` instance shares the same name, description, and schema. There's no reason to create them fresh for each instance.

The `execute` method is yours. It runs on your machine with access to your files, your processes, your network. Claude never executes this code directly — it asks you to run it and reports the result back. That distinction is crucial. Claude is the brain. Your code is the body.

## Describing Tools to the API

Before Claude can use a tool, it needs to know the tools exist. We send the tool definitions as part of the API request:

```python
def tool_definitions(tools):
    return [
        {"name": t.name, "description": t.description, "input_schema": t.input_schema}
        for t in tools
    ]
```

This function converts our Python tool objects into the JSON format the API expects. When we call `brain.think(conversation)`, the brain includes these definitions in the payload:

```python
if self.tools:
    payload["tools"] = self.tools   # the list of JSON definitions
```

Claude reads the definitions and decides whether to use any of them. If it decides to use one, its response will include a `tool_use` block alongside (or instead of) the usual text.

## The raw_content Problem

Here is where something subtle happens that we need to handle carefully.

When Claude wants to call a tool, its API response looks like this:

```json
"content": [
  {"type": "text", "text": "I'll read that file for you."},
  {
    "type": "tool_use",
    "id": "toolu_abc123",
    "name": "read_file",
    "input": {"path": "main.py"}
  }
]
```

The response has two blocks: a text block and a `tool_use` block. The `tool_use` block has a unique `id` field. When we report the tool result back to Claude, we must reference that same `id` — that's how Claude matches the result to the request.

This creates a problem. In Chapter 3, when we stored Claude's reply in the conversation, we stored just the text string:

```python
self.conversation.append({"role": "assistant", "content": text})
```

If we do that here, the `tool_use` block — and its `id` — is lost. On the next API call, when we send the tool result back with `"tool_use_id": "toolu_abc123"`, Claude will have no record of ever requesting that tool. The API will reject it or behave unpredictably.

The fix is to store the *original content list* from the API response, not just the extracted text:

```python
class Thought:
    def __init__(self, text=None, tool_calls=None, raw_content=None):
        self.text = text
        self.tool_calls = tool_calls or []
        self.raw_content = raw_content   # the original content list
```

And when we append to the conversation:

```python
self.conversation.append({"role": "assistant", "content": thought.raw_content})
```

Now the full content list — text block, tool_use block, id and all — is preserved in the history exactly as Claude produced it.

## The Agentic Loop

With tools available, a single call to `brain.think()` is no longer sufficient. Claude might request a file, read its contents, then want to request another file based on what it found. Then it might want to write a modified version. Each of those is a separate tool call, each requiring a round trip.

The agentic loop handles this automatically:

```python
def _agentic_loop(self):
    output_parts = []

    while True:
        thought = self.brain.think(self.conversation)

        # Store the full response, preserving tool_use blocks
        self.conversation.append({"role": "assistant", "content": thought.raw_content})

        if thought.text:
            output_parts.append(thought.text)

        # If Claude didn't request any tools, we're done
        if not thought.tool_calls:
            break

        # Execute each tool Claude requested
        tool_results = []
        for tool_call in thought.tool_calls:
            result = self._execute_tool(tool_call.name, tool_call.args)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": tool_call.id,
                "content": result
            })

        # Feed the results back as a user message
        self.conversation.append({"role": "user", "content": tool_results})

        # Loop: Claude will read the results and continue
```

The loop structure is simple. Think. If no tool calls, break. If there are tool calls, execute them, feed the results back, and think again. Repeat until Claude produces a response with no tool calls — which means it's satisfied and has a final answer.

```
┌──────────────────────────────────────────────────────┐
│                THE AGENTIC LOOP                      │
│                                                      │
│   brain.think(conversation)                          │
│          │                                           │
│          ▼                                           │
│   tool_calls empty? ──yes──▶ break, return output   │
│          │ no                                        │
│          ▼                                           │
│   execute each tool                                  │
│          │                                           │
│          ▼                                           │
│   append results to conversation                     │
│          │                                           │
│          └──────────────────── loop again            │
└──────────────────────────────────────────────────────┘
```

## Why Tool Results Are "User" Messages

The tool results go back into the conversation with `"role": "user"`. This might look strange — you're not the user, the tools are. But from Claude's perspective, the distinction is simple: `"assistant"` means things Claude said, and `"user"` means everything that comes from outside Claude. Tool results come from your program — they're external information arriving from outside the model. So they travel as user messages.

This also keeps the conversation alternation intact. API rules require user and assistant messages to alternate. Delivering tool results as user messages satisfies that constraint naturally.

## Executing Tools

The `_execute_tool` method is a dispatcher: find the right tool by name, call it with the given arguments.

```python
def _execute_tool(self, name, args):
    tool = get_tool(self.tools, name)
    if tool is None:
        return f"Error: Tool '{name}' not found"
    try:
        return tool.execute(**args)
    except TypeError as e:
        return f"Error: Invalid arguments - {e}"
```

The `get_tool` helper uses a generator expression to find the matching tool by name:

```python
def get_tool(tools, name):
    return next((t for t in tools if t.name == name), None)
```

Notice that `_execute_tool` never raises an exception. It catches errors and returns them as strings. This is intentional: Claude receives the error message as a tool result and can respond to it — perhaps by trying a different approach or asking you to fix the problem. An unhandled exception would crash the agentic loop entirely, losing all context.

## WriteFile

The second tool is `WriteFile`, which lets Claude create or overwrite files:

```python
class WriteFile:
    name = "write_file"
    description = "Writes content to a file. OVERWRITES existing content."
    input_schema = {
        "type": "object",
        "properties": {
            "path":    {"type": "string", "description": "The relative path to the file"},
            "content": {"type": "string", "description": "The full content to write"}
        },
        "required": ["path", "content"]
    }

    def execute(self, path, content):
        with open(path, 'w', encoding='utf-8') as f:
            f.write(content)
        return f"Successfully wrote {len(content)} characters to {path}"
```

The description says "OVERWRITES existing content" in capitals. This is deliberate — Claude reads tool descriptions to understand their effects. Making the destructive behavior explicit helps Claude decide when to use `WriteFile` versus a more surgical edit. We'll add `EditFile` in Chapter 9 for cases where overwriting the whole file isn't appropriate.

## A Session in Motion

Here is what an agent session looks like now that tools are involved. The user asks Claude to read a file and explain it.

Claude's first response includes a `tool_use` block: it wants to call `read_file` with the path you mentioned. The loop executes `ReadFile.execute()` on your machine, gets back the numbered file contents, and appends them as a tool result. Claude reads that result and, in its second response, explains what the file does. That response has no tool calls. The loop breaks.

The user sees only the final explanation. The tool calls happen silently in the background — unless you look at the `"(Claude is thinking...)"` print statements, you wouldn't know they occurred.

This is the quality that makes an agent feel intelligent: it gathers information autonomously, processes it, and delivers a considered response. The user doesn't need to paste file contents into the chat. The agent fetches what it needs.
