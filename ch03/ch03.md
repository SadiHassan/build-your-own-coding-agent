# Chapter 3: The Brain — Memory and the Claude Class

> *"A goldfish has no memory of last Tuesday. A good assistant does. The difference is a list."*

## What This Chapter Is About

In Chapter 1 we built the skeleton: a loop that accepts input and returns output. In Chapter 2 we made a raw API call and studied every field Claude returns. Now we connect the two.

This chapter introduces three things that will stay with us for the rest of the book:

1. The **`Claude` class** — a dedicated "brain" object that owns the API call
2. The **`conversation` list** — the agent's memory, stored as a list of `{role, content}` dicts
3. The **`Thought` return type** — a standardized envelope for everything the brain sends back

When this chapter is done, Nanocode will hold a real conversation. Ask it something, get a reply, ask a follow-up — it will remember what you said before.

## The Conversation List — This IS the Memory

Before we look at any code, we need to understand the central data structure.

The Anthropic API has no concept of a "session" or a "conversation ID." Every request is stateless from the server's point of view. When you want Claude to remember what was said earlier, you have to send it again — in full, every time.

The `conversation` list is exactly that: the complete record of everything said so far, passed to the API on each call.

```
Turn 1                        Turn 2                        Turn 3
──────                        ──────                        ──────
You send:                     You send:                     You send:
  [                             [                             [
    {role:"user",                 {role:"user",                 {role:"user",
     content:"hi"}                content:"hi"},                content:"hi"},
  ]                               {role:"assistant",            {role:"assistant",
                                   content:"Hello!"},            content:"Hello!"},
                                  {role:"user",                 {role:"user",
                                   content:"my name                content:"my name
                                    is Alice"}                       is Alice"},
                                ]                               {role:"assistant",
                                                                 content:"Nice to
                                                                  meet you Alice"},
                                                                {role:"user",
                                                                 content:"what is
                                                                  my name?"}
                                                              ]
```

Every API call sends the whole history. Claude reads it top to bottom, exactly as it arrived, and answers with full context. By Turn 3, Claude knows your name — not because there's a database, not because the server has state — but because your code re-sent the earlier turns.

The conversation list is a list of Python dicts. Each dict has exactly two keys:

```
{"role": "user",      "content": "...what the user typed..."}
{"role": "assistant", "content": "...what Claude replied..."}
```

Roles must strictly alternate: `user`, `assistant`, `user`, `assistant`, ... The API will reject a request that starts with `assistant` or has two `user` turns in a row.

### Memory grows linearly

```
┌─────────────────────────────────────────────────────────────────────┐
│               conversation list grows with every turn               │
│                                                                     │
│  After turn 1:  [ U ]                                               │
│                   │                                                 │
│                   └─ user message                                   │
│                                                                     │
│  After turn 1:  [ U ][ A ]                                          │
│                   │     │                                           │
│                   │     └─ assistant reply (appended after API call) │
│                   └─ user message                                   │
│                                                                     │
│  After turn 2:  [ U ][ A ][ U ][ A ]                                │
│                                                                     │
│  After turn N:  [ U ][ A ][ U ][ A ] ... [ U ][ A ]                 │
│                                                                     │
│  U = {"role": "user",      "content": "..."}                        │
│  A = {"role": "assistant", "content": "..."}                        │
│                                                                     │
│  The ENTIRE list is re-sent to the API on every call.               │
└─────────────────────────────────────────────────────────────────────┘
```

This has a practical consequence: long conversations get expensive. The API charges per token, and every input token includes all previous turns. Chapter 10 will address this with context compaction. For now, the list just grows.

## The `Claude` Class — The Brain Abstraction

```python
class Claude:
    """Claude API - the brain of our agent."""

    def __init__(self):
        self.api_key = os.getenv("ANTHROPIC_API_KEY")
        if not self.api_key:
            raise ValueError("ANTHROPIC_API_KEY not found in .env")
        self.model = "claude-sonnet-4-6"
        self.url = "https://api.anthropic.com/v1/messages"

    def think(self, conversation):
        headers = {
            "x-api-key": self.api_key,
            "anthropic-version": "2023-06-01",
            "content-type": "application/json"
        }
        payload = {
            "model": self.model,
            "max_tokens": 4096,
            "messages": conversation
        }

        print("(Claude is thinking...)")
        response = requests.post(self.url, headers=headers, json=payload, timeout=120)
        response.raise_for_status()
        return self._parse_response(response.json()["content"])

    def _parse_response(self, content):
        """Convert Claude's response format to Thought."""
        text_parts = []
        tool_calls = []

        for block in content:
            if block["type"] == "text":
                text_parts.append(block["text"])
            elif block["type"] == "tool_use":
                tool_calls.append(ToolCall(
                    id=block["id"],
                    name=block["name"],
                    args=block["input"]
                ))

        return Thought(
            text="\n".join(text_parts) if text_parts else None,
            tool_calls=tool_calls
        )
```

Notice what `Claude` contains: the API key, the model name, the URL, the headers, the payload construction, the HTTP call, and the response parsing. Every detail of "how to talk to Anthropic" lives here. Nothing leaks out.

The `Agent` class — which we'll see in a moment — never looks at headers, JSON, or HTTP status codes. It only calls `brain.think(conversation)` and gets back a `Thought`. The agent doesn't care whether the brain is Claude, a local model, or a mock. It just calls `think`.

This is the **brain abstraction** at work:

```
┌──────────────────────────────────────────────────────────────────┐
│                        handle_input()                            │
│                                                                  │
│   user_input                                                     │
│       │                                                          │
│       ▼                                                          │
│   append to conversation list                                    │
│       │                                                          │
│       ▼                                                          │
│   brain.think(conversation)  ─────────────────────▶  Claude     │
│       │                                                  │       │
│       │                                           HTTP POST      │
│       │                                                  │       │
│       │                                          Anthropic API   │
│       │                                                  │       │
│       │                                         JSON response    │
│       │                                                  │       │
│       │                                          _parse_response │
│       │                                                  │       │
│       │  ◀──────────────────────── Thought(text, tool_calls) ───┘
│       │
│       ▼
│   append Thought.text to conversation
│       │
│       ▼
│   return text to main()
└──────────────────────────────────────────────────────────────────┘
```

The `Agent` interacts only with the left side of this diagram. The right side — the HTTP call, the JSON parsing, the API URL — is entirely the `Claude` class's concern.

### Why separate `think` from `__init__`?

`__init__` reads the API key and sets configuration. `think` does the actual work. This separation lets you create the brain once (checking the key exists) and call `think` many times. If we put the API call in `__init__`, the agent would make an API call at startup even before the user typed anything.

## The `ToolCall` and `Thought` Data Classes

```python
class ToolCall:
    """A tool invocation request from the brain."""

    def __init__(self, id, name, args):
        self.id = id
        self.name = name
        self.args = args  # dict


class Thought:
    """Standardized response from any Brain."""

    def __init__(self, text=None, tool_calls=None):
        self.text = text  # str or None
        self.tool_calls = tool_calls or []  # list of ToolCall
```

These are **data classes** — their only job is to hold structured data and give fields readable names.

In Chapter 3, `Thought.tool_calls` will always be an empty list. Claude won't be given any tools to call yet. But `Thought` is designed to carry them anyway. Why?

Because we're building for tomorrow, not just today. When Chapter 5 introduces tools, `handle_input` will check `thought.tool_calls` to decide whether to run a tool or return text. The check is already correct even now — `if not thought.tool_calls` — it just always passes. The shape of the return type is established here so every layer above it is already ready.

Think of it like a postal envelope with fields for both a sender and a return address. Even if you never use the return address field, the envelope is designed to hold one.

```
┌─────────────────────────────────────────────┐
│                   Thought                   │
│                                             │
│  text:       "Here's how to sort a list..." │
│                                             │
│  tool_calls: []    ← empty now              │
│                    ← will have entries in   │
│                       Chapter 5 onward      │
└─────────────────────────────────────────────┘
```

`ToolCall` holds three fields matching exactly what the Anthropic API sends back when it wants to invoke a tool:
- `id` — a unique identifier for this particular invocation (needed when you report the result back)
- `name` — which tool to call, e.g. `"read_file"`
- `args` — a dict of arguments, e.g. `{"path": "main.py"}`

## The Updated `Agent` Class

```python
class Agent:
    """A coding agent with conversation memory."""

    def __init__(self, brain):
        self.brain = brain
        self.conversation = []

    def handle_input(self, user_input):
        """Handle user input. Returns output string, raises AgentStop to quit."""
        if user_input.strip() == "/q":
            raise AgentStop()

        if not user_input.strip():
            return ""

        self.conversation.append({"role": "user", "content": user_input})

        try:
            thought = self.brain.think(self.conversation)
            text = thought.text or ""
            self.conversation.append({"role": "assistant", "content": text})
            return text
        except Exception as e:
            self.conversation.pop()  # Remove failed user message
            return f"Error: {e}"
```

Compare this to Chapter 1's `Agent`:

- `__init__` now takes a `brain` argument and stores it. It also initializes `self.conversation = []`.
- `handle_input` now appends to `self.conversation` instead of just returning a string.
- The brain is called via `self.brain.think(self.conversation)`.
- The result is appended to `self.conversation` before returning.

### Error Recovery: Popping the Failed Message

The `try/except` block does something subtle:

```python
self.conversation.append({"role": "user", "content": user_input})

try:
    thought = self.brain.think(self.conversation)
    ...
except Exception as e:
    self.conversation.pop()  # Remove failed user message
    return f"Error: {e}"
```

The user message is appended to `conversation` *before* calling the API. This is correct — the API needs the message in the list to respond to it. But if the API call fails (network error, bad key, rate limit), the user message is now stuck in the conversation history without a matching assistant reply. The next API call would see a conversation ending with a user turn — fine — but the user might not retry the same message, leaving an orphaned entry.

The `self.conversation.pop()` removes the message that was just appended, restoring the list to the state it was in before the failed call. The conversation stays consistent: it only ever contains turns where both sides spoke.

```
Before API call:           After failure (without pop):    After failure (with pop):
─────────────────          ────────────────────────────    ─────────────────────────
[ U ][ A ][ U* ]           [ U ][ A ][ U* ]                [ U ][ A ]
                              ↑ orphaned — no reply!          ↑ clean state
```

## Wiring It Together: `main()`

```python
def main():
    brain = Claude()
    agent = Agent(brain)
    print("⚡ Nanocode v0.2 (Conversation Memory)")
    print("Type '/q' to quit.\n")

    while True:
        try:
            user_input = input("❯ ")
            output = agent.handle_input(user_input)
            if output:
                print(f"\n{output}\n")

        except (AgentStop, KeyboardInterrupt):
            print("\nExiting...")
            break
```

Two new lines at the top: `brain = Claude()` and `agent = Agent(brain)`. Everything else is identical to Chapter 1. The loop didn't change. The contract of `handle_input` didn't change. Only what happens inside the agent changed — and the main loop doesn't need to know.

This is dependency injection in its simplest form. The brain is created outside the agent and passed in. The agent doesn't construct its own brain. This means in tests, you can pass a fake brain:

```python
class FakeBrain:
    def think(self, conversation):
        return Thought(text="fake response")

agent = Agent(FakeBrain())
result = agent.handle_input("hello")
assert result == "fake response"
```

No API calls. No API key. Just Python.

## Summary

| Concept | What It Does |
|---------|-------------|
| `conversation` list | The agent's memory — a list of `{role, content}` dicts, re-sent to the API on every call |
| `Claude` class | Owns all API details: key, URL, headers, payload, response parsing |
| `think(conversation)` | The only interface the agent uses — give it history, get back a `Thought` |
| `Thought` | Standardized return type with `text` and `tool_calls` fields |
| `ToolCall` | Carries `id`, `name`, `args` — ready for Chapter 5's tool support |
| Error recovery | `conversation.pop()` on failure keeps history consistent |
| Dependency injection | `Agent(brain)` — brain is passed in, not created inside |
