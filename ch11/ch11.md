# Chapter 11: SearchWeb — Giving the Agent a Window to the World

> *"A craftsman with every tool in the workshop is still helpless when the manual they need was printed after they graduated. Give them internet access and they can build anything."*

## What This Chapter Is About

In Chapter 10 we added context compaction, keeping the agent alive through long sessions without hitting token limits. Nanocode could read, write, edit, search, remember, and run commands — but it was still operating from a fixed point in time.

Claude's training data has a cutoff. It knows everything that was written before that date and nothing that was written after. For most tasks that's fine. But when the task involves a library version released last month, a bug fix merged last week, or a new API that went live yesterday — the agent is flying blind.

This chapter adds the final tool: **`SearchWeb`**. With it, Nanocode reaches out of its training window and retrieves real-time information from the internet. No API key required. No browser to parse. Just a lightweight DuckDuckGo query that returns three concise results and feeds them back into the conversation.

When this chapter is done, Nanocode is version 1.0 — a complete, production-ready coding agent.

By the end of this chapter you'll understand:
- Why a training cutoff creates a real gap, and when it matters
- How DuckDuckGo's `ddgs` library works and why it needs no API key
- The graceful import pattern for optional dependencies
- Why `max_results=3` is a deliberate constraint, not a limitation
- Why the tool returns summaries rather than following links
- The complete v1.0 feature set across all 8 tools

## The Training Cutoff Problem

Every large language model is trained on a snapshot of the internet. The training process is expensive and slow — it takes months — so the snapshot is frozen at a point in time called the training cutoff. After that point, the model knows nothing new unless you tell it.

Think of it like an encyclopedia published in a particular year. The 1987 World Book knew everything up to 1987. Ask it about the Berlin Wall coming down and it would have no entry — because that hadn't happened yet. The encyclopedia itself was not wrong or broken. It just had an expiration date on its relevance.

Claude is the same. It has encyclopedic knowledge of Python, algorithms, frameworks, and a thousand other topics — but that knowledge has an edge. Ask it what version of `requests` was current last month, and it will answer confidently with whatever it knew at training time, which may be outdated.

For a coding agent, this matters in four common situations:

```
┌─────────────────────────────────────────────────────────────────┐
│               When the cutoff bites                             │
│                                                                 │
│  1. Library version pinning                                     │
│     "What's the latest stable FastAPI version?"                 │
│     Claude knows: 0.103.x (trained Oct 2024)                    │
│     Reality: 0.115.x (released Jan 2025)                        │
│                                                                 │
│  2. Newly discovered bugs and CVEs                              │
│     "Is there a security issue in pydantic < 2.5?"              │
│     Claude: "I'm not aware of one" — but one was found in       │
│     Nov 2024, after its training snapshot                        │
│                                                                 │
│  3. New APIs and changed interfaces                             │
│     "How do I use the new OpenAI Responses API?"                │
│     Claude: blank stare — released March 2025                   │
│                                                                 │
│  4. Deprecations and removals                                   │
│     Framework X removed a method in its 3.0 release.            │
│     Claude confidently suggests the old method.                 │
└─────────────────────────────────────────────────────────────────┘
```

The fix is not to retrain Claude. The fix is to give the agent a way to look things up — the same way a human engineer would Google something before writing the code.

That is `SearchWeb`.

## Why DuckDuckGo?

There are many search APIs in the world. Google has one. Bing has one. SerpAPI aggregates several. Almost all of them require registering for an API key, agreeing to terms of service, and staying within rate limits that may cost money.

DuckDuckGo's search is available without any of that. The `ddgs` Python library wraps DuckDuckGo's text search interface. No account. No key. No billing.

There are two other reasons to prefer it:

**Privacy.** DuckDuckGo does not build user profiles or track searches across sessions. When your agent searches for debugging information about an internal project, you probably don't want that query associated with your account in a commercial search provider's logs.

**Simplicity.** The `ddgs` API is three lines: import, instantiate, call. The results come back as a list of dicts with `title`, `href`, and `body`. No OAuth, no pagination to handle, no quota exceeded errors at 3 PM on a Thursday.

## The Code: SearchWeb

The full tool is 22 lines:

```python
try:
    from ddgs import DDGS
except ImportError:
    DDGS = None

class SearchWeb:
    name = "search_web"
    description = "Searches the internet for current information. Use when you need knowledge beyond your training data."
    input_schema = {
        "type": "object",
        "properties": {
            "query": {"type": "string", "description": "The search query"}
        },
        "required": ["query"]
    }

    def execute(self, context, query):
        print(f"  → Searching web for '{query}'")
        if DDGS is None:
            return "Error: ddgs package not installed. Run: pip install ddgs"
        try:
            results = DDGS().text(query, max_results=3)
            if not results:
                return "No results found."
            formatted = []
            for r in results:
                formatted.append(f"Title: {r['title']}\nURL: {r['href']}\nSummary: {r['body']}\n")
            return "\n".join(formatted)
        except Exception as e:
            return f"Error searching web: {e}"
```

Let's walk through every decision.

---

### The Graceful Import Pattern

```python
try:
    from ddgs import DDGS
except ImportError:
    DDGS = None
```

This appears at the top of the file, before any class definitions. It is a module-level sentinel.

The question it answers is: *what happens to users who haven't installed `ddgs`?*

Without this pattern, the answer is: `ModuleNotFoundError` the moment the file is imported, before the agent even starts. Every user who hasn't installed `ddgs` — including users who don't need web search — cannot run the agent at all.

With this pattern, the import failure is caught silently. `DDGS` is set to `None`. The agent starts normally. The `SearchWeb` tool exists in the tools list and is visible to Claude. The only thing that changes is what happens when Claude actually calls the tool:

```python
if DDGS is None:
    return "Error: ddgs package not installed. Run: pip install ddgs"
```

Instead of a crash, the agent returns a clear, actionable error message. Claude reads this message, understands the tool is unavailable, and either proceeds without it or tells the user what to install. The agent degrades gracefully.

```
┌──────────────────────────────────────────────────────────────────┐
│              Graceful Degradation vs. Hard Failure               │
│                                                                  │
│  Without the pattern:                                            │
│    import ddgs  ──▶  ModuleNotFoundError  ──▶  agent crashes     │
│                                                                  │
│  With the pattern:                                               │
│    try: import ddgs                                              │
│    except: DDGS = None                                           │
│         │                                                        │
│         ▼                                                        │
│    Agent starts normally                                         │
│    Claude calls search_web                                       │
│         │                                                        │
│         ├── DDGS is not None  ──▶  performs search               │
│         │                                                        │
│         └── DDGS is None      ──▶  returns helpful error         │
│                                    Claude adapts                 │
└──────────────────────────────────────────────────────────────────┘
```

This pattern works for any optional dependency. If you later add a tool that requires `playwright` for browser automation, or `sqlalchemy` for database access, the same `try/except ImportError` wrapper handles it.

---

### `max_results=3` — The Token Budget Argument

```python
results = DDGS().text(query, max_results=3)
```

DuckDuckGo can return many results. Why limit to three?

Every result that comes back — title, URL, and summary paragraph — adds tokens to the conversation. Those tokens cost money. More importantly, they dilute the signal: if the first result answers the question, results four through ten are noise that Claude has to sort through.

Three results is a deliberately small number born from a simple observation: in practice, the answer to a specific technical question is almost always in the first result or the second. The third exists as a fallback when the first two are ambiguous.

You can raise this limit if you need it. But start conservative. Tokens are scarce, and padding the context with irrelevant web results pushes out space that could be used for code.

---

### Why the Tool Does Not Follow Links

The result from DuckDuckGo includes a `body` field — a paragraph-length summary of the page. The tool returns this summary. It does not fetch the URL and parse the HTML.

Following links would add three things the tool currently does not have:

1. **An HTTP request** — another network call, another failure mode
2. **HTML parsing** — the web is full of navigation menus, cookie banners, and script tags that bury the content
3. **Latency** — fetching a page is slower than fetching a search result

The DuckDuckGo summaries are extracted by DuckDuckGo itself from the page's meta description and opening text. For documentation pages, changelogs, and release notes — the most common targets for agent searches — these summaries typically contain the version number, the key change, or the API signature the agent was looking for.

The tool stays simple because simplicity is the right default. If you need full page content, you can add a `FetchURL` tool later (see Chapter 12 for ideas). But `SearchWeb` as written will handle 90% of lookup needs without it.

## How SearchWeb Fits Into a Real Workflow

The most powerful use of `SearchWeb` is as the first step in a research-then-code pattern. The agent searches for current information, synthesizes it, then writes code that uses it.

Here is a typical flow for "use the latest httpx version":

```
┌─────────────────────────────────────────────────────────────────┐
│            Research → Plan → Code workflow                      │
│                                                                 │
│  User: "Write a script using httpx to call a REST API"          │
│                │                                                │
│                ▼                                                │
│  Claude thinks: I should check the current httpx version        │
│  and confirm the API hasn't changed since my training.          │
│                │                                                │
│                ▼                                                │
│  search_web("httpx latest version Python HTTP client")          │
│                │                                                │
│                ▼                                                │
│  Result: "Title: httpx 0.28.1 released..."                      │
│          "Summary: async-capable, HTTP/2 support..."            │
│                │                                                │
│                ▼                                                │
│  Claude now knows: 0.28.1 is current, API is stable             │
│                │                                                │
│                ▼                                                │
│  write_file("api_client.py", <code using httpx 0.28.1>)         │
│                │                                                │
│                ▼                                                │
│  run_command("pip install httpx==0.28.1 && python api_client.py")│
│                │                                                │
│                ▼                                                │
│  (If it errors) search_web("httpx 0.28.1 <error message>")      │
│                │                                                │
│                ▼                                                │
│  Iterate until it works                                         │
└─────────────────────────────────────────────────────────────────┘
```

Notice the pattern: search first, then code, then run, then search again if the run fails. The agent is using the web the same way a human developer uses Stack Overflow — as a reference that sits alongside the code.

## The Complete Tool Inventory

Here is the full set of tools in Nanocode v1.0:

| Tool | Description | Plan Mode | Added |
|------|-------------|-----------|-------|
| `read_file` | Read any file with line numbers | Allowed | ch05 |
| `write_file` | Write a file (PLAN.md always allowed) | PLAN.md only | ch05 |
| `edit_file` | Surgical text replacement in a file | Blocked | ch06 |
| `list_files` | Walk the directory tree | Allowed | ch06 |
| `search_codebase` | Find text across all files | Allowed | ch06 |
| `save_memory` | Persist a note to `.nanocode/memory.md` | Allowed | ch08 |
| `run_command` | Execute a shell command | Blocked | ch09 |
| `search_web` | DuckDuckGo search, 3 results | Allowed | **ch11** |

Eight tools. Each added when we needed it, not before. Each as small as it could be while doing its job.

The mode column tells you what happens in plan mode (the safe default). Read operations are always allowed — the agent needs to see the codebase to make a plan. Write operations and command execution are blocked — those change state. `save_memory` is allowed in plan mode because remembering a preference doesn't change the project. `search_web` is allowed in plan mode because looking something up doesn't touch files.

