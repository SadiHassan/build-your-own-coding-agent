# Chapter 4: Resilience and Portability — Retry Logic and Swappable Brains

> *"A good postal system doesn't give up because the first truck broke down. It reroutes, waits, and tries again."*

## What This Chapter Is About

Chapter 3 gave us a real conversation. But it has a fragile assumption baked in: the API call will always succeed on the first try. In practice, that's not true.

APIs impose rate limits. Networks drop packets. Servers have bad moments. A single 429 response right now crashes the agent and leaves the user staring at a traceback. Chapter 4 fixes this — and adds a second brain provider while we're at it.

By the end of this chapter, Nanocode will:

1. Automatically retry failed requests with exponential backoff
2. Respect the server's `retry-after` header when present
3. Support both Claude and DeepSeek as interchangeable brains
4. Let the user switch brains at runtime with `/switch`

---

## Why APIs Fail (and Why That's Normal)

If you've only called APIs from small scripts, you may not have hit rate limits yet. In a long session with an AI agent, you will.

**Rate limits (HTTP 429)** exist because every API provider has limited GPU capacity. When too many requests arrive in a short window, the server starts refusing extras. This is not an error in your code — it's the server telling you to slow down and try again in a moment.

**Server errors (HTTP 5xx)** are transient failures on the provider's infrastructure. A machine restarted, a load balancer had a hiccup, a deployment rolled out. Retrying after a short wait almost always succeeds.

**Network failures** (`ConnectionError`, `Timeout`) happen when packets get lost or a router goes offline. Again: transient, and retryable.

The correct response to all three is the same: wait a moment and try again. The question is how long to wait.

---

## Exponential Backoff — The Right Way to Wait

**Fixed-delay retry** (waiting 1 second every time) is tempting but wrong. If 100 clients all get a 429 and all retry after exactly 1 second, they arrive together at the server and generate another wave of 429s. Synchronized retries amplify the problem.

**Exponential backoff** solves this by making each retry wait twice as long as the previous one:

```
Attempt 1: fails  →  wait  1s  (2^0)
Attempt 2: fails  →  wait  2s  (2^1)
Attempt 3: fails  →  wait  4s  (2^2)
Attempt 4: fails  →  wait  8s  (2^3)
Attempt 5: fails  →  wait 16s  (2^4)
...
```

The waits grow fast enough that the traffic from different clients quickly spreads out across time, letting the server recover. If 100 clients all start with different attempt numbers (because they started at different times), their retries land at different seconds — no synchronized thundering herd.

```
┌──────────────────────────────────────────────────────────────────────┐
│                     request_with_retry() flow                        │
│                                                                      │
│  attempt = 0                                                         │
│       │                                                              │
│       ▼                                                              │
│  ┌─────────────────┐                                                 │
│  │  requests.post()│ ──── network error? ─────────────────────┐      │
│  └─────────────────┘                                           │      │
│       │                                                        │      │
│       ▼                                                        ▼      │
│  status == 429                                      wait 2^attempt s  │
│  or status >= 500?  ──── yes ──────────────────────────────────┤      │
│       │                                                        │      │
│       │ no                                              attempt += 1  │
│       ▼                                                        │      │
│  status >= 400?                                                │      │
│  (non-retryable)  ──── yes ──▶  raise Exception               │      │
│       │                                                        │      │
│       │ no                                       attempt < max_retries?
│       ▼                                          yes ──▶ loop back    │
│  return response  ◀── success                   no  ──▶ raise        │
└──────────────────────────────────────────────────────────────────────┘
```

### The `retry-after` Header

Some servers don't make you guess. When they return a 429, they include a `retry-after` header telling you exactly how many seconds to wait:

```
HTTP/1.1 429 Too Many Requests
retry-after: 12
```

When this header is present, we use it instead of the exponential formula. The server knows its own recovery time better than we do:

```python
retry_after = response.headers.get("retry-after")
wait_time = int(retry_after) if retry_after else 2 ** attempt
```

This one line encodes a clear priority: trust the server's guidance first, fall back to our own calculation otherwise.

## The Code: `request_with_retry`

```python
def request_with_retry(url, headers, payload, max_retries=10):
    """Make HTTP POST with retry on rate limit (429), server errors (5xx), and network failures."""
    for attempt in range(max_retries):
        try:
            response = requests.post(url, headers=headers, json=payload, timeout=120)
        except requests.exceptions.RequestException as e:
            wait_time = 2 ** attempt
            print(f"Network error: {e}. Retrying in {wait_time}s...")
            time.sleep(wait_time)
            continue

        if response.status_code == 429 or response.status_code >= 500:
            retry_after = response.headers.get("retry-after")
            wait_time = int(retry_after) if retry_after else 2 ** attempt
            print(f"Error {response.status_code}. Retrying in {wait_time}s...")
            time.sleep(wait_time)
            continue

        if response.status_code >= 400:
            try:
                error_msg = response.json()["error"]["message"]
            except (KeyError, ValueError):
                error_msg = response.text
            raise Exception(f"API error ({response.status_code}): {error_msg}")

        return response

    raise Exception(f"Request failed after {max_retries} retries")
```

Let's walk through the logic in order:

**The outer loop** runs up to `max_retries` times (default 10). Each iteration is one attempt.

**Network error catch** — `requests.exceptions.RequestException` is the base class for all `requests` errors: `ConnectionError`, `Timeout`, `TooManyRedirects`, etc. Any of these triggers a wait and a retry. The wait is `2 ** attempt` — 1, 2, 4, 8, ... seconds.

**429 and 5xx check** — If we got a response but it signals a retryable failure, we apply the same backoff. The `retry-after` check lives here. Then `continue` jumps to the next loop iteration.

**4xx (non-retryable) errors** — A 400 (bad request), 401 (unauthorized), or 403 (forbidden) won't get better on retry — they indicate a problem with our request itself. We raise immediately with a useful error message extracted from the JSON body.

**Return on success** — Any status below 400 is a success. We return the response object.

**Exhausted retries** — If we loop all the way to `max_retries` without a successful response, we raise. Ten attempts with exponential backoff means we've waited up to 2^0 + 2^1 + ... + 2^9 = 1023 seconds before giving up — nearly 17 minutes. That's generous enough for almost any transient failure.

Notice that `request_with_retry` is a standalone function, not a method. It has no knowledge of Claude, DeepSeek, or any specific API. The `Claude` and `DeepSeek` classes both call it — it's shared infrastructure.

## The `Brain` Base Class — Designing for Extensibility

In Chapter 3, `Claude` was a standalone class. Nothing enforced that it had a `think` method or that `think` had a particular signature. That worked because there was only one brain.

Chapter 4 introduces a second brain (DeepSeek) and a mechanism to switch between them. For the agent to treat both interchangeably, they need to agree on a contract. That's what the `Brain` base class provides:

```python
class Brain:
    """Base class for LLM providers."""

    def think(self, conversation):
        """Process conversation, return Thought."""
        raise NotImplementedError
```

`Brain` defines the interface. Any class that inherits from it and implements `think` can serve as the agent's brain. The agent itself only calls `self.brain.think(conversation)` — it never calls any other method on the brain, and it never checks which concrete class it has.

```
┌──────────────────────────────────────────────────────────────────────┐
│                      Brain as an abstraction layer                   │
│                                                                      │
│                         ┌──────────┐                                 │
│                         │  Agent   │                                 │
│                         └────┬─────┘                                 │
│                              │ brain.think(conversation)             │
│                              │                                       │
│                         ┌────▼─────┐                                 │
│                         │  Brain   │  ← abstract interface           │
│                         │ (base)   │                                 │
│                         └────┬─────┘                                 │
│                              │                                       │
│                 ┌────────────┴────────────┐                          │
│                 │                         │                          │
│          ┌──────▼──────┐          ┌───────▼─────┐                   │
│          │   Claude    │          │  DeepSeek   │                   │
│          │             │          │             │                   │
│          │ api.anthro- │          │ api.deepseek│                   │
│          │  pic.com    │          │  .com       │                   │
│          └─────────────┘          └─────────────┘                   │
│                                                                      │
│  Adding a third provider (e.g. Gemini) means:                        │
│   1. Create class Gemini(Brain): implement think()                   │
│   2. Add to BRAINS registry                                          │
│   3. Touch nothing else                                              │
└──────────────────────────────────────────────────────────────────────┘
```

The Agent never needs to change when a new brain is added. This is the **open/closed principle** applied simply: the system is open to extension (add `Gemini`) but closed to modification (don't touch `Agent`).

### `Claude(Brain)` and `DeepSeek(Brain)`

Both subclasses follow the same pattern: read an API key from the environment, set a model name and URL, implement `think` by building a payload, calling `request_with_retry`, and returning a parsed `Thought`.

The striking thing about `DeepSeek` is how little changes:

```python
class DeepSeek(Brain):
    def __init__(self):
        self.api_key = os.getenv("DEEPSEEK_API_KEY")
        self.model = "deepseek-chat"
        self.url = "https://api.deepseek.com/anthropic/v1/messages"
```

The URL is different. The key name is different. The model name is different. The headers are the same — DeepSeek's Anthropic-compatible endpoint accepts the same `x-api-key` and `anthropic-version` headers. The request payload structure is the same. The response format is the same.

This is deliberate on DeepSeek's part: their API speaks the Anthropic wire format, so you can swap the URL and key and your existing code works. The same `_parse_response` method that handles Claude's response handles DeepSeek's. They share identical implementations in the current code.

```
Claude request:                    DeepSeek request:
──────────────                     ─────────────────
POST api.anthropic.com/v1/messages POST api.deepseek.com/anthropic/v1/messages
x-api-key: sk-ant-...              x-api-key: sk-ds-...
anthropic-version: 2023-06-01      anthropic-version: 2023-06-01
{"model": "claude-sonnet-4-6"...}  {"model": "deepseek-chat"...}

              same payload structure, different URL and key
```

## The `BRAINS` Registry

```python
BRAINS = {
    "claude": Claude,
    "deepseek": DeepSeek,
}
```

This is a **registry pattern**: a dictionary mapping names to constructors. The values are the classes themselves — not instances of the classes, but the classes as objects.

Why constructors instead of instances? Because brain construction requires API key validation. If we stored instances, both brains would be constructed at startup even if you only wanted one of them. Storing constructors means we only instantiate the brain we're actually using:

```python
# Registry holds constructors (the classes themselves):
BRAINS = {"claude": Claude, "deepseek": DeepSeek}

# Instantiate only when needed:
brain = BRAINS["claude"]()   # same as: brain = Claude()
```

This also makes switching brains cheap — create the new brain, throw away the old reference:

```python
def _switch_brain(self):
    names = list(BRAINS.keys())
    idx = names.index(self.brain_name)
    new_name = names[(idx + 1) % len(names)]

    try:
        self.brain = BRAINS[new_name]()
        self.brain_name = new_name
        return f"Switched to: {new_name}"
    except ValueError as e:
        return f"Cannot switch to {new_name}: {e}"
```

`_switch_brain` cycles through the registry in order. With two brains, it toggles. With three, it would cycle through all three. Adding a third brain requires only two changes: create the class, add it to `BRAINS`. The switch logic never changes.

The `ValueError` catch handles the case where the new brain's API key is missing — the constructor raises if the key isn't set. A missing DeepSeek key shouldn't crash the whole agent; it should report the problem and leave the current brain in place.

## The Updated `Agent` Class

```python
class Agent:
    """A coding agent with switchable brain."""

    def __init__(self, brain, brain_name="claude"):
        self.brain = brain
        self.brain_name = brain_name
        self.conversation = []

    def handle_input(self, user_input):
        if user_input.strip() == "/q":
            raise AgentStop()

        if user_input.strip() == "/switch":
            return self._switch_brain()

        if not user_input.strip():
            return ""

        self.conversation.append({"role": "user", "content": user_input})

        try:
            thought = self.brain.think(self.conversation)
            text = thought.text or ""
            self.conversation.append({"role": "assistant", "content": text})
            return text
        except Exception as e:
            self.conversation.pop()
            return f"Error: {e}"
```

Two additions over Chapter 3:

1. `brain_name` field — tracks which brain is active, shown in the prompt
2. `/switch` command — calls `_switch_brain()` instead of the API

The conversation history is preserved across brain switches. You can start a conversation with Claude, switch to DeepSeek, and continue — DeepSeek receives the full prior history. Both understand the same message format.

## The Updated `main()`

```python
def main():
    brain_name = os.getenv("NANOCODE_BRAIN", "claude")
    brain = BRAINS[brain_name]()
    agent = Agent(brain, brain_name)

    print(f"⚡ Nanocode v0.3")
    print(f"Commands: /q quit, /switch toggle brain")
    print(f"Brain: {brain_name}\n")

    while True:
        try:
            user_input = input(f"[{agent.brain_name}] ❯ ")
            output = agent.handle_input(user_input)
            if output:
                print(f"\n{output}\n")

        except (AgentStop, KeyboardInterrupt):
            print("\nExiting...")
            break
```

`NANOCODE_BRAIN` is an environment variable that selects the initial brain. Default is `"claude"`. This lets you start with DeepSeek without editing code:

```bash
NANOCODE_BRAIN=deepseek python ch04/nanocode.py
```

The prompt now includes the active brain name:

```
[claude] ❯ hello
(Claude is thinking...)

Hello! How can I help you today?

[claude] ❯ /switch
Switched to: deepseek

[deepseek] ❯ what did I just say?
(DeepSeek is thinking...)

You said "hello" a moment ago.
```

## Summary

| Concept | What It Does |
|---------|-------------|
| `request_with_retry` | HTTP POST with retry on 429/5xx/network errors; exponential backoff; honors `retry-after` |
| Exponential backoff | Wait `2^attempt` seconds between retries — spreads out traffic, prevents thundering herd |
| `retry-after` header | Server's explicit wait time — used instead of backoff when present |
| `Brain` base class | Defines the `think(conversation)` interface; any class implementing it can be a brain |
| `Claude(Brain)` | Anthropic API implementation |
| `DeepSeek(Brain)` | DeepSeek implementation — identical wire format, different URL and key |
| `BRAINS` registry | `{"name": ConstructorClass}` — constructors stored, not instances |
| `_switch_brain()` | Cycles through registry; creates new brain instance; preserves conversation |
| `NANOCODE_BRAIN` env var | Selects initial brain at startup without editing code |
