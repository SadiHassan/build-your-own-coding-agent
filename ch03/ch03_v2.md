## Concept 1 — Conversation as Memory

The most important idea in this chapter: **the model has no memory between API calls**. You must send the entire conversation history every time.

```
┌──────────────────────────────────────────────────────────────┐
│              CONVERSATION HISTORY GROWS OVER TIME            │
│                                                              │
│  Turn 1:                                                     │
│  conversation = [                                            │
│    {role: "user",      content: "What is Python?"}          │
│  ]                                                           │
│  ───────────────────────────────────────────────────────     │
│  Turn 2:                                                     │
│  conversation = [                                            │
│    {role: "user",      content: "What is Python?"},         │
│    {role: "assistant", content: "Python is a language..."}, │
│    {role: "user",      content: "Give me an example"}       │
│  ]                                                           │
│  ───────────────────────────────────────────────────────     │
│  Turn 3:                                                     │
│  conversation = [                                            │
│    {role: "user",      content: "What is Python?"},         │
│    {role: "assistant", content: "Python is a language..."}, │
│    {role: "user",      content: "Give me an example"},      │
│    {role: "assistant", content: "Here's hello world..."},   │
│    {role: "user",      content: "Now make it a function"}   │
│  ]       ↑                                                   │
│          └── entire history re-sent on every call           │
└──────────────────────────────────────────────────────────────┘
```

```python
class Agent:
    def __init__(self, brain):
        self.brain = brain
        self.conversation = []    # ← this list IS the memory

    def handle_input(self, user_input):
        # 1. Add user's message
        self.conversation.append({"role": "user", "content": user_input})

        # 2. Think (send the whole list)
        thought = self.brain.think(self.conversation)

        # 3. Add Claude's reply
        self.conversation.append({"role": "assistant", "content": thought.text or ""})

        return thought.text
```

## Concept 2 — The Claude Class (The Brain)

The `Claude` class wraps the HTTP call from Chapter 2 into a clean object with one public method: `think()`.

```python
class Claude:
    def __init__(self):
        self.api_key = os.getenv("ANTHROPIC_API_KEY")
        self.model = "claude-sonnet-4-6"
        self.url = "https://api.anthropic.com/v1/messages"

    def think(self, conversation):        # ← the only public method
        headers = { ... }
        payload = {
            "model": self.model,
            "max_tokens": 4096,
            "messages": conversation       # ← pass the full history
        }
        response = requests.post(self.url, headers=headers, json=payload, timeout=120)
        response.raise_for_status()
        return self._parse_response(response.json()["content"])

    def _parse_response(self, content):   # ← private helper
        text_parts = []
        tool_calls = []
        for block in content:
            if block["type"] == "text":
                text_parts.append(block["text"])
            elif block["type"] == "tool_use":
                tool_calls.append(ToolCall(
                    id=block["id"],
                    name=block["name"],
                    args=block["input"]
                ))
        return Thought(
            text="\n".join(text_parts) if text_parts else None,
            tool_calls=tool_calls
        )
```

## Concept 3 — Thought and ToolCall: Value Objects

```python
class ToolCall:
    def __init__(self, id, name, args):
        self.id = id
        self.name = name
        self.args = args      # dict of arguments

class Thought:
    def __init__(self, text=None, tool_calls=None):
        self.text = text               # str or None
        self.tool_calls = tool_calls or []
```

Why not just return the raw dict from the API?

```
┌─────────────────────────────────────────────────────────┐
│         RAW DICT  vs  VALUE OBJECT                      │
│                                                         │
│  Raw dict:                  Thought object:             │
│  response["content"]        thought.text                │
│    [0]["text"]              thought.tool_calls          │
│                                                         │
│  Problems with raw dict:    Benefits of Thought:        │
│  ✗ Must know API structure  ✓ Clean attribute access    │
│  ✗ Breaks if API changes    ✓ Isolated from API changes │
│  ✗ No defaults              ✓ tool_calls defaults to [] │
│  ✗ No documentation         ✓ Self-documenting fields   │
└─────────────────────────────────────────────────────────┘
```

> `tool_calls` is always `[]` in ch03 — no tools yet. But the field exists so the Agent can check `if thought.tool_calls` without special-casing.

## Concept 4 — The Agent Class

The `Agent` owns the conversation and delegates thinking to the brain.

```python
class Agent:
    def __init__(self, brain):    # ← receives brain, doesn't create it
        self.brain = brain
        self.conversation = []

    def handle_input(self, user_input):
        if user_input.strip() == "/q":
            raise AgentStop()             # ← sentinel exception

        if not user_input.strip():
            return ""                     # ← guard clause

        self.conversation.append({"role": "user", "content": user_input})

        try:
            thought = self.brain.think(self.conversation)
            text = thought.text or ""
            self.conversation.append({"role": "assistant", "content": text})
            return text
        except Exception as e:
            self.conversation.pop()       # ← undo the failed append
            return f"Error: {e}"
```

### Error Recovery: The `pop()` Pattern

When the API call fails, we remove the user message we just added. Otherwise the conversation list gets a dangling user message with no assistant reply — which confuses the API on the next call.

```
┌──────────────────────────────────────────────────────────┐
│                 ERROR RECOVERY PATTERN                   │
│                                                          │
│  BEFORE API CALL:          AFTER FAILED API CALL:       │
│  conversation = [          conversation = [              │
│    ...,                      ...,                        │
│    {role:"user",             {role:"user",               │
│     content:"hello"}  ──X──   content:"hello"}           │
│  ]            ↑            ]            ↑                │
│               │                         └── pop() this  │
│               └── append this                            │
│                                                          │
│  Result: conversation stays consistent, user sees error  │
└──────────────────────────────────────────────────────────┘
```

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| `conversation = []` | The list IS the memory — appended every turn, re-sent every call |
| `Claude.think()` | Wraps HTTP call, returns a clean `Thought` object |
| `Thought` | Value object: `.text` for words, `.tool_calls` for future tool use |
| `Agent(brain)` | Dependency injection — agent receives brain, enabling test fakes |
| `_parse_response` | Private method — implementation detail, not public API |
| `pop()` on error | Keeps conversation consistent if API call fails |
| `raise_for_status()` | Shorthand for HTTP error checking |
