# Chapter 9: The Feedback Loop

The previous chapters gave the agent the ability to read and write files. But a developer doesn't just write code — they run it, watch it fail, read the error, fix the problem, and run it again. That cycle of execution and correction is where most real development happens.

This chapter closes the loop. We add `RunCommand`, which lets the agent execute shell commands and read their output. We add `EditFile`, a surgical alternative to `WriteFile` for making precise changes. We harden the agentic loop with an iteration limit. And we add context compaction so the conversation doesn't grow without bound. Finally, we introduce `FakeBrain` — a test double that lets us verify all of this without making a single API call.

---

## EditFile: Surgical Changes

Until now, modifying a file meant rewriting it entirely with `WriteFile`. That works, but it's blunt. If you want to change one function in a 500-line file, you don't want to regenerate all 500 lines. You want to find the relevant section and replace it.

```python
class EditFile:
    name = "edit_file"
    description = "Replaces specific text in a file. Use for surgical edits instead of rewriting entire files."
    input_schema = {
        "type": "object",
        "properties": {
            "path":     {"type": "string", "description": "Path to the file"},
            "old_text": {"type": "string", "description": "Exact text to find and replace"},
            "new_text": {"type": "string", "description": "Text to replace it with"}
        },
        "required": ["path", "old_text", "new_text"]
    }

    def execute(self, context, path, old_text, new_text):
        if context.mode == "plan":
            return "BLOCKED: Cannot edit files in plan mode."

        with open(path, 'r', encoding='utf-8') as f:
            content = f.read()
        if old_text not in content:
            return f"Error: Could not find the specified text in {path}"
        new_content = content.replace(old_text, new_text, 1)
        with open(path, 'w', encoding='utf-8') as f:
            f.write(new_content)
        return f"Successfully edited {path}"
```

The key is `content.replace(old_text, new_text, 1)`. The third argument `1` means replace only the first occurrence. Without it, if the same string appears multiple times in the file, every occurrence would be replaced — usually not the intent.

The exact-match requirement matters too. If `old_text` isn't found verbatim in the file, the tool returns an error instead of silently doing nothing. This is better behavior: Claude knows the edit failed and can try a different approach, rather than assuming it succeeded when nothing changed.

## RunCommand: Executing Shell Commands

```python
class RunCommand:
    name = "run_command"
    description = "Executes a terminal command. Use this to run scripts, tests, or install packages. Blocked in plan mode."
    input_schema = {
        "type": "object",
        "properties": {
            "command": {"type": "string", "description": "The shell command to run"}
        },
        "required": ["command"]
    }

    def execute(self, context, command):
        if context.mode == "plan":
            return "BLOCKED: Cannot run commands in plan mode. Switch to act mode first."

        result = subprocess.run(
            command,
            shell=True,
            capture_output=True,
            text=True,
            timeout=int(os.environ.get("NANOCODE_TIMEOUT", "30")),
            cwd=os.getcwd()
        )

        output = ""
        if result.stdout:
            output += f"STDOUT:\n{result.stdout}\n"
        if result.stderr:
            output += f"STDERR:\n{result.stderr}\n"
        if not output:
            output = "(No output)"

        return output.strip()
```

`subprocess.run()` launches a child process and waits for it to finish. `shell=True` means the command string is passed to the system shell — `/bin/sh` on Unix — which allows pipes, environment variable expansion, and all other shell features. `capture_output=True` captures both stdout and stderr as strings rather than letting them print to the terminal. `text=True` returns strings instead of bytes.

The timeout is configurable via the `NANOCODE_TIMEOUT` environment variable, defaulting to 30 seconds. Without a timeout, a runaway command would block the agent indefinitely. 30 seconds is long enough for most development tasks — running tests, installing packages, executing scripts — but short enough to surface hangs quickly.

We return both stdout and stderr, labeled. This is important for test runners: `pytest` writes test results to stdout but some tools write errors to stderr. Claude needs to see both to understand what happened.

## The Feedback Loop in Action

The combination of `RunCommand`, `ReadFile`, `EditFile`, and `WriteFile` enables a pattern that looks like actual development:

1. Claude writes code to a file
2. Claude runs the tests
3. The tests fail — the output shows an error
4. Claude reads the relevant file and finds the bug
5. Claude edits the file to fix it
6. Claude runs the tests again
7. Tests pass

```
Write code ──▶ Run tests ──▶ Read error ──▶ Fix code ──▶ Run tests ──▶ Pass
                   │                             ▲
                   └── failure output ───────────┘
```

This cycle can run many times in a single session, driven entirely by the agentic loop. The agent keeps going until the tests pass or it's stuck. No manual intervention required.

## Hardening the Agentic Loop

The `while True` loop from Chapter 5 is fine for most conversations, but an agent with command execution could get stuck: run a command, read an error, try a fix, run again, see the same error, try another fix, run again... indefinitely.

We add a ceiling:

```python
def _agentic_loop(self):
    output_parts = []
    max_iterations = 50

    for _iteration in range(max_iterations):
        thought = self.brain.think(self.conversation)
        ...
        if not thought.tool_calls:
            break
        ...
    else:
        output_parts.append("(Stopped: too many iterations)")

    return "\n".join(output_parts)
```

The `for` loop replaces `while True`. When the loop completes all 50 iterations without hitting `break`, Python executes the `else` clause — a feature specific to `for` and `while` loops in Python that many developers don't know about. The `else` runs only when the loop exhausted all iterations without breaking. We use it to append a notification that the loop was stopped, so the user knows what happened.

50 iterations is generous. Most tasks complete in under 10. If an agent reaches 50, something has gone wrong — the code isn't converging, the tests have an environment issue, Claude is stuck in a reasoning loop. Stopping with a message is better than running forever.

## Context Compaction

Long agent sessions accumulate a lot of conversation history. Tool results can be verbose — a file listing of a large project, the output of a test run, several file contents. Each of these goes into the conversation list, which is re-sent to the API on every call. After enough iterations, the input token count can approach the model's context limit.

We track this and summarize proactively:

```python
# In Brain base class:
context_limit = 200_000    # tokens
last_input_tokens = 0      # updated after each think()

# In Claude.think():
data = response.json()
self.last_input_tokens = data.get("usage", {}).get("input_tokens", 0)

# In _agentic_loop():
if self.brain.last_input_tokens > self.brain.context_limit * 0.75:
    self._compact_conversation()
```

The threshold is 75% of the context limit. We compact before hitting the limit, not after. Once you exceed the limit, the API starts rejecting requests. Compacting early leaves room for the compaction API call itself and for the rest of the current task.

The compaction itself asks Claude to summarize the conversation:

```python
def _compact_conversation(self):
    history = "\n".join(
        f"{m['role']}: {str(m['content'])[:500]}"
        for m in self.conversation
    )
    prompt = [{
        "role": "user",
        "content": f"Summarize this conversation for continuity. "
                   f"Focus on what was accomplished, what's in progress, "
                   f"and key decisions:\n\n{history}"
    }]
    thought = self.brain.think(prompt)
    self.conversation = [
        {"role": "user", "content": f"Previous conversation summary: {thought.text}"}
    ]
```

The full conversation is replaced with a single message containing the summary. The agent loses the fine details of what was said but keeps the essential continuity — what was accomplished, what's in progress. For most tasks, that's enough to continue productively.

## Testing Without an API Key

All the new capability we've added — command execution, iteration limits, compaction, mode enforcement — needs to be tested. But tests that call the real API are slow, expensive, and non-deterministic. A test run shouldn't cost money.

The solution is `FakeBrain`:

```python
class FakeBrain:
    context_limit = 200_000
    last_input_tokens = 0

    def __init__(self, responses=None, memory=None, tools=None):
        self.responses = responses or [
            Thought(text="Fake response",
                    raw_content=[{"type": "text", "text": "Fake response"}])
        ]
        self.call_count = 0

    def think(self, conversation):
        if self.call_count < len(self.responses):
            response = self.responses[self.call_count]
            self.call_count += 1
            return response
        return Thought(text="No more responses",
                       raw_content=[{"type": "text", "text": "No more responses"}])
```

`FakeBrain` takes a list of pre-scripted `Thought` objects. Each call to `think()` returns the next one in the list. When the list is exhausted, it returns a fallback. No HTTP. No API key. Just predictable, deterministic behavior.

This enables tests like:

```python
def test_agentic_loop_executes_tool():
    tool_call = ToolCall(id="t1", name="read_file", args={"path": "test.py"})
    responses = [
        Thought(tool_calls=[tool_call],
                raw_content=[{"type": "tool_use", "id": "t1", ...}]),
        Thought(text="Done reading.",
                raw_content=[{"type": "text", "text": "Done reading."}])
    ]
    agent = Agent(brain=FakeBrain(responses=responses), tools=tools)
    result = agent.handle_input("read test.py")
    assert "Done reading." in result
```

The test exercises the full agentic loop — tool call, tool execution, second think, final response — without touching the network. Tests run in milliseconds. The full suite covers the loop, tool execution, mode enforcement, memory persistence, and compaction. All without an API key.
