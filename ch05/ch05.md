# Chapter 5: Tools and the Agentic Loop — Giving Claude Hands

> *"A brilliant architect locked in a room with no pencil can design nothing. Give them a pencil and they can build cathedrals."*

## What This Chapter Is About

Up to now, Claude has been a conversationalist — it can think and respond, but it cannot *do* anything to your files. In this chapter we give it hands.

We introduce two tools — `ReadFile` and `WriteFile` — and the machinery that lets Claude decide when to use them, trigger them, receive their output, and continue thinking. That machinery is called the **agentic loop**.

By the end of this chapter you'll understand:
- What a tool is and how Claude learns about it (JSON Schema)
- The exact message format Claude requires when tools are involved
- Why we store `raw_content` instead of just text in the conversation
- How the agentic loop works as a `while True` that only breaks on silence
- How tool results feed back into the conversation as "user" messages

The new file is `ch05/nanocode.py`. Run it with `python ch05/nanocode.py` and try asking Claude to read or modify a file.

## The Big Picture: From Thinker to Actor

In Chapter 4, Claude was a mind without a body. It could reason, but every answer it gave was pure text — it had no way to look at your actual files.

Here is what changes in this chapter:

```
BEFORE (ch04):

  You ─────────────────────────────────────────────▶ Claude
                                                       │
                                                       │ (thinks)
                                                       │
  You ◀──────────────────────── text response ─────────┘


AFTER (ch05):

  You ─────────────────────────────────────────────▶ Claude
                                                       │
                                                       │ "I need to see that file"
                                                       ▼
                                               [tool_use block]
                                                       │
                                            Your code executes
                                            ReadFile("utils.py")
                                                       │
                                               [tool_result]
                                                       │
                                                       ▼
                                                     Claude
                                                       │
                                                       │ (continues thinking)
                                                       │
  You ◀──────────────────────── text response ─────────┘
```

Claude doesn't call your functions directly. It *announces* that it wants to call one (by returning a structured `tool_use` block), your code executes the actual function, and you feed the result back. Claude then continues. This back-and-forth can happen many times before Claude gives its final answer.

## What Is a Tool?

Think of a tool as a contract between you and Claude. The contract has three parts:

1. **Name** — a string identifier, like `"read_file"`
2. **Description** — plain English that Claude reads to decide *when* to use this tool
3. **Input schema** — a JSON Schema object that tells Claude *what arguments to pass*

Claude reads all three when you build the API payload. The description is how it decides whether to reach for a tool at all. The schema is what tells it the exact shape of the arguments. If you write a vague description, Claude will use the tool at the wrong times. If you write a wrong schema, Claude will pass bad arguments.

Here is the full `ReadFile` class:

```python
class ReadFile:
    """Reads a file from the filesystem."""
    name = "read_file"
    description = "Reads a file from the filesystem. Use this to examine code."
    input_schema = {
        "type": "object",
        "properties": {
            "path": {"type": "string", "description": "The relative path to the file"}
        },
        "required": ["path"]
    }

    def execute(self, path):
        print(f"  → Reading {path}")
        try:
            with open(path, 'r', encoding='utf-8') as f:
                lines = f.readlines()
            numbered_lines = [f"{i+1} | {line}" for i, line in enumerate(lines)]
            return "".join(numbered_lines)
        except Exception as e:
            return f"Error reading file: {e}"
```

The class attributes — `name`, `description`, `input_schema` — are metadata *for Claude*. The `execute()` method is the actual work *your code* runs.

Why numbered lines? Because when Claude wants to edit a specific region, it needs to refer to line numbers. Returning `"42 | def process(x):"` is far more useful than returning the raw text.

`WriteFile` follows the same pattern:

```python
class WriteFile:
    """Writes content to a file."""
    name = "write_file"
    description = "Writes content to a file. OVERWRITES existing content."
    input_schema = {
        "type": "object",
        "properties": {
            "path": {"type": "string", "description": "The relative path to the file"},
            "content": {"type": "string", "description": "The full content to write"}
        },
        "required": ["path", "content"]
    }

    def execute(self, path, content):
        print(f"  → Writing {path}")
        try:
            with open(path, 'w', encoding='utf-8') as f:
                f.write(content)
            return f"Successfully wrote {len(content)} characters to {path}"
        except Exception as e:
            return f"Error writing file: {e}"
```

Note that `WriteFile` overwrites the entire file. There's no partial-edit mechanism — Claude must write the complete new content. This keeps the implementation simple and Claude honest: it must read before it writes.

## The Tool Helpers

Two small functions convert tool objects into the format the API expects:

```python
def tool_definitions(tools):
    """Return tool definitions for the API."""
    return [
        {"name": t.name, "description": t.description, "input_schema": t.input_schema}
        for t in tools
    ]


def get_tool(tools, name):
    """Find a tool by name, or None if not found."""
    return next((t for t in tools if t.name == name), None)
```

`tool_definitions()` is called once at startup to build the list that goes into the API payload. It strips the tool objects down to their metadata — the API has no knowledge of your Python classes, only the JSON they produce.

`get_tool()` is called at runtime by `_execute_tool()`. It walks the list and returns the first tool whose `.name` matches. If no match is found, it returns `None` and the caller handles the error.

```
┌───────────────────────────────────────────────────────┐
│                   tool_definitions()                  │
│                                                       │
│  [ReadFile(), WriteFile()]                            │
│         │                                             │
│         ▼                                             │
│  [                                                    │
│    {                                                  │
│      "name": "read_file",                             │
│      "description": "Reads a file...",                │
│      "input_schema": { "type": "object", ... }        │
│    },                                                 │
│    {                                                  │
│      "name": "write_file",                            │
│      "description": "Writes content...",              │
│      "input_schema": { "type": "object", ... }        │
│    }                                                  │
│  ]                                                    │
│         │                                             │
│         └─────────────────▶ payload["tools"]          │
└───────────────────────────────────────────────────────┘
```

## Why `raw_content`? The Critical Detail

In Chapter 3, when Claude replied we did this:

```python
self.conversation.append({"role": "assistant", "content": thought.text})
```

We stored the text string. That worked because Claude's response *was* a text string — one block, one type.

When tools are involved, Claude's response is a *list of blocks*:

```json
[
  {"type": "text", "text": "Let me check that file for you."},
  {"type": "tool_use", "id": "toolu_01A...", "name": "read_file", "input": {"path": "utils.py"}}
]
```

If we strip this down to just the text and store `"Let me check that file for you."`, then on the next turn we send Claude a history that says it responded with only text — but its actual response included a `tool_use` block. When Claude sees the `tool_result` block in the *next* user message (which references `tool_use_id: "toolu_01A..."`), it will be confused: that ID doesn't exist in its history.

The Anthropic API enforces this strictly. **The assistant message must contain the original `tool_use` block verbatim.** Otherwise you get a validation error.

This is why `Thought` gained a `raw_content` field:

```python
class Thought:
    def __init__(self, text=None, tool_calls=None, raw_content=None):
        self.text = text          # human-readable text (for display)
        self.tool_calls = tool_calls or []  # parsed ToolCall objects (for dispatch)
        self.raw_content = raw_content      # the original list (for history)
```

And why `_agentic_loop` stores it this way:

```python
self.conversation.append({"role": "assistant", "content": thought.raw_content})
#                                                          ^^^^^^^^^^^^^^^^
#                                          NOT thought.text — the whole block list
```

Think of `raw_content` as a receipt. When the tool result arrives, the API needs to match it to the original request by ID. Without the receipt, there's no match.

```
  └─────────────────────────────────────────────────────────────┘

  ┌─ Turn 2: Claude responds with tool_use ─────────────────────┐
  │ {                                                           │
  │   "role": "assistant",                                      │
  │   "content": [                          ← raw_content       │
  │     {                                                       │
  │       "type": "text",                                       │
  │       "text": "Let me read that file."                      │
  │     },                                                      │
  │     {                                                       │
  │       "type": "tool_use",                                   │
  │       "id": "toolu_01A2B3",             ← ID stamped here   │
  │       "name": "read_file",                                  │
  │       "input": {"path": "utils.py"}                         │
  │     }                                                       │
  │   ]                                                         │

  └─────────────────────────────────────────────────────────────┘

  ┌─ Turn 3: Tool result fed back as "user" ────────────────────┐
  │ {                                                           │
  │   "role": "user",                       ← yes, "user"!     │
  │   "content": [                                              │
  │     {                                                       │
  │       "type": "tool_result",                                │
  │       "tool_use_id": "toolu_01A2B3",    ← matches above     │
  │       "content": "1 | def process(x):\n2 | ..."             │
  │     }                                                       │
  │   ]                                                         │
  │ }                                                           │
  └─────────────────────────────────────────────────────────────┘

  ┌─ Turn 4: Claude gives final text response ──────────────────┐
  │ {                                                           │
  │   "role": "assistant",                                      │
  │   "content": [                                              │
  │     {"type": "text", "text": "The file contains..."}        │
  │   ]                                                         │
  │ }                                                           │
  └─────────────────────────────────────────────────────────────┘

]
```

Two things look strange until you understand the protocol:

**The assistant message has a list for `content`, not a string.** When tools are involved, `content` is always a list of typed blocks, never a plain string. This is why we store `raw_content` — it is that list.

**The tool result arrives as a "user" message.** This feels backwards — the environment is sending a result, not the human. But from the API's perspective, anything that is not Claude is "the user." The environment, your tools, the file system — they all speak through the `"role": "user"` channel.

## The Agentic Loop

The agentic loop is the `while True` at the heart of the agent. Here is its complete flowchart:

```
                    ┌──────────────────────┐
                    │   User sends input   │
                    └──────────┬───────────┘
                               │
                               ▼
                    ┌──────────────────────┐
                    │  Append user message │
                    │  to conversation     │
                    └──────────┬───────────┘
                               │
                               │  ◀──────────────────────────────┐
                               ▼                                  │
                    ┌──────────────────────┐                      │
                    │  brain.think()       │                      │
                    │  (POST to API)       │                      │
                    └──────────┬───────────┘                      │
                               │                                  │
                               ▼                                  │
                    ┌──────────────────────┐                      │
                    │  Append raw_content  │                      │
                    │  to conversation     │                      │
                    └──────────┬───────────┘                      │
                               │                                  │
                    ┌──────────┴───────────┐                      │
                    │                      │                      │
              tool_calls?            no tool_calls                │
                    │                      │                      │
                    ▼                      ▼                      │
       ┌────────────────────┐   ┌──────────────────────┐         │
       │  For each tool     │   │  Return collected    │         │
       │  call:             │   │  text to user        │         │
       │  _execute_tool()   │   └──────────────────────┘         │
       └────────────┬───────┘                                     │
                    │                                             │
                    ▼                                             │
       ┌────────────────────┐                                     │
       │  Collect results   │                                     │
       │  into tool_results │                                     │
       └────────────┬───────┘                                     │
                    │                                             │
                    ▼                                             │
       ┌────────────────────┐                                     │
       │  Append tool_results│                                    │
       │  as "user" message  │                                    │
       └────────────┬───────┘                                     │
                    │                                             │
                    └─────────────────────────────────────────────┘
                              (loop again)
```

The key insight: we loop *until Claude stops asking for tools*. Claude controls the exit condition by returning no `tool_calls`. When it does that, we break and return all the text it accumulated along the way.

Here is the complete implementation:

```python
def _agentic_loop(self):
    """Process brain responses, executing tools until done."""
    output_parts = []

    while True:
        thought = self.brain.think(self.conversation)

        # Store raw content for message history (Claude expects this format)
        self.conversation.append({"role": "assistant", "content": thought.raw_content})

        # Collect text output
        if thought.text:
            output_parts.append(thought.text)

        # Check for tool calls
        if not thought.tool_calls:
            break  # No more tools to execute

        # Execute tools and collect results
        tool_results = []
        for tool_call in thought.tool_calls:
            result = self._execute_tool(tool_call.name, tool_call.args)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": tool_call.id,
                "content": result
            })

        self.conversation.append({"role": "user", "content": tool_results})

    return "\n".join(output_parts)
```

Walk through the loop once:

1. `brain.think(self.conversation)` — sends the whole conversation to the API, gets a `Thought` back
2. `self.conversation.append(...)` — stores the raw block list (not just the text)
3. `if thought.text` — if there was a text portion, save it for the final response
4. `if not thought.tool_calls: break` — if Claude didn't ask for tools, we're done
5. Otherwise, execute each tool, package results as `tool_result` blocks, append as a "user" message, and loop

Note that Claude can ask for *multiple* tools in a single response. The `for tool_call in thought.tool_calls` loop handles that — all results get packaged together in one `tool_results` list and sent back in one "user" message.

## Dispatching Tools: `_execute_tool`

```python
def _execute_tool(self, name, args):
    """Execute a tool by name with given arguments."""
    tool = get_tool(self.tools, name)
    if tool is None:
        return f"Error: Tool '{name}' not found"
    try:
        return tool.execute(**args)
    except TypeError as e:
        return f"Error: Invalid arguments - {e}"
```

Three things happen here:

1. **Lookup by name** — `get_tool()` finds the right tool object from the list
2. **Error on unknown name** — if Claude hallucinated a tool name, we return an error string (not raise — the error string goes back to Claude as a `tool_result`, which is more informative than a crash)
3. **`**args` unpacking** — `args` is a dict like `{"path": "utils.py"}`. `**args` unpacks it so `execute()` receives keyword arguments matching its signature

Returning an error string rather than raising is a deliberate choice. The error goes back to Claude in the conversation, and Claude often self-corrects on the next turn. A Python exception, by contrast, would surface to the user with no context.

## Wiring Tools into the Brain

The `Claude` constructor now accepts `tools`:

```python
class Claude(Brain):
    def __init__(self, tools=None):
        self.tools = tools or []
        ...

    def think(self, conversation):
        payload = {
            "model": self.model,
            "max_tokens": 4096,
            "messages": conversation
        }
        if self.tools:
            payload["tools"] = self.tools
        ...
```

`payload["tools"]` is the list produced by `tool_definitions()` — the JSON Schema metadata list, not the Python tool objects. Claude reads this metadata to know what tools exist and what arguments they take. If `self.tools` is empty (e.g. you initialized without tools), we skip the field entirely rather than sending an empty list, which keeps the payload clean.

At startup, the wiring happens in `main()`:

```python
tools = [ReadFile(), WriteFile()]
brain = Claude(tools=tool_definitions(tools))
agent = Agent(brain=brain, tools=tools, brain_name=brain_name)
```

Note that `brain` receives `tool_definitions(tools)` — the JSON metadata — while `agent` receives `tools` — the actual Python objects with their `execute()` methods. The brain needs the schemas. The agent needs the executors. They use the same list but different views of it.

## A Complete Example Exchange

Suppose you ask: `"What does the main() function do in nanocode.py?"`

Here is the conversation that flows through the system:

```
[claude] ❯ What does the main() function do in nanocode.py?

(Claude is thinking...)          ← Turn 1: POST to API

  → Reading nanocode.py          ← Claude returns tool_use, we execute ReadFile

(Claude is thinking...)          ← Turn 2: POST with file contents

"The main() function initializes the brain and agent, prints a startup
 banner, then runs the event loop — waiting for input, passing it to
 agent.handle_input(), and printing the result."
```

Internally, three entries were added to `self.conversation`:
1. Your question (user)
2. Claude's tool_use block (assistant, stored as `raw_content`)
3. The file contents (user, as `tool_result`)

Then `brain.think()` was called a second time with all three entries. The second call returned a pure text response (no tool_calls), so the loop broke and printed the answer.

## Summary

| Concept | What It Does |
|---------|-------------|
| Tool class | Groups `name`, `description`, `input_schema` (for Claude) and `execute()` (for your code) |
| `input_schema` | JSON Schema that tells Claude what arguments the tool accepts |
| `tool_definitions()` | Converts tool objects to the JSON list sent in `payload["tools"]` |
| `raw_content` | The original API response block list — stored verbatim so `tool_use` IDs are preserved |
| `_agentic_loop()` | `while True` that calls Claude, executes tools, feeds results back, breaks on silence |
| `tool_result` as "user" | API convention: tool output enters the conversation under `"role": "user"` |
| `_execute_tool()` | Dispatches by name, unpacks args with `**`, returns error strings instead of raising |

## What We Didn't Add (And Why)

`WriteFile` overwrites the whole file — there's no line-level patch. That keeps the implementation to 10 lines. In later chapters we'll add a shell command tool that can run `git diff` to verify changes before committing.

There's no permission system. Any file path Claude names gets read or written. Chapter 8 adds a `mode` concept (read-only vs. write) and Chapter 9 adds sandboxing via allowed-path lists.

There's no iteration limit. If Claude keeps asking for tools forever, this loop will keep running. Chapter 9 adds a `max_iterations` guard.
