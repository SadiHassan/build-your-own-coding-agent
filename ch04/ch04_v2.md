# Chapter 4: Multiple Providers

Coding agents run long tasks. A task that takes fifty API calls might encounter a rate limit on call thirty-seven, or a transient server error on call forty-two. If the agent crashes on either, all the work done so far is lost. And if it's tied to a single provider, an outage means a dead tool.

This chapter addresses both problems. We add automatic retry logic so temporary failures become invisible, and we introduce a second AI provider — DeepSeek — so the agent isn't locked to one vendor. Along the way, we formalize the brain interface with a base class, making it trivially easy to add new providers in the future.

---

## Building a Reliable HTTP Layer

The raw `requests.post()` call from Chapter 3 is fragile. If anything goes wrong — rate limit, server error, dropped connection — it raises an exception and the agent dies. In a short script, that's acceptable. In an agent that might run for ten minutes, it isn't.

We replace the bare call with a function that automatically handles the common failure modes:

```python
def request_with_retry(url, headers, payload, max_retries=10):
    for attempt in range(max_retries):
        try:
            response = requests.post(url, headers=headers, json=payload, timeout=120)
        except requests.exceptions.RequestException as e:
            wait_time = 2 ** attempt
            print(f"Network error: {e}. Retrying in {wait_time}s...")
            time.sleep(wait_time)
            continue

        if response.status_code == 429 or response.status_code >= 500:
            retry_after = response.headers.get("retry-after")
            wait_time = int(retry_after) if retry_after else 2 ** attempt
            print(f"Error {response.status_code}. Retrying in {wait_time}s...")
            time.sleep(wait_time)
            continue

        if response.status_code >= 400:
            try:
                error_msg = response.json()["error"]["message"]
            except (KeyError, ValueError):
                error_msg = response.text
            raise Exception(f"API error ({response.status_code}): {error_msg}")

        return response

    raise Exception(f"Request failed after {max_retries} retries")
```

The logic has three branches, handling three distinct situations.

**Network errors** — when `requests.post()` raises an exception before even getting a response, something went wrong at the TCP layer. DNS failure, connection refused, timeout during the request itself. These are almost always transient. We wait and retry.

**Rate limits and server errors** — status codes 429 and 500+ mean the server received the request but couldn't fulfill it right now. A 429 means you've sent too many requests too quickly. A 5xx means the server itself is having trouble. Both are temporary. We wait and retry.

**Client errors** — status codes 400–499 (excluding 429) indicate something wrong with the request itself: bad API key (401), malformed payload (400), non-existent model (404). Retrying won't help. We raise immediately.

The wait between retries isn't fixed. It doubles with each attempt — 1 second, then 2, then 4, then 8, then 16, and so on. This is *exponential backoff*, and it has a specific purpose: when a service is under load, hammering it with constant retries makes the problem worse. Exponential backoff gives the service time to recover while keeping the agent patient instead of aggressive.

```
Attempt 0  →  wait  1s
Attempt 1  →  wait  2s
Attempt 2  →  wait  4s
Attempt 3  →  wait  8s
Attempt 4  →  wait 16s
Attempt 5  →  wait 32s
```

There is one refinement: the `retry-after` response header. When a server returns a 429, it sometimes includes this header with a recommended wait time in seconds. Anthropic does this. When it's present, we respect it over our own calculation — the server knows better than we do when it'll be ready.

## The Brain Interface

With two providers on the way, we need a common contract. Both Claude and DeepSeek must implement the same interface so the `Agent` can use either interchangeably.

In Python, the conventional way to express this is a base class with a method that raises `NotImplementedError`:

```python
class Brain:
    def think(self, conversation):
        raise NotImplementedError
```

Any class that inherits from `Brain` and doesn't override `think()` will raise `NotImplementedError` when called. It's not enforced at class definition time — Python isn't Java — but it communicates a clear contract: subclasses must provide their own implementation.

```
                    Brain
                   (base)
                think() → NotImplementedError
                      │
          ┌───────────┴───────────┐
          │                       │
       Claude                  DeepSeek
      think() ✓               think() ✓
```

Now `Claude` becomes `Claude(Brain)`:

```python
class Claude(Brain):
    def __init__(self):
        self.api_key = os.getenv("ANTHROPIC_API_KEY")
        self.model = "claude-sonnet-4-6"
        self.url = "https://api.anthropic.com/v1/messages"

    def think(self, conversation):
        headers = { ... }
        payload = { "model": self.model, "max_tokens": 4096, "messages": conversation }
        response = request_with_retry(self.url, headers, payload)
        return self._parse_response(response.json()["content"])
```

The body of `think()` barely changed — we swapped the bare `requests.post()` for `request_with_retry()`. The public interface is identical.

## Adding DeepSeek

DeepSeek is a Chinese AI company that offers an Anthropic-compatible API. The wire format — the JSON structure of requests and responses — is identical to Anthropic's. The only differences are the URL, the API key, and the model name.

```python
class DeepSeek(Brain):
    def __init__(self):
        self.api_key = os.getenv("DEEPSEEK_API_KEY")
        self.model = "deepseek-chat"
        self.url = "https://api.deepseek.com/anthropic/v1/messages"

    def think(self, conversation):
        headers = {
            "x-api-key": self.api_key,
            "anthropic-version": "2023-06-01",
            "content-type": "application/json"
        }
        payload = {
            "model": self.model,
            "max_tokens": 4096,
            "messages": conversation
        }
        response = request_with_retry(self.url, headers, payload)
        return self._parse_response(response.json()["content"])

    def _parse_response(self, content):
        # identical to Claude's _parse_response
```

The `_parse_response` method is a copy of Claude's. That's not a mistake — it's a consequence of the compatible wire format. Both APIs return the same JSON structure. Both produce the same `Thought` objects. The Agent never needs to know which one it's talking to.

This is the practical payoff of the Brain abstraction. The Agent calls `self.brain.think(conversation)` and receives a `Thought`. Whether that `Thought` came from a server in California or one in Beijing is invisible to the Agent.

## The Provider Registry

With two brains available, we need a way to select and switch between them. A simple dictionary handles this:

```python
BRAINS = {
    "claude":   Claude,
    "deepseek": DeepSeek,
}
```

Notice what the dictionary contains: the classes themselves, not instances. `Claude` (the class) not `Claude()` (an instance). The reason is subtle but important.

If you stored instances — `BRAINS = {"claude": Claude()}` — Python would instantiate `Claude` when the module loads, which means it would immediately look for `ANTHROPIC_API_KEY` in the environment. If the key isn't there, the module crashes at import time. Storing the class defers that check until you actually need the brain.

When you want a brain, you retrieve the class and call it:

```python
brain_name = os.getenv("NANOCODE_BRAIN", "claude")
brain = BRAINS[brain_name]()      # call the class to create an instance
```

This also makes switching brains at runtime straightforward. To switch to the next brain in the registry:

```python
def _switch_brain(self):
    names = list(BRAINS.keys())
    idx = names.index(self.brain_name)
    new_name = names[(idx + 1) % len(names)]

    try:
        self.brain = BRAINS[new_name]()
        self.brain_name = new_name
        return f"Switched to: {new_name}"
    except ValueError as e:
        return f"Cannot switch to {new_name}: {e}"
```

The modulo arithmetic `(idx + 1) % len(names)` cycles through the list, wrapping back to 0 after the last item. With two brains, typing `/switch` alternates between them. With five brains, it cycles through all five. The logic doesn't change.

One important detail: when you switch brains, the `conversation` list stays intact. The new brain inherits the full history. Both Claude and DeepSeek can pick up mid-conversation seamlessly, because the conversation is just a list of strings — it doesn't depend on which model was used to generate it.

## What Inheritance Gives Us

This is a good moment to notice what object-oriented inheritance is actually doing for us here. The `Brain` class doesn't do anything useful on its own — it just raises `NotImplementedError`. Its value is conceptual: it establishes a contract.

When you write `class Claude(Brain)`, you are making a declaration: Claude is a Brain. It has all the responsibilities of a Brain, including implementing `think()`. When the Agent receives any object typed as a Brain — whether it's Claude, DeepSeek, or a future `OllamaBrain` we haven't written yet — it can call `think()` and trust that the method exists and returns a `Thought`.

This is *polymorphism*: one interface, many implementations. The Agent is written once, against the Brain interface, and it works with every Brain ever created. Adding Ollama in Chapter 10 requires exactly one change: adding a new class and adding it to the `BRAINS` dictionary. The Agent never touches.
