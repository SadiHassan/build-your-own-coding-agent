## What This Chapter Is About

In Chapter 1 we built a loop that accepts input and echoes it back. The agent's brain was a placeholder: `"(Agent not yet connected)"`. This chapter connects it.

There's no new `nanocode.py` this chapter. Instead, we have a single script — `test_api.py` — that fires one raw HTTP request at the Anthropic API and prints exactly what comes back.

Why do this before wiring it into the agent? Because understanding the raw API response is essential. When you know what JSON Claude actually returns, the code that parses it in Chapter 3 will make immediate sense instead of feeling like magic.

By the end of this chapter you'll know:
- How to authenticate with the Anthropic API
- What an API request payload looks like
- What the full JSON response looks like (and what every field means)
- How to debug API calls when things go wrong

## How the Anthropic API Works

The Anthropic API is an HTTP endpoint. You send it a JSON payload describing the conversation, and it sends back a JSON payload containing Claude's response.

```
┌──────────────────────────────────────────────────────────────┐
│                      Your Machine                            │
│                                                              │
│  ┌────────────┐    HTTP POST + JSON     ┌─────────────────┐  │
│  │  Your      │ ──────────────────────▶ │  Anthropic      │  │
│  │  Script    │                         │  API Server     │  │
│  │            │ ◀────────────────────── │                 │  │
│  └────────────┘    JSON response        └─────────────────┘  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

No SDKs. No client libraries. Just an HTTP POST — the same mechanism your browser uses to submit a form, or a payment terminal uses to charge a card. You could make this request with `curl`. We're using Python's `requests` library for readability, but the mechanics are identical.

## The Code, Step by Step

```python
import os
import requests
import json
from dotenv import load_dotenv
```

Four imports:
- `os` — to read environment variables
- `requests` — to make the HTTP call
- `json` — to pretty-print the response
- `dotenv` — to load `.env` into the environment

### Step 1: Load the Vault

```python
load_dotenv()
api_key = os.getenv("ANTHROPIC_API_KEY")

if not api_key:
    print("Error: ANTHROPIC_API_KEY not found in .env")
    exit(1)
```

`load_dotenv()` reads the `.env` file and pushes each `KEY=VALUE` line into the process environment. After this call, `os.getenv("ANTHROPIC_API_KEY")` works just like it would if you'd typed `export ANTHROPIC_API_KEY=...` in your shell.

The guard after it is important. Without it, a missing key produces an `AttributeError: 'NoneType' object has no attribute...` error several lines later — confusing because it points to the HTTP call, not the real problem. Failing fast with a clear message is always better than confusing error trails.

### Step 2: Define the Target

```python
url = "https://api.anthropic.com/v1/messages"
```

This is the single endpoint we'll use throughout the entire book. The Anthropic API has other endpoints (for embeddings, batch processing, etc.), but `/v1/messages` is the one that runs conversations.

### Step 3: Authenticate

```python
headers = {
    "x-api-key": api_key,
    "anthropic-version": "2023-06-01",
    "content-type": "application/json"
}
```

HTTP headers are metadata attached to every request — like the envelope a letter travels in. Three are required here:

| Header | Purpose |
|--------|---------|
| `x-api-key` | Your credential. Anthropic uses this to identify and bill you. |
| `anthropic-version` | Which version of the API you're targeting. Pinning this means Anthropic can make changes to newer versions without breaking your code. |
| `content-type` | Tells the server the request body is JSON, not a form or plain text. |

### Step 4: Construct the Payload

```python
payload = {
    "model": "claude-sonnet-4-6",
    "max_tokens": 4096,
    "messages": [
        {"role": "user", "content": "Hello, are you ready to code?"}
    ]
}
```

This is the conversation you're sending. Three fields:

**`model`** — Which Claude model to use. `claude-sonnet-4-6` is the current balance of capability and speed. We'll keep using it throughout the book.

**`max_tokens`** — The maximum number of tokens Claude is allowed to generate in its response. A *token* is roughly 3–4 characters of English text. 4096 tokens ≈ about 3000 words — more than enough for any single response.

**`messages`** — A list of conversation turns. Each turn has:
- `role`: either `"user"` (you) or `"assistant"` (Claude)
- `content`: the text of that turn

Right now there's only one message: your question. In Chapter 3, this list will grow — each response from Claude and each follow-up from you gets appended here, building the conversation history.

```
messages = [
  {"role": "user",      "content": "Hello, are you ready to code?"}
]

# After Chapter 3, it will look like:
messages = [
  {"role": "user",      "content": "Hello, are you ready to code?"},
  {"role": "assistant", "content": "Absolutely! What would you like..."},
  {"role": "user",      "content": "Write me a function that..."},
  ...
]
```

### Step 5: Fire

```python
print("📡 Sending request to Claude...")
response = requests.post(url, headers=headers, json=payload, timeout=120)
```

`requests.post()` opens a TCP connection to `api.anthropic.com`, sends the HTTP request, waits for the full response, then returns. The `json=payload` argument automatically serializes the dict to JSON and sets `content-type: application/json` (which is why the header we set manually is technically redundant, but being explicit doesn't hurt).

`timeout=120` sets a 120-second limit. Without it, a stalled request would block your script indefinitely — not a problem for a quick test, but a serious bug in a long-running agent. We'll make this more robust in Chapter 9.

### Step 6: Inspect the Result

```python
print(f"Status: {response.status_code}")

if response.status_code == 200:
    print("Response:")
    print(json.dumps(response.json(), indent=2))
else:
    print("Error:", response.text)
```

HTTP status codes are the server's shorthand for "here's what happened":

| Code | Meaning | What to do |
|------|---------|------------|
| `200` | Success | Parse the JSON response |
| `401` | Unauthorized | Your API key is missing or wrong |
| `429` | Too many requests | You've hit a rate limit — back off and retry |
| `500` | Server error | Anthropic's problem — retry later |

On success we use `json.dumps(..., indent=2)` to pretty-print the response — nested JSON on one line is unreadable. On failure we print `response.text` (raw string, not parsed JSON) because error responses sometimes aren't valid JSON, and calling `.json()` on them would crash.

## Anatomy of the Response

This JSON is the raw material for everything that follows. Let's understand every field:

```
┌─────────────────────────────────────────────────────────────┐
│                     API Response JSON                       │
│                                                             │
│  id          ── unique message ID (useful for debugging)    │
│  type        ── always "message" for this endpoint          │
│  role        ── always "assistant"                          │
│                                                             │
│  content  ── list of content blocks                         │
│    └─ [0]                                                   │
│         type  ── "text" (or "tool_use" in later chapters)   │
│         text  ── ← THIS is Claude's actual response         │
│                                                             │
│  model       ── exact model version that answered           │
│  stop_reason ── why Claude stopped generating:              │
│                   "end_turn"    = natural conclusion        │
│                   "max_tokens"  = hit the limit             │
│                   "tool_use"    = wants to call a tool      │
│                                                             │
│  usage                                                      │
│    input_tokens   ── tokens in your request (billed)        │
│    output_tokens  ── tokens in Claude's response (billed)   │
└─────────────────────────────────────────────────────────────┘
```

A few things to internalize:

**`content` is a list, not a string.** Claude's response isn't `response["text"]` — it's `response["content"][0]["text"]`. The list structure exists because Claude can return multiple blocks: a text block followed by a tool-use block, for example. We'll handle that in Chapter 5.

**`stop_reason` tells you why Claude stopped.** `"end_turn"` means it finished naturally. `"tool_use"` (coming in Chapter 5) means it wants to call one of your tools before continuing. `"max_tokens"` means you cut it off — increase `max_tokens` or trim your prompt.

**`usage` is your bill.** Anthropic charges per token. Watching `input_tokens` matters especially in long conversations because the entire history is re-sent with every request. In Chapter 10 we'll add context compaction to manage this.

## What Just Happened

Here's the full request-response flow in one diagram:

```
test_api.py
    │
    ├─ load_dotenv()           → reads ANTHROPIC_API_KEY from .env
    │
    ├─ Build headers           → api key + version + content-type
    │
    ├─ Build payload           → model + max_tokens + messages list
    │
    ├─ requests.post()
    │      │
    │      │  POST /v1/messages HTTP/1.1
    │      │  Host: api.anthropic.com
    │      │  x-api-key: sk-ant-...
    │      │  Content-Type: application/json
    │      │
    │      │  {"model": "claude-sonnet-4-6", "messages": [...]}
    │      │
    │      ▼
    │  Anthropic API
    │      │
    │      │  HTTP/1.1 200 OK
    │      │  Content-Type: application/json
    │      │
    │      │  {"id": "msg_...", "content": [{"type":"text","text":"..."}], ...}
    │      │
    │      ▼
    └─ response.json()         → Python dict
           │
           └─ json.dumps(..., indent=2) → pretty-printed to terminal
```

## Common Errors and What They Mean

**`ANTHROPIC_API_KEY not found in .env`**
Your `.env` file doesn't exist or the key name is misspelled. Check that you copied `.env.example` and filled in the key.

**`Status: 401` — `{"error": {"type": "authentication_error"}}`**
The key exists but is wrong — check for extra spaces or a truncated copy-paste.

**`Status: 429` — rate limit**
You've sent too many requests too quickly. Wait a moment and try again. In Chapter 9 we'll add automatic retry-with-backoff for exactly this.

**`requests.exceptions.Timeout`**
The request took longer than 120 seconds. Anthropic's API is usually fast (under 5 seconds for short messages), so this likely means a network issue.

**`requests.exceptions.ConnectionError`**
No internet connection, or a firewall is blocking outbound HTTPS.

## What We Didn't Add (And Why)

This script is intentionally minimal. There's no retry logic, no streaming, no error recovery. If it fails, it tells you why and stops.

That's the right call for a diagnostic script. You're running `test_api.py` to answer one question: *can my machine talk to the API?* Once you've confirmed that, you never need to run this script again. The production-quality HTTP handling — retries, timeouts, rate-limit backoff — goes into `nanocode.py`, starting in Chapter 3.
