# Chapter 3: The Stateful Chatbot

In Chapter 2, we fired a single HTTP request at the Anthropic API and watched it respond. We understood the wire format. But the script was stateless — send one message, get one reply, exit. There was no memory of what had been said.

A coding agent that forgets everything after each turn isn't useful. This chapter builds the foundation that fixes that: a `Claude` class that wraps the API call, a `Thought` type that represents its response, and an `Agent` class that holds the conversation in memory across turns.

When you finish this chapter, you will have a real chatbot — one that remembers what you said three messages ago, and one whose internal structure is designed to grow into a full agent over the coming chapters.

---

## The Problem with Stateless APIs

Here is something important to understand about how language models work: **the model itself has no memory.** Every API call is completely independent. The server does not store your previous messages. It does not know you called it five seconds ago.

This sounds like a serious limitation. It isn't. It's actually an elegant design. Because there is no server-side state, the API can handle millions of simultaneous conversations without tracking any of them. Infinite scale, zero session management.

The trade-off is that *you* must manage the context. If you want Claude to remember that you asked about Python three turns ago, you must include that turn — and every turn since — in the `messages` list you send with the current request.

This is the entire secret behind conversation memory in LLM applications. There is no magic. There is just a list that grows with every exchange, and you re-send that list with every API call.

---

## The Conversation List

Here is the core data structure:

```python
conversation = [
    {"role": "user",      "content": "What is Python?"},
    {"role": "assistant", "content": "Python is a high-level programming language..."},
    {"role": "user",      "content": "Give me a simple example."},
    {"role": "assistant", "content": "Here's hello world: print('Hello, World!')"},
    {"role": "user",      "content": "Now make it a function."},
]
```

Each item in the list is a turn. Each turn has a `role` — either `"user"` or `"assistant"` — and the text of that turn. When you send this list to the API, Claude reads every message from the beginning, understands the entire thread of conversation, and continues from where it left off.

The list grows with every exchange. After Claude responds to the last message, you append its reply. Then you add the user's next message. Then you call the API again with the updated list. And so on.

```
┌────────────────────────────────────────────────────────┐
│            THE CONVERSATION LIST OVER TIME             │
│                                                        │
│  After turn 1:    [user]                               │
│  After turn 2:    [user, assistant]                    │
│  After turn 3:    [user, assistant, user]              │
│  After turn 4:    [user, assistant, user, assistant]   │
│                                                        │
│  Every API call receives the COMPLETE list so far.     │
│  The model re-reads everything each time.              │
└────────────────────────────────────────────────────────┘
```

This has a practical consequence you should keep in mind: longer conversations are more expensive. Because the whole history travels with every request, input token counts grow linearly with the number of turns. We will deal with this in Chapter 10.

## The Claude Class

Chapter 2's API call was written as a flat script — variables, a request, a print statement. That works once. It doesn't work when you need to make the same call from inside an object, with different conversations, potentially with different settings.

We wrap the API call in a class:

```python
class Claude:
    def __init__(self):
        self.api_key = os.getenv("ANTHROPIC_API_KEY")
        if not self.api_key:
            raise ValueError("ANTHROPIC_API_KEY not found in .env")
        self.model = "claude-sonnet-4-6"
        self.url = "https://api.anthropic.com/v1/messages"

    def think(self, conversation):
        headers = {
            "x-api-key": self.api_key,
            "anthropic-version": "2023-06-01",
            "content-type": "application/json"
        }
        payload = {
            "model": self.model,
            "max_tokens": 4096,
            "messages": conversation
        }
        print("(Claude is thinking...)")
        response = requests.post(self.url, headers=headers, json=payload, timeout=120)
        response.raise_for_status()
        return self._parse_response(response.json()["content"])
```

The public interface is a single method: `think(conversation)`. You hand it a list of messages; it returns a structured result. What happens in between — the headers, the URL, the HTTP call, the JSON parsing — is the class's private business.

Notice `raise_for_status()`. This is a convenience method on the `requests` response object that automatically raises an exception if the HTTP status code indicates an error (any 4xx or 5xx). Without it, a failed request would silently return a response object with an error body, and we'd have to check `response.status_code` ourselves. One line does the work of several.

## Parsing the Response into a Thought

The raw API response is a nested dictionary. Extracting the relevant parts from it every time we need them would be tedious and fragile. Instead, we parse it once into a clean Python object:

```python
class ToolCall:
    def __init__(self, id, name, args):
        self.id = id
        self.name = name
        self.args = args

class Thought:
    def __init__(self, text=None, tool_calls=None):
        self.text = text
        self.tool_calls = tool_calls or []
```

And the parsing itself:

```python
def _parse_response(self, content):
    text_parts = []
    tool_calls = []

    for block in content:
        if block["type"] == "text":
            text_parts.append(block["text"])
        elif block["type"] == "tool_use":
            tool_calls.append(ToolCall(
                id=block["id"],
                name=block["name"],
                args=block["input"]
            ))

    return Thought(
        text="\n".join(text_parts) if text_parts else None,
        tool_calls=tool_calls
    )
```

The `content` field in Claude's response is a list of blocks. In this chapter, every block will be of type `"text"`. But starting in Chapter 5, Claude will also return `"tool_use"` blocks — requests to call a function. By parsing both types now and putting them in `Thought`, the `Agent` class never has to reach into raw API dictionaries. It always works with clean, predictable Python objects.

`ToolCall` and `Thought` are simple value objects — their only purpose is holding data. `tool_calls` defaults to an empty list, so the agent can safely check `if thought.tool_calls` without needing to handle `None`.

## The Agent Class

The `Agent` owns the conversation list and coordinates the loop. It does not call the API directly — it delegates that to the brain.

```python
class Agent:
    def __init__(self, brain):
        self.brain = brain
        self.conversation = []

    def handle_input(self, user_input):
        if user_input.strip() == "/q":
            raise AgentStop()

        if not user_input.strip():
            return ""

        self.conversation.append({"role": "user", "content": user_input})

        try:
            thought = self.brain.think(self.conversation)
            text = thought.text or ""
            self.conversation.append({"role": "assistant", "content": text})
            return text
        except Exception as e:
            self.conversation.pop()
            return f"Error: {e}"
```

The flow is straightforward: append the user's message, call the brain, append the response, return the text. But the error handling deserves attention.

If `brain.think()` fails — network error, rate limit, bad response — we call `self.conversation.pop()` before returning the error message. This removes the user message we just added. Why does that matter?

Without the pop, the conversation list ends with a user message that has no assistant reply. The next time you call the API with that list, the API receives two consecutive user messages — which is invalid. The model or the API may behave unexpectedly. The pop ensures the conversation always ends in a valid state: alternating user and assistant turns.

```
┌────────────────────────────────────────────────────────┐
│              ERROR RECOVERY: BEFORE AND AFTER          │
│                                                        │
│  WITHOUT pop():              WITH pop():               │
│  ─────────────               ─────────────             │
│  [user: "hello"]             [user: "hello"]           │
│  [asst: "hi"]                [asst: "hi"]              │
│  [user: "next"]   ← stranded [user: "next"] ← removed │
│                                                        │
│  Next call: two user turns in a row → API error       │
│  Next call: clean alternation → works normally         │
└────────────────────────────────────────────────────────┘
```

## Dependency Injection

Notice how `Agent` receives its brain:

```python
agent = Agent(brain=Claude())
```

The agent doesn't create `Claude` inside its `__init__`. It receives it from outside. This pattern is called *dependency injection*, and it has an important consequence: you can pass any object that has a `think(conversation)` method, not just `Claude`.

In tests, this means you can pass a fake brain that returns scripted responses without making any network calls. No API key required. No waiting for HTTP responses. Just fast, deterministic tests. We will use this technique extensively starting in Chapter 9.

The Agent doesn't need to know *how* thinking works. It just needs something it can call `think()` on. This is the design principle that lets us add DeepSeek in Chapter 4, Ollama in Chapter 10, and any future model with zero changes to the Agent code.

## Putting It Together

In `main()`, everything wires up:

```python
def main():
    brain = Claude()
    agent = Agent(brain)

    while True:
        try:
            user_input = input("❯ ")
            output = agent.handle_input(user_input)
            if output:
                print(f"\n{output}\n")
        except (AgentStop, KeyboardInterrupt):
            print("\nExiting...")
            break
```

`Claude` is created once. `Agent` is created once, holding a reference to that brain. The loop runs forever, feeding input to the agent and printing output, until either `/q` is typed or `Ctrl+C` is pressed.

The conversation list inside the agent accumulates with every turn. By message 10, the API call is carrying a list of 19 or 20 messages. By message 50, it's carrying 99 or 100. This is the cost of stateful conversation with a stateless API.

## Two Patterns Worth Naming

Two design patterns appear in this chapter that will recur throughout the book.

**The brain abstraction.** The `Agent` and the `Claude` class are kept separate on purpose. The agent is the *coordinator* — it manages conversation state and orchestrates the loop. The brain is the *thinker* — it knows how to call a model and return a `Thought`. Keeping these concerns separate makes each class simpler, testable in isolation, and replaceable without touching the other.

**Value objects for structured data.** `Thought` and `ToolCall` are classes with no behavior — just data. You could use plain dictionaries instead, but classes give you named attributes (`thought.text` instead of `thought["text"]`), default values (`tool_calls or []`), and a stable interface that isolates you from changes in the underlying API response format. If Anthropic changes the JSON structure of a response, you update `_parse_response` in one place. The rest of the code never changes.
