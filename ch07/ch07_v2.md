# Chapter 7: Plan Mode — The Safety Harness

An agent that can write files autonomously is powerful. It is also dangerous. One misunderstood instruction, one hallucinated file path, and it overwrites something important. Without a safety gate, every agent session carries that risk.

The solution isn't to restrict the agent's capabilities. The solution is to separate planning from acting. In plan mode, the agent reads, explores, and reasons — but does not write. It can examine your codebase, draft a strategy, and tell you exactly what it intends to do. Only when you flip to act mode does it start making changes.

This chapter builds that gate.

---

## The Core Idea

The agent's operating mode is a single string: `"plan"` or `"act"`. It lives on the Agent object and changes its behavior throughout the system. In plan mode, any tool that would modify state — write a file, run a command — returns a blocked message instead of executing. In act mode, those same tools work normally.

The enforcement happens inside each tool, not in the Agent. This is deliberate. If the Agent checked the mode before calling each tool, the check would be one layer — easy to forget, easy to bypass as code grows. By putting the check inside the tool itself, it's enforced at the last possible moment, immediately before the dangerous action. No code path can bypass it.

---

## Adding Mode to ToolContext

The first change is the simplest. `ToolContext` already exists to carry agent state into tools. We add one field:

```python
class ToolContext:
    def __init__(self, mode=None, memory=None):
        self.mode = mode       # "plan" or "act"
        self.memory = memory
```

That's the entire change to `ToolContext`. One new field. Because tools already receive a context object, none of their signatures change. The container absorbs the extension exactly as designed.

---

## Mode-Aware WriteFile

Now `WriteFile` checks the mode before writing:

```python
def execute(self, context, path, content):
    # Special case: PLAN.md is always allowed
    if os.path.basename(path) == "PLAN.md":
        with open(path, 'w', encoding='utf-8') as f:
            f.write(content)
        return "Plan saved successfully to PLAN.md"

    # Block everything else in plan mode
    if context.mode == "plan":
        return f"BLOCKED: Cannot write to '{path}' in plan mode. Write to 'PLAN.md' instead."

    # Act mode: proceed normally
    with open(path, 'w', encoding='utf-8') as f:
        f.write(content)
    return f"Successfully wrote {len(content)} characters to {path}"
```

The logic has three branches. The `PLAN.md` check comes first because it's an unconditional exception — even in plan mode, the agent should be able to write its plan. This lets you review what the agent intends to do before giving it permission to act. The plan mode block comes second. Act mode falls through to the normal write.

The order of these branches matters. If you checked mode before checking for `PLAN.md`, you'd block the agent from writing its plan during plan mode — the opposite of what you want. Guard clauses work best when the most specific condition comes first.

## The Agent Gains a Mode

The Agent now carries a mode and handles the `/mode` command:

```python
class Agent:
    def __init__(self, brain, tools, memory=None, mode="plan", brain_name="claude"):
        self.mode = mode          # default: safe

    def handle_input(self, user_input):
        if user_input.strip().startswith("/mode"):
            return self._handle_mode_command(user_input)
        ...

    def _handle_mode_command(self, user_input):
        parts = user_input.strip().split()
        if len(parts) > 1 and parts[1] == "act":
            self.mode = "act"
            return "⚠️  Switched to ACT MODE (Writing Enabled)"
        else:
            self.mode = "plan"
            return "🛡️  Switched to PLAN MODE (Read-Only)"
```

And when executing a tool, the agent passes the current mode through ToolContext:

```python
def _execute_tool(self, name, args):
    context = ToolContext(mode=self.mode, memory=self.memory)
    return tool.execute(context, **args)
```

The mode is threaded all the way from the Agent's state into every tool call. A mode switch takes effect immediately on the next tool execution.

---

## The `--act` Flag

The default mode is `"plan"`. To change that, you pass `--act` on the command line:

```python
def main():
    mode = "act" if len(sys.argv) > 1 and sys.argv[1] == "--act" else "plan"
```

This reads `sys.argv`, the list of command-line arguments. `sys.argv[0]` is always the script name. `sys.argv[1]` is the first argument after it.

```bash
python nanocode.py          # mode = "plan"
python nanocode.py --act    # mode = "act"
```

Requiring an explicit flag to enable write mode is a considered choice. Accidents happen at startup, when you're not fully thinking about consequences. Making plan mode the default means a stray shell command or a mistyped path won't accidentally launch an agent with write access. You have to explicitly opt in to the dangerous mode, every time.

The prompt shows the current state at all times:

```
[claude:plan] ❯
[claude:act]  ❯
```

This is defensive UI. The user always knows what mode they're in. There's no ambiguity.

---

## Defense in Depth

The mode enforcement is layered. The `--act` flag controls the starting mode. The `/mode` command changes it mid-session. The mode check inside the tool enforces it at execution time. None of these layers trusts the others to have done the job.

```
Command line flag  ──▶  sets initial mode
                              │
/mode command      ──▶  changes mode at runtime
                              │
_execute_tool      ──▶  passes mode via ToolContext
                              │
tool.execute()     ──▶  checks mode immediately before acting
```

This is defense in depth — the same principle that protects secure systems. No single layer is the entire protection. If one check is bypassed or forgotten, the others still hold. The tool-level check is the strongest: it runs at the last possible moment, with no way around it short of modifying the tool itself.

---

## The PLAN.md Convention

The `PLAN.md` exception deserves a moment of attention. In plan mode, the agent can still write to `PLAN.md`. This creates a useful workflow:

Ask the agent to analyze your codebase and write a plan for a refactor. It reads files, understands the structure, and writes its proposed changes to `PLAN.md`. You review the plan — maybe it's perfect, maybe you redirect it. When you're happy with the direction, you type `/mode act` and ask the agent to execute the plan. It knows what to do because it wrote it down.

This is not just a safety feature. It's a good development practice. Thinking before acting, writing down the plan, reviewing before committing. The agent does it not because it's cautious but because the tool architecture guides it that way.

---

## What Changes and What Doesn't

The mode system is built into the existing architecture without disrupting it. The `Brain` classes didn't change. The conversation list didn't change. The agentic loop didn't change. One new field on `ToolContext`, a few lines in each mode-aware tool, a new method in `Agent`.

This is the payoff of the design decisions made in earlier chapters. `ToolContext` was introduced in Chapter 6 precisely so that state could be added here without changing tool signatures. The `Agent` doesn't need to enumerate every tool and check its effects — each tool is responsible for its own behavior. Adding a new dangerous tool in the future means adding the mode check to that tool. Nothing else touches.
