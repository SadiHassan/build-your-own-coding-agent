# Chapter 8: Codebase Awareness — Giving the Agent Eyes

> *"A brilliant surgeon who cannot see the operating field is helpless. Give the agent sight before you give it tools."*

## What This Chapter Is About

In every chapter so far, the agent could only work on files you explicitly mentioned. You typed "read `main.py`" and it read `main.py`. You typed "fix the bug in `utils.py`" and it fixed `utils.py`. The agent was brilliant but blind — it could not see your codebase unless you described it.

This chapter changes that. We add two new tools:

- **`ListFiles`** — walks the directory tree and returns an indented map of every file
- **`SearchCodebase`** — walks every file and returns every line matching a query

Together, these tools give the agent independent sight. It can now explore a codebase it has never seen, find where a function is defined, trace which files import a module, and understand the project structure — all without you narrating the tour.

By the end of this chapter you'll understand:
- Why directory walking is the right primitive for codebase discovery
- What directories to skip and why skipping them matters
- Why case-insensitive search is almost always correct for code
- How `errors='ignore'` makes text search binary-safe
- Why `filepath:lineno: line` is the output format every developer already knows

## ListFiles: Mapping the Territory

`ListFiles` uses `os.walk()` to traverse a directory tree and builds an indented text representation. Think of it as the `tree` command, but implemented in 15 lines of Python with sensible exclusions for coding agents.

```python
class ListFiles:
    """Lists files in the project structure."""
    name = "list_files"
    description = "Lists all files in the project structure. Useful to understand the project layout."
    input_schema = {
        "type": "object",
        "properties": {
            "path": {"type": "string", "description": "The root path (default '.')"}
        }
    }

    def execute(self, context, path="."):
        print(f"  → Listing {path}")
        try:
            file_list = []
            for root, dirs, files in os.walk(path):
                if any(p in root for p in [".git", "__pycache__", "venv", ".nanocode"]):
                    continue

                level = root.replace(path, '').count(os.sep)
                indent = ' ' * 4 * (level)
                file_list.append(f"{indent}{os.path.basename(root)}/")
                subindent = ' ' * 4 * (level + 1)
                for f in files:
                    file_list.append(f"{subindent}{f}")

            return "\n".join(file_list)
        except Exception as e:
            return f"Error listing files: {e}"
```

Walk through the key decisions:

### os.walk(): Depth-First, Batteries Included

`os.walk(path)` is a generator that yields `(root, dirs, files)` for every directory in the tree. `root` is the current directory path, `dirs` is a list of subdirectory names, `files` is a list of file names. It handles all the recursion for you.

```
  os.walk("my_project")
       │
       ├── yields ("my_project", ["src", "tests"], ["README.md", "setup.py"])
       ├── yields ("my_project/src", ["utils"], ["main.py", "config.py"])
       ├── yields ("my_project/src/utils", [], ["helpers.py"])
       └── yields ("my_project/tests", [], ["test_main.py"])
```

The indented tree is built from these yields. The level (depth) is derived by counting path separators:

```python
level = root.replace(path, '').count(os.sep)
```

Replacing `path` removes the root prefix, leaving only the relative portion. Counting `os.sep` (which is `/` on Unix, `\` on Windows) gives the depth. Level 0 is the root directory, level 1 is its immediate children, and so on.

```
  path = "my_project"
  root = "my_project/src/utils"
  root.replace(path, '') = "/src/utils"
  .count(os.sep)          = 2   (two slashes)
  indent = ' ' * 4 * 2   = "        " (8 spaces)
```

### What the Output Looks Like

Given a project:

```
my_project/
  src/
    main.py
    config.py
    utils/
      helpers.py
  tests/
    test_main.py
  README.md
  setup.py
```

`list_files` returns:

```
my_project/
    src/
        main.py
        config.py
        utils/
            helpers.py
    tests/
        test_main.py
    README.md
    setup.py
```

This is the indentation ladder — each level gets four more spaces than its parent.

```
  Directory tree               Indented output
  ─────────────                ───────────────
  my_project/              →   my_project/
  ├── src/                 →       src/
  │   ├── main.py          →           main.py
  │   ├── config.py        →           config.py
  │   └── utils/           →           utils/
  │       └── helpers.py   →               helpers.py
  ├── tests/               →       tests/
  │   └── test_main.py     →           test_main.py
  ├── README.md            →       README.md
  └── setup.py             →       setup.py
```

Four spaces per level is the Python indentation standard. Claude reads this naturally — it's the same indentation scheme Claude uses when writing Python code.

### What Gets Excluded and Why

```python
if any(p in root for p in [".git", "__pycache__", "venv", ".nanocode"]):
    continue
```

Four exclusions, each with a reason:

| Excluded | Why |
|----------|-----|
| `.git` | Git internals: object store, refs, hooks. Claude has no use for these and they contain thousands of files. |
| `__pycache__` | Python bytecode cache. Always derived from source files — never edit these directly. |
| `venv` | Installed third-party packages. Can contain thousands of files. Claude should read the package's documentation, not its installed source. |
| `.nanocode` | Nanocode's own data directory (memory.md, etc.). Internal state the agent should not accidentally recurse into. |

The check `any(p in root for p in [...])` uses `in` (substring match, not equality). This means `.git` matches `my_project/.git`, `my_project/.git/objects`, `my_project/.git/refs/heads`, and any deeper path. A single check skips the entire subtree.

Excluding noise is not just about aesthetics. Every token in Claude's context window costs money and competes with actual useful content. A typical Python project with a `venv` contains 50,000+ files. Showing Claude all of them would be useless at best and context-destroying at worst.

## SearchCodebase: Finding a Needle in Any Haystack

`SearchCodebase` is the agent's `grep`. It walks every file in the tree, reads it line by line, and collects every line that matches the query.

```python
class SearchCodebase:
    """Searches for a string in all files."""
    name = "search_codebase"
    description = "Searches the entire codebase for a text string. Useful to find where functions or variables are defined."
    input_schema = {
        "type": "object",
        "properties": {
            "query": {"type": "string", "description": "The string to search for"},
            "path": {"type": "string", "description": "The root path (default '.')"}
        },
        "required": ["query"]
    }

    def execute(self, context, query, path="."):
        print(f"  → Searching for '{query}'")
        results = []
        try:
            for root, dirs, files in os.walk(path):
                if any(p in root for p in [".git", "__pycache__", "venv", ".nanocode"]):
                    continue

                for file in files:
                    file_path = os.path.join(root, file)
                    try:
                        with open(file_path, 'r', encoding='utf-8', errors='ignore') as f:
                            for i, line in enumerate(f):
                                if query.lower() in line.lower():
                                    results.append(f"{file_path}:{i+1}: {line.strip()}")
                    except Exception:
                        continue

            return "\n".join(results) if results else "No matches found."
        except Exception as e:
            return f"Error searching: {e}"
```

The structure mirrors `ListFiles` — same `os.walk`, same exclusions — but instead of recording directory names, it opens every file and searches inside.

### How the Search Walks

```
  os.walk(".")
       │
       ├── for each directory (skipping .git, __pycache__, venv, .nanocode)
       │       │
       │       └── for each file in that directory
       │               │
       │               └── open file, read line by line
       │                       │
       │                       └── if query matches line → record result
       │
       └── return all results joined by newline
```

### Case-Insensitive Search: Why It's Almost Always Right

```python
if query.lower() in line.lower():
```

Both sides are lowercased before comparison. This means searching for `"deprecated_func"` finds `deprecated_func`, `Deprecated_Func`, `DEPRECATED_FUNC`, and `DeprecatedFunc` (if they appear as substrings in a line).

Why does this matter for code? Consider these scenarios:

- You search for `"myclass"` and the class is actually named `MyClass`
- You search for `"config"` and it appears as `CONFIG`, `Config`, `config`, and `config_file` in different files
- You search for `"TODO"` and some developers wrote `todo`, `Todo`, `TODO`

Case-insensitive search catches all of these. The cost is negligible — `.lower()` is a fast string operation. The benefit is a tool that works the way developers think, not the way filesystems store strings.

If you need exact case matching, you can always add it as an option later. Defaulting to case-insensitive is the right call for a general-purpose search tool.

### errors='ignore': Binary-Safe by Default

```python
with open(file_path, 'r', encoding='utf-8', errors='ignore') as f:
```

The `errors='ignore'` argument tells Python to silently skip bytes that are not valid UTF-8. Without this, opening a binary file — a compiled `.pyc`, a PNG, a compiled extension `.so` — would raise a `UnicodeDecodeError` and crash the search.

With `errors='ignore'`, the search runs cleanly over every file. Binary files produce garbled text when interpreted as UTF-8, but the query (which is always printable ASCII) will never match the garbled output. So binary files are effectively skipped without an explicit type check.

The inner `try/except Exception: continue` provides a second layer of protection for files that fail to open at all (permission denied, symlink loops, etc.).

### The Output Format: filepath:lineno: line

```python
results.append(f"{file_path}:{i+1}: {line.strip()}")
```

An example result:

```
./src/main.py:14: result = deprecated_func(x)
./src/utils.py:203: return deprecated_func(item)
./tests/test_main.py:41: assert deprecated_func(1) == 2
```

This is the universal grep output format: `file:line: content`. Every Unix developer has seen this thousands of times. It is:

- **Parseable by Claude** — the colon-separated structure is unambiguous
- **Familiar to developers** — it matches what `grep -n`, `ripgrep`, and VS Code's search output look like
- **Actionable** — Claude can use `file_path` directly in a `read_file` call
- **Compact** — one result per line, no wasted space

`i+1` because `enumerate` starts at 0 but humans (and editors) count lines from 1. `line.strip()` removes leading and trailing whitespace — indented code lines would otherwise have useless leading spaces in the output.

## Summary

| Concept | Key Detail |
|---------|-----------|
| `os.walk()` | Yields `(root, dirs, files)` for every directory — handles all recursion |
| Exclusions | `.git`, `__pycache__`, `venv`, `.nanocode` — skip noise, save tokens |
| Depth calculation | `root.replace(path, '').count(os.sep)` — counts separators for indentation level |
| Four-space indent | Matches Python convention — Claude reads it naturally |
| Case-insensitive search | `query.lower() in line.lower()` — catches mixed-case identifiers |
| `errors='ignore'` | Binary files produce no matches instead of crashing |
| `filepath:lineno: line` | Universal grep format — parseable by Claude, familiar to developers |
| `path` defaults to `"."` | Optional argument — works on the whole project or a subdirectory |
