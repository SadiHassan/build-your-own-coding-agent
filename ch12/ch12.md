## The Session: Building a Snake Game

What follows is a narrative walkthrough of how the agent behaves when given this task. This is representative of what the agent does — the exact responses will vary by model version and search results, but the *pattern* of behavior is consistent and predictable.

### The Opening Prompt

```
[claude:act] ❯ Build a Snake game in Python using pygame.
               Save all files to ch12/snake_game/.
```

The agent receives this. Before writing a single line of code, it does something important: it goes looking for information it might not have.

Now the agent knows: `pygame 2.6.1`, `pip install pygame`, and a reminder of the core primitives. It did not know to verify these things — it chose to, because it learned from memory that current versions matter.

---

### Step 2: The Agent Plans

```
(Claude is thinking...)
  → Writing ch12/snake_game/PLAN.md
```

Wait — we are in act mode. Why write a PLAN.md?

Because the agent was *trained* to plan before coding, and `PLAN.md` is how it externalizes that planning. Even in act mode, it chooses to write a plan because producing one makes the subsequent code better. This is emergent behavior — not something we programmed directly.

The plan it writes:

```markdown
# Snake Game Plan

## Files
- snake_game.py     — main game loop and entry point
- snake.py          — Snake class (body segments, direction, growth)
- food.py           — Food class (random placement, collision)
- constants.py      — screen size, colors, speeds

## Game Loop
1. Handle events (quit, arrow keys)
2. Update snake position
3. Check collisions (wall, self, food)
4. Draw frame
5. Clock tick

## Key Rules
- Snake starts at center, 3 segments, moving right
- Food spawns at random grid position
- Eating food: score +1, snake grows by 1
- Hitting wall or self: game over, show score
```

Four files. A clear structure. The plan is readable by a human and executable by the agent.

---

### Step 3: The Agent Writes the Code

Now the writing begins in earnest. The agent calls `write_file` multiple times in sequence:

```
(Claude is thinking...)
  → Writing ch12/snake_game/constants.py
  → Writing ch12/snake_game/food.py
  → Writing ch12/snake_game/snake.py
  → Writing ch12/snake_game/snake_game.py
```

The constants file first — it has no dependencies. Then food and snake, which depend only on constants. Then the main game file last, which depends on everything. This ordering is deliberate: the agent builds from the bottom of the dependency tree up.

---

### Step 4: The Agent Runs It

```
(Claude is thinking...)
  → Running: cd ch12/snake_game && pip install pygame --quiet && python snake_game.py
```

The agent does not ask for permission. It does not pause and say "should I run this?" In act mode, running the code is part of the task. The `RunCommand` tool captures stdout and stderr and returns both.

The first run produces an error:

```
STDERR:
Traceback (most recent call last):
  File "snake_game.py", line 3, in <module>
    from snake import Snake
  File "snake.py", line 14, in <module>
    self.body = [(GRID_W // 2, GRID_H // 2)]
NameError: name 'GRID_W' is not defined
```

The agent reads this. It does not panic. An import error telling it exactly which file and which line has the problem is, from the agent's perspective, simply more information.

---

### Step 5: The Agent Iterates

```
(Claude is thinking...)
  → Reading ch12/snake_game/snake.py
  → Editing ch12/snake_game/snake.py
```

It reads the file to confirm what it contains, then uses `edit_file` to make a surgical fix — adding the missing import at the top of `snake.py`:

```python
from constants import GRID_W, GRID_H
```

One line. No need to rewrite the whole file. Then it runs again:

```
  → Running: cd ch12/snake_game && python snake_game.py
```

This time: no error. The pygame window opens. The snake appears. It moves. It eats food. It grows.

## The Inner Loop: How the Agent Thinks

What you just watched is the agentic loop in action. It is worth making the mechanics explicit, because they explain *why* the agent behaves the way it does.

```
┌──────────────────────────────────────────────────────────────────┐
│                    The Agentic Inner Loop                        │
│                                                                  │
│  User prompt arrives                                             │
│         │                                                        │
│         ▼                                                        │
│  ┌─────────────────────────────────────────────────┐             │
│  │                 Iteration N                     │             │
│  │                                                 │             │
│  │  1. brain.think(conversation)                   │             │
│  │     → Claude reads full history, decides        │             │
│  │       what to do next                           │             │
│  │                                                 │             │
│  │  2. Did Claude request tool calls?              │             │
│  │     NO  → return collected text, we are done   │             │
│  │     YES → continue to step 3                   │             │
│  │                                                 │             │
│  │  3. Execute each tool_call                      │             │
│  │     result string appended to conversation      │             │
│  │                                                 │             │
│  │  4. Loop back to step 1                         │             │
│  └─────────────────────────────────────────────────┘             │
│         │                                                        │
│         │ (up to 50 iterations)                                  │
│         ▼                                                        │
│  Final response returned to user                                 │
└──────────────────────────────────────────────────────────────────┘
```

The loop has no hardcoded behaviors. The agent searches because Claude *decides* to search, having learned that current information helps produce better code. It plans because Claude *decides* to plan. It reads before editing because Claude *decides* to verify before changing.

All of this comes from the model's training and the system prompt (its memory). You did not write decision logic. You built the loop infrastructure and gave Claude tools. Claude filled in the strategy.

## What Makes This Different from Just Using ChatGPT

A natural question at this point: ChatGPT can write a Snake game too. I could paste the code it gives me into a file. What did building this agent actually buy me?

Four things. Each one is concrete.

### 1. It Can See Your Actual Files

ChatGPT does not know what is in your project. You can paste code snippets to it, but the moment your project has ten files, pasting is slow and error-prone. Nanocode has `ReadFile`, `ListFiles`, and `SearchCodebase`. It sees your project the way you see it — all of it, exactly as it is on disk, with line numbers.

When the import failed in the Snake game session, the agent read `snake.py` to see the actual current contents before editing. ChatGPT would have asked you to paste it.

### 2. It Runs Commands and Sees Real Errors

ChatGPT generates code and hands it to you. You run it. You get an error. You paste the error back. ChatGPT suggests a fix. You apply the fix. You run it again.

Nanocode eliminates the human relay. It runs the code, reads the error from stderr, diagnoses it, edits the file, and runs again — all in one agentic loop iteration. The fix-run-check cycle happens in seconds, not minutes.

### 3. It Remembers Across Sessions

If you tell ChatGPT that you prefer two-space indentation, you will have to tell it again next session. And the session after that.

Nanocode's memory persists to `.nanocode/memory.md`. Tell it your preferences once — preferred style, project conventions, your name — and it reads those preferences from its system prompt on every subsequent session. Across days. Across projects. Indefinitely.

### 4. It Can Use Local Models

ChatGPT requires an internet connection and an OpenAI account. Nanocode can run with Ollama, entirely on your machine, with no data leaving your computer. For proprietary codebases, regulated environments, or simply offline work, this is not a nice-to-have — it is the only option.

```
┌──────────────────────────────────────────────────────────────────┐
│         Nanocode vs. Chat Interface: Side by Side                │
│                                                                  │
│  Task           │ Chat (ChatGPT)       │ Nanocode               │
│  ───────────────┼──────────────────────┼────────────────────────│
│  Read files     │ You paste snippets   │ Agent reads directly   │
│  Fix errors     │ You relay the error  │ Agent runs, reads, fixes│
│  Preferences    │ Repeat each session  │ Persisted to memory.md │
│  Run commands   │ Not possible         │ RunCommand tool        │
│  Web search     │ Built into product   │ SearchWeb tool         │
│  Local models   │ Not available        │ Ollama brain           │
│  Source visible │ Black box            │ 767 lines, all yours   │
└──────────────────────────────────────────────────────────────────┘
```

## How to Extend It

You are now at the interesting edge. The agent works. Where do you take it?

Here are concrete ideas, ordered roughly from simpler to more involved:

### New Tools

```
┌────────────────────────────────────────────────────────────────────────────┐
│                        Tool Ideas and Difficulty                           │
│                                                                            │
│  Tool            What it does                              Difficulty      │
│  ──────────────  ────────────────────────────────────────  ──────────────  │
│  FetchURL        Download and return the text of a URL     Easy            │

│                                                                            │

│                  (subprocess, like RunCommand but scoped)                  │

│                  rewriting it (useful for log files)                       │
│                                                                            │
│  AskUser         Pause and prompt the user for input       Medium          │
│                  during the agentic loop                                   │
│                                                                            │
│  DatabaseTool    Run SQL queries on a SQLite database       Medium          │
│                  sqlite3 is in the stdlib — no install                     │
│                                                                            │
│  BrowserTool     Drive a real browser with Playwright       Hard            │
│                  for JavaScript-heavy pages                                │
│                                                                            │
│  ImageTool       Send a screenshot or diagram to Claude    Hard            │
│                  (Claude is multimodal)                                    │

│                  using threading or asyncio                                │
└────────────────────────────────────────────────────────────────────────────┘
```

Every new tool follows the same pattern as every existing one: a class with `name`, `description`, `input_schema`, and `execute()`. Add it to the `tools` list. Done. No framework changes. No registration. The agent discovers it through the tool definitions sent to the API.

### New Brains

The `Brain` base class has one method: `think(conversation) → Thought`. Any LLM provider that accepts a conversation and returns structured output can be wrapped into a `Brain` subclass.

- **GPT-4o**: OpenAI's API uses a different request format than Anthropic's, but the response parsing is similar
- **Gemini**: Google's API; also requires its own request format
- **Claude via Bedrock**: Same model, enterprise deployment, different auth
- **LM Studio**: Like Ollama, runs local models; different endpoint format

Each new brain is roughly 40-60 lines of code, following the shape of `Claude`, `DeepSeek`, or `Ollama`.

### New Features

**Streaming output.** Right now the agent waits for the full response before printing anything. Streaming lets you print tokens as they arrive — the response feels faster and more alive. The Anthropic API supports this with `"stream": true` and server-sent events. The appendix covers this.

**Parallel tool calls.** When the agent calls multiple tools in one iteration — say, reading three files at once — it currently calls them sequentially. Python's `threading` or `concurrent.futures` can run them in parallel, cutting wait time.

**Web UI.** Replace `main()`'s `input()`/`print()` loop with a Flask or FastAPI server. The frontend becomes a simple text box. The agent runs in a thread. The response streams to the page.

**Agent-to-agent.** Use the `Brain` abstraction to let one Nanocode instance spawn a second. The first plans; the second acts. Or one reviews the other's code. Multi-agent systems are built from exactly the primitives you now have.

## The "Zero Magic" Payoff

Every chapter began with the warning that we were doing things the hard way — no SDK, no framework, no magic. This was a promise with a delayed payoff. You are holding the payoff now.

When something goes wrong with a framework-based agent, debugging means reading framework source code, filing issues, waiting for maintainers. When something goes wrong with Nanocode, debugging means reading your own code — 767 lines you wrote, every one of which you can explain.

The lines-per-concept ratio in Nanocode is deliberately low:

```
┌─────────────────────────────────────────────────────────────────┐
│              Lines of Code per Major Concept                    │
│                                                                 │
│  Event loop          ~20 lines    main() + handle_input()       │
│  HTTP + retry        ~30 lines    request_with_retry()          │
│  Brain abstraction   ~60 lines    Brain base + Claude class     │
│  Agentic loop        ~40 lines    _agentic_loop()               │
│  Memory              ~30 lines    Memory class                  │
│  Plan/act mode       ~20 lines    ToolContext + mode checks      │
│  Context compaction  ~15 lines    _compact_conversation()       │
│  Each tool           ~15-30 lines per tool                      │
│                                                                 │
│  Total: 767 lines. Every line traceable to a decision.          │
└─────────────────────────────────────────────────────────────────┘
```

When you read your own code in six months, you will understand it. When a collaborator reads it, they will understand it. When you want to change something — add a tool, swap a brain, adjust the safety policy — you will know exactly where to look and exactly what to change.

That is what Zero Magic buys you: a system that is fully yours.

The interesting engineering is everything around it: the loop that drives it, the tools that extend it, the memory that grounds it, the safety model that constrains it.

You built that engineering, from scratch, one chapter at a time. And now you understand it the way you understand your own code — because it is your own code.

The Snake game in `ch12/snake_game/` is not the achievement. The achievement is that you can tell the agent to build anything, in any language, using any library, with current web documentation — and then extend the agent itself when you need it to do something new.

The coding agent you built is a tool for building tools. That loop does not have a natural end.
