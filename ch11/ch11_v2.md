# Chapter 11: Web Search — The Final Tool

Every language model has a knowledge cutoff. Claude's training data ends at a specific date. Ask it about a library that released a breaking change last month, and it will confidently give you the old API. Ask it about a CVE disclosed last week, and it won't know it exists. Ask it to use a framework announced after its cutoff, and it will make things up.

This is not a flaw in the model. It's a structural reality: training is expensive and doesn't happen continuously. The model's knowledge is a snapshot, not a live feed.

The fix is simple: give the agent access to a search engine. When Claude needs current information, it searches the web, reads the results, and incorporates them. This chapter adds that capability with a single new tool and sends the agent's version number to 1.0.

---

## The SearchWeb Tool

```python
try:
    from ddgs import DDGS
except ImportError:
    DDGS = None

class SearchWeb:
    name = "search_web"
    description = "Searches the internet for current information. Use when you need knowledge beyond your training data."
    input_schema = {
        "type": "object",
        "properties": {
            "query": {"type": "string", "description": "The search query"}
        },
        "required": ["query"]
    }

    def execute(self, context, query):
        if DDGS is None:
            return "Error: ddgs package not installed. Run: pip install ddgs"
        try:
            results = DDGS().text(query, max_results=3)
            if not results:
                return "No results found."

            formatted = []
            for r in results:
                formatted.append(
                    f"Title: {r['title']}\nURL: {r['href']}\nSummary: {r['body']}\n"
                )
            return "\n".join(formatted)
        except Exception as e:
            return f"Error searching web: {e}"
```

The `ddgs` library wraps the DuckDuckGo search API. No API key. No account. No billing. DuckDuckGo allows programmatic access from libraries like this one without authentication.

The import uses the graceful degradation pattern: `try: from ddgs import DDGS` / `except ImportError: DDGS = None`. If the package isn't installed, `DDGS` is `None`. The `execute` method checks for `None` and returns an informative error message instead of crashing. The agent tells you what to do: `"Run: pip install ddgs"`. This is better behavior than a traceback.

We limit to three results. More results mean more tokens. Three is usually enough to extract the information Claude needs — a current API signature, the latest version number, the fix for a recently-discovered bug. If three results aren't sufficient, Claude can refine the query and search again.

The tool does not follow links. It returns the title, URL, and DuckDuckGo's summary snippet for each result. That's often enough. If Claude needs the full content of a specific page, it could theoretically use `RunCommand` to curl it — but in practice, the snippet is sufficient for most "what's the current API for X" queries.

## Completing the Agent

With `SearchWeb` added, the tools list is:

```python
tools = [
    ReadFile(),
    WriteFile(),
    EditFile(),
    ListFiles(),
    SearchCodebase(),
    SaveMemory(),
    RunCommand(),
    SearchWeb(),
]
```

Eight tools covering the full capability set:
- Reading and writing files (`ReadFile`, `WriteFile`, `EditFile`)
- Navigating the codebase (`ListFiles`, `SearchCodebase`)
- Persisting knowledge (`SaveMemory`)
- Running code (`RunCommand`)
- Accessing current information (`SearchWeb`)

The version number in `main()` moves to `v1.0`. This is deliberate. Every chapter before this was a development increment. This is the complete, production-ready agent.

## How It All Connects

Looking back across the twelve chapters, each one added exactly one capability:

```
ch01  Event loop                 — a program that accepts input
ch02  API call                   — first contact with Claude
ch03  Conversation memory        — the agent remembers the session
ch04  Multiple providers + retry — resilience and choice
ch05  Tools + agentic loop       — the agent can act
ch06  Persistent memory          — the agent remembers across sessions
ch07  Plan/act mode              — the agent is safe by default
ch08  ListFiles + SearchCodebase — the agent can explore
ch09  RunCommand + EditFile      — the agent can execute and fix
ch10  Ollama                     — the agent can run locally
ch11  SearchWeb                  — the agent has current knowledge
```

Each addition was about ten to forty lines. None of them required rewriting existing code. The architecture absorbed each new capability because the core abstractions — Brain, Tool, Agent, ToolContext — were designed to be extended.

That's the engineering lesson of this book, as much as the AI one. Agents built on clean, well-isolated abstractions grow gracefully. Agents built on tangled spaghetti calcify after the second or third feature.
