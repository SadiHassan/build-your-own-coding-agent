# Chapter 6: Persistent Memory

Every time you close the terminal, the agent loses everything. The conversation list empties. Any facts Claude learned about your preferences, your project structure, your coding style — gone. The next session starts completely fresh, as if you'd never spoken.

This is fine for a demo. It's frustrating for a tool you use daily.

Chapter 6 fixes it. We add a `Memory` class that persists to a Markdown file, a `ToolContext` that threads state into tools without coupling them to the agent, and a `SaveMemory` tool that lets Claude update its own memory. After this chapter, the agent will remember things you told it last week.

---

## Two Kinds of Memory

It helps to think about what kind of memory we're building before we look at the code.

The `conversation` list from Chapter 3 is *session memory* — short-term, in-RAM, lost on exit. It holds the current thread: what was said this session, in order. The API needs it to maintain coherence within a conversation.

What we're adding now is *persistent memory* — long-term, on-disk, survives restart. It holds facts that should carry across sessions: the user's preferences, the project's conventions, anything the agent has learned that's worth keeping. This memory doesn't live in the `messages` list. It lives in the `system` field.

```
┌──────────────────────────────────────────────────────────┐
│                 SHORT-TERM vs LONG-TERM                  │
│                                                          │
│  conversation = []           .nanocode/memory.md         │
│                                                          │
│  Lives in RAM                Lives on disk               │
│  Grows each turn             Updated by SaveMemory tool  │
│  Lost on exit                Survives every restart      │
│  Sent as "messages"          Sent as "system"            │
└──────────────────────────────────────────────────────────┘
```

## The Memory Class

```python
class Memory:
    def __init__(self, path=".nanocode/memory.md"):
        self.path = path
        self._ensure_exists()
        self.content = self._load()

    def _ensure_exists(self):
        os.makedirs(os.path.dirname(self.path), exist_ok=True)
        if not os.path.exists(self.path):
            default = "I am Nanocode, a helpful coding assistant.\n"
            with open(self.path, "w") as f:
                f.write(default)

    def _load(self):
        with open(self.path, 'r') as f:
            return f.read()

    def save(self, content):
        self.content = content
        with open(self.path, 'w') as f:
            f.write(content)
```

The class is deliberately simple. On initialization, it creates the `.nanocode/` directory and file if they don't exist — using `os.makedirs(exist_ok=True)` so it doesn't fail if they're already there — then reads the file contents into `self.content`. That in-memory copy is what gets sent to Claude. When the agent saves new memory, both the in-memory copy and the file on disk are updated simultaneously.

The `.nanocode/` directory follows the Unix dotfile convention: configuration and state for a specific tool lives in a hidden directory named after the tool. It's gitignored by default, which means your memory file won't accidentally be committed to source control.

Calling `save()` after any update is important. If the agent updates `self.content` in RAM but doesn't write to disk, the next session won't see the change. The `save()` method does both atomically: update the attribute, then write the file.

## The System Prompt

The system prompt is a special field in the API payload, separate from the `messages` list. It's not a turn in the conversation — it's a persistent briefing that precedes every conversation. Claude reads it before processing any messages.

```python
def think(self, conversation):
    payload = {
        "model": self.model,
        "max_tokens": 4096,
        "messages": conversation
    }
    if self.memory:
        payload["system"] = self.memory.content
```

When the memory file contains something like "I am Nanocode. The user prefers type hints in Python. The project uses pytest for testing," that context shapes every response Claude gives. It knows who it is, who it's talking to, and the conventions of the project — not because the user reminded it each session, but because the agent remembered.

This is the mechanism behind every AI assistant that seems to "know you." There's no magic model training happening. There's a system prompt that gets prepended to every conversation, updated over time to reflect what the assistant has learned.

## ToolContext: Threading State Into Tools

Now that tools need access to the agent's memory, we have a design problem. The `execute` method signature has been `execute(self, path)` or `execute(self, path, content)` — just the tool's specific arguments. If we add `memory` as a parameter to every tool, we have to change every tool's signature. And when we add `mode` in Chapter 7, we'd have to change them all again.

The solution is a small container class that bundles all the agent state a tool might need:

```python
class ToolContext:
    def __init__(self, memory=None):
        self.memory = memory
```

Every tool's `execute` method now takes `context` as its first argument:

```python
class ReadFile:
    def execute(self, context, path):       # context is new
        ...

class SaveMemory:
    def execute(self, context, content):
        context.memory.save(content)
        return "Memory updated successfully."
```

And in `_execute_tool`, we create a fresh context and pass it in:

```python
def _execute_tool(self, name, args):
    tool = get_tool(self.tools, name)
    context = ToolContext(memory=self.memory)
    return tool.execute(context, **args)
```

Most tools ignore the context entirely. `ReadFile` and `WriteFile` don't need memory — they just use their path and content arguments. But `SaveMemory` does, and it accesses the shared `Memory` object through `context.memory`.

The value of `ToolContext` becomes clear in Chapter 7 when we add `mode`. We just add one field to `ToolContext`. No tool signatures change. No callers change. The container absorbs the extension.

## The SaveMemory Tool

```python
class SaveMemory:
    name = "save_memory"
    description = "Updates your internal memory/scratchpad. Use this to remember user preferences, project context, or anything important for future sessions."
    input_schema = {
        "type": "object",
        "properties": {
            "content": {
                "type": "string",
                "description": "The full text to save. This replaces the entire memory."
            }
        },
        "required": ["content"]
    }

    def execute(self, context, content):
        if context.memory is None:
            return "Error: Memory not available"
        context.memory.save(content)
        return "Memory updated successfully."
```

The description is worth reading carefully. We tell Claude to use this tool to remember "preferences, project context, or anything important for future sessions." This shapes when Claude decides to call it. A well-written description is a form of behavior specification.

The tool replaces the entire memory file with new content. This means if Claude wants to add something, it should include the existing memory in the new content — not just write the new fact. It's the agent's responsibility to manage the memory file the same way a developer manages a configuration file: read first, then write a complete updated version.

## The Memory Lifecycle

Here is how everything connects, end to end:

When the agent starts, `Memory.__init__()` reads `.nanocode/memory.md`. If the file doesn't exist, it's created with a default identity string. The content is stored in `memory.content`.

On every API call, `memory.content` is passed as `payload["system"]`. Claude reads it before processing any messages, so it knows who it is and what it already knows about you.

If Claude decides to update its memory — perhaps you mentioned you prefer a particular coding style — it calls `SaveMemory`. The tool calls `memory.save(new_content)`, which updates both the in-memory copy and the file on disk.

The next time you start the agent, the updated memory is loaded from disk. Claude picks up where it left off.

```
startup → read memory.md → send as system prompt → Claude
                                                       │
                                                       │ (if SaveMemory called)
                                                       ▼
                                              memory.save(new_content)
                                               │              │
                                               ▼              ▼
                                         self.content   memory.md
                                         (in RAM)       (on disk)
                                               │
                                next startup → read → send → Claude
```

## What Good Memory Looks Like

The default memory is a single line: "I am Nanocode, a helpful coding assistant." Over time, through saved updates, a mature memory file might look like this:

```markdown
I am Nanocode, a helpful coding assistant.

## User Preferences
- Prefers Python with type hints
- Uses pytest for testing, never unittest
- Wants verbose comments in complex functions
- Working on a FastAPI project

## Project Context
- Main application: src/app.py
- Tests in: tests/
- Database: PostgreSQL via SQLAlchemy
```

This is free-form text. Claude wrote it; Claude reads it. As long as it's coherent, the format doesn't matter. The memory file is a living document that grows more useful the longer you work with the agent.
